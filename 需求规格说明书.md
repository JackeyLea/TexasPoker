# 1. 引言与总体描述

## 1.1 文档概述

### 1.1.1 文档目的、适用范围与目标读者定义

本文档为**通用棋牌游戏框架软件需求规格说明书**（Software Requirements Specification，以下简称SRS），旨在全面、精确、可验证地定义该框架的全部功能性需求与非功能性需求。框架的核心目标是通过插件化架构实现"一次搭建，多游戏复用"——即一套统一的运行时环境支撑多种棋牌游戏的接入与运营，消除每款新游戏重复开发账号、社交、房间、支付等通用系统的资源浪费。

本文档的适用范围覆盖框架本身的全部模块（大厅系统、插件管理、房间与匹配、状态机抽象、AI托管、社交通讯、成就体系、回放系统），以及框架与游戏插件之间的全部接口契约。本文档**不涉及**任何具体游戏的内部规则实现（如德州扑克的牌型比较算法或象棋的走法规则），此类内容属于各游戏插件的独立设计范畴。

目标读者包括：项目经理（用于进度规划与资源分配）、系统架构师与后端开发工程师（用于框架核心实现）、客户端开发工程师（用于跨平台大厅与插件接口对接）、游戏插件开发者（用于理解接口规范以开发新游戏）、测试工程师（用于制定测试计划与验收标准），以及运营方产品人员（用于评估运营活动的技术可行性）。

## 1.2 项目背景与愿景

### 1.2.1 通用棋牌游戏框架的产品定位：一次搭建、多游戏复用的插件化平台

当前棋牌游戏市场的开发模式存在显著的重复建设问题：每款新游戏（无论是五子棋还是德州扑克）都需要独立开发或复制一套包含账号登录、好友关系、房间管理、聊天系统、支付对接、成就体系在内的基础能力。这种模式导致开发周期长、维护成本高、各游戏间数据孤岛严重、玩家无法跨游戏共享资产与社交关系。

本产品定位为**棋牌游戏平台启动器与通用运行时环境**。框架一次性构建并持续维护全部基础能力，具体棋牌游戏则作为**可热插拔的独立插件**动态接入。新游戏开发者的唯一工作是实现框架定义的标准接口——包括注册元数据、提供参数配置面板、实现游戏逻辑控制器、接入AI决策、渲染游戏专属UI——即可自动获得大厅展示、快速匹配、好友邀请、跨游戏成就、回放复盘等全部通用能力。框架从架构层面将"公共基础设施"与"游戏专属逻辑"彻底解耦，前者由框架团队集中维护迭代，后者由各游戏开发团队独立负责。

### 1.2.2 目标用户群体与使用场景分析

框架面向三类核心用户群体，各自的使用场景与核心诉求存在显著差异。

**普通玩家**是框架终端服务的直接使用者。其典型场景为：打开App后进入统一大厅，浏览已安装的棋类和牌类游戏，选择与好友创建私人房间或点击"快速匹配"加入陌生对局。核心诉求在于"一个应用、多种体验"——无需为每种棋牌游戏下载独立App，账号资产与好友关系在所有游戏中共享互通，对局记录可在个人中心统一查看。

**游戏开发者**是利用框架接口开发新游戏插件的第三方或内部开发者。其典型场景为：阅读插件接口规范，使用熟悉的编程语言和工具实现 `GameController` 接口与AI决策逻辑，打包提交插件资源包，经框架审核后上架至游戏大厅。核心诉求在于"专注规则、零重复开发"——开发者将全部精力投入游戏规则设计与AI算法优化，无需关心账号认证、网络同步、房间管理、多平台适配等基础设施问题。

**运营方**是框架及插件的运营维护团队。其典型场景为：监控各游戏在线人数与收入数据，策划跨游戏联合活动（如"周末棋类 Festival"），决定新游戏插件的上架/下架/灰度发布，处理玩家举报与异常行为。核心诉求在于"统一管控、灵活运营"——通过一套后台管理系统同时管理多款游戏的内容更新、活动配置和经济参数调整。

### 1.2.3 插件化架构的核心设计哲学：框架提供公共能力，游戏专注规则实现

框架的设计哲学可凝练为八个字：**"框架筑基，插件竞技"**。框架负责构建稳定、通用、可扩展的基础设施层，游戏插件负责在其上实现差异化、富有创意的规则与玩法。这一哲学在需求层面具体体现为三个核心原则。

**接口优先原则**：框架的所有能力均通过明确定义的接口向外暴露，所有需求均通过接口参数和契约来表达。插件开发者仅需理解接口规范即可开发新游戏，无需阅读框架内部源代码。**状态隔离原则**：每局游戏的状态完全封装在 `GameController` 实例内部，框架不介入、不解释游戏状态的具体语义（如"顺子"或"将军"的含义），仅负责传递操作指令与广播状态变更事件。**能力复用原则**：任何可在多游戏间共享的能力（如倒计时器、聊天组件、回放UI、成就触发器）都必须由框架统一实现，禁止各插件自行重复建设。

## 1.3 系统总体描述

### 1.3.1 框架与插件的关系模型：框架作为运行时环境，游戏作为可热插拔资源包

框架与插件之间构成"容器-内容"（Container-Content）关系模型。框架是持续运行的基础平台，提供以下运行时服务：插件生命周期管理（发现、注册、加载、卸载）、客户端大厅渲染与导航、房间创建与匹配调度、玩家操作路由与超时管控、AI托管调度、虚拟资产结算、聊天消息中继、对局记录持久化、回放控制。

游戏插件则是以独立资源包形式存在的可插拔单元，每个插件包含：二进制逻辑文件（实现框架接口的游戏规则控制器）、AI决策模块、UI视图组件及渲染资源（纹理、模型、特效、音效）、国际化翻译文件、参数配置Schema定义。插件可在框架运行过程中动态加载（如玩家首次点击某游戏图标时后台下载资源包）和热卸载（如下架某游戏时清理内存与存储），无需更新框架本体。

### 1.3.2 支持的游戏类型全景清单：棋类、牌类、麻将类及未来扩展

框架在初始版本中明确支持以下7款棋牌游戏，按类别分组如下：

**棋类游戏**（完全信息博弈，无隐藏信息）：五子棋（Gomoku），支持双人对弈及观战，规则目标为在棋盘上先形成五子连线；黑白棋（Reversi/Othello），支持双人对弈，通过翻转对方棋子占领棋盘；围棋（Go），支持双人对弈，包含中国规则、日本规则、韩国规则三种计法配置；象棋（Xiangqi），支持双人对弈，包含标准规则及让子、限时等变体。棋类游戏插件的共同特征是无随机发牌过程，状态完全公开，操作形式为在棋盘网格上落子或移动棋子。

**牌类游戏**（非完全信息博弈，含隐藏信息与下注）：斗地主（Dou Dizhu），支持3人标准模式及4人变体，包含叫牌、出牌、抢地主等核心机制；德州扑克（Texas Hold'em），支持2至10人，包含翻牌前、翻牌圈、转牌圈、河牌圈四轮下注及摊牌比牌。牌类游戏插件的共同特征是存在隐藏手牌信息、随机发牌过程及复杂的下注/筹码流转机制。

**麻将类游戏**：标准麻将（Mahjong），支持4人对局，包含多种地方规则变体（如四川血战到底、广东推倒和）。麻将类游戏插件的显著特征是牌张数量大、胡牌规则组合丰富、存在吃/碰/杠等复杂的牌面操作类型。

以上7款游戏覆盖了棋类（完全信息、无随机）、牌类（非完全信息、有下注）、麻将类（组合规则复杂、地方变体多）三大核心品类，其接口实现将为后续新游戏的接入提供参考范例。框架的插件接口规范在设计时已预留扩展字段（如 `supportedModes` 支持自定义模式列表、`currencySlug` 支持注册新币种），理论上可支持任意棋牌品类乃至非棋牌类回合制桌游的接入。

### 1.3.3 系统边界与外部依赖：账号服务、支付网关、第三方RNG认证、社交平台

框架的系统边界定义为：从客户端游戏大厅的首次渲染开始，到服务端房间管理、对局状态协调、资产结算、数据持久化的完整闭环。以下功能模块位于框架边界之外，以外部依赖形式存在：

**账号认证服务**：框架本身不维护独立的账号注册体系，而是通过OAuth 2.0协议对接手机号/邮箱验证服务、Apple Sign-In、Google Sign-In及主流社交平台（如Facebook、微信）的第三方登录。框架接收外部认证服务返回的身份令牌（Token），将其映射为框架内部统一的用户唯一标识。

**支付网关**：虚拟货币（金币、各游戏专属积分）的充值入口对接外部支付渠道（Apple App Store、Google Play、支付宝、微信支付）。框架负责定义虚拟商品SKU与定价，实际的支付交易、货币转换、退款处理由支付网关完成，框架通过Webhook回调确认到账后向玩家钱包发放对应虚拟资产。

**第三方RNG认证**：框架内建的RNG服务必须接受iTech Labs或GLI等权威机构的周期性审计认证。认证机构对RNG算法的统计分布特性进行测试，通过后颁发认证证书。框架客户端需在游戏内展示RNG认证标识，认证报告摘要可供玩家查阅。RNG服务的具体实现由框架维护，但认证过程依赖外部机构。

**社交平台**：好友关系可通过社交平台导入（如"添加Facebook好友"功能），玩家间可分享战绩卡片至社交网络。这些功能依赖社交平台提供的SDK与API，框架做适配封装。

### 1.3.4 关键假设与约束条件：虚拟资产不可兑换真实货币、中低端设备适配、多地区法规遵从

以下假设与约束条件在后续需求定义中作为前置条件存在，若前提发生变化，相关需求可能需要重新评估：

**FR-CONST-01 虚拟资产封闭性约束**：框架内所有虚拟货币与积分仅限在框架生态内部流通使用，**绝对禁止**提供向真实货币兑换、提现、转账的功能。任何涉及虚拟资产奖励的活动（如比赛奖金）均以游戏内道具或积分的形式发放。此约束出于多地区法律法规合规性要求，是不可突破的红线。

**FR-CONST-02 中低端设备适配约束**：客户端框架须保证在运行内存2GB及以下、Android 8.0 / iOS 13及以上版本的移动设备上流畅运行。游戏大厅界面帧率不低于30FPS，对局操作响应延迟不超过200ms。插件的资源包需经过压缩与分级加载，避免一次性加载导致低端设备内存溢出。

**FR-CONST-03 多地区法规遵从约束**：框架须支持按运营地区（中国大陆、东南亚、北美、欧洲等）差异化启用功能模块。例如，部分地区的法规禁止模拟赌博类游戏展示RNG认证标识以外的任何"技巧提示"；部分地区要求实名认证与防沉迷系统对接。框架的配置系统需支持地区级别的功能开关与参数覆盖。

**FR-ASSUM-01 网络基础设施假设**：假定运营区域内玩家的平均网络延迟不超过150ms（4G/5G或稳定WiFi环境），丢包率不超过2%。在此假设下，框架的对局状态同步采用乐观锁+服务端校验的模式，不实现复杂的延迟隐藏与回滚机制。

## 1.4 用例总览

### 1.4.1 核心参与者定义：普通玩家、房主、观战者、游戏开发者、系统管理员

本系统涉及5类核心参与者（Actor），各自代表与框架交互的不同角色身份。普通玩家（Player）是最基础的参与者，可浏览大厅、创建/加入房间、进行对局、管理个人资产与好友关系。房主（Room Owner）是普通玩家在创建房间后获得的派生身份，拥有房间管理权限。观战者（Spectator）是加入进行中房间的旁观者角色，仅能接收对局状态广播而无操作权限。游戏开发者（Plugin Developer）是开发新游戏插件并通过框架审核流程将其上架的第三方开发者。系统管理员（System Admin）是运营团队成员，负责插件审核、玩家行为监控、系统配置与运营活动管理。值得注意的是，同一自然人可在不同场景中扮演不同参与者角色——例如一名普通玩家可以创建房间成为房主，也可以申请成为游戏开发者提交新插件。

### 1.4.3 用例优先级与实现阶段规划

框架的开发划分为三个里程碑阶段。第一阶段（MVP）聚焦于P0级用例的实现，目标是使框架具备"大厅展示→安装插件→创建房间→进行对局→结算"的完整闭环能力，支持至少2款游戏插件（建议为五子棋与斗地主，分别代表棋类和牌类）的接入。第二阶段（功能完善）在MVP基础上补充P1级用例，包括观战、好友系统、回放器、聊天系统等，并将首批7款游戏插件全部接入。第三阶段（生态扩展）实现P2级用例，开放插件开发者注册与提交审核流程，建立运营后台的完整功能，支持第三方开发者在框架平台上发布新游戏。

## 2. 系统架构与框架设计需求

本章定义通用棋牌游戏框架的总体技术架构与核心子系统设计规范，涵盖客户端-服务端分层结构、网络通信协议、游戏状态同步、数据持久化、插件生命周期管理及服务端核心服务六个维度。框架采用插件化微内核架构模式，以最小核心功能集支撑多种棋牌游戏类型的动态扩展。本章规范为后续第3章游戏插件接口设计提供底层架构约束与技术上下文。

### 2.1 总体技术架构

#### 2.1.1 客户端-服务端分层架构

**FR-FW-ARCH-01** 框架必须采用三层分层架构，从顶至下依次为展示层（Presentation Layer）、业务逻辑层（Business Logic Layer）、数据访问层（Data Access Layer），各层职责划分如下。

| 层级 | 英文名称 | 核心职责 | 关键组件 |
|:---|:---|:---|:---|
| 展示层 | Presentation Layer | 用户界面渲染、用户输入采集、动画与特效呈现、本地化文本显示 | 游戏桌面渲染器、UI管理器、动画引擎、音频管理器 |
| 业务逻辑层 | Business Logic Layer | 游戏规则执行、状态机管理、网络消息处理、插件调度与隔离 | 游戏主控制器、规则引擎、状态机、网络代理、插件容器 |
| 数据访问层 | Data Access Layer | 数据持久化读写、缓存管理、对象存储访问、数据库连接池 | 数据访问对象（DAO）、缓存代理、迁移引擎、连接池管理器 |

上表中的三层架构遵循依赖倒置原则（Dependency Inversion Principle），即上层可依赖下层提供的抽象接口，但禁止下层反向依赖上层。展示层与业务逻辑层之间通过事件总线进行解耦通信，业务逻辑层内部的游戏通用逻辑与游戏特定逻辑通过插件机制进行隔离，数据访问层对所有上层提供统一的数据访问抽象（Data Access Object, DAO），屏蔽底层存储介质的差异。客户端与服务端各自维护独立的三层结构，客户端侧重展示层与轻量级业务逻辑，服务端承载权威业务逻辑与数据持久化。

#### 2.1.2 插件化微内核架构模式

**FR-FW-ARCH-02** 框架核心必须实现微内核（Microkernel）架构模式，内核仅提供插件管理、事件总线、网络通信与数据访问四项基础能力，所有游戏类型逻辑通过插件形式动态加载。

**FR-FW-ARCH-03** 插件（Plugin）定义为符合框架插件接口规范（参见第3章）的独立可加载模块，包含游戏规则实现、资源文件与元数据描述三部分。框架内核不得包含任何特定游戏类型的业务逻辑代码。

**FR-FW-ARCH-04** 微内核必须提供插件注册表（Plugin Registry），维护已加载插件的元数据索引，包括插件ID、版本号、依赖列表、暴露的游戏类型标识符（Game Type ID）及资源路径映射。注册表在运行时动态构建，支持插件的热加载与热卸载。

#### 2.1.3 各层交互协议

**FR-FW-ARCH-05** 层间通信必须遵循严格的接口契约：展示层仅可通过业务逻辑层暴露的Facade接口发起操作请求，禁止直接调用数据访问层。业务逻辑层通过DAO抽象接口与数据访问层交互，禁止在业务逻辑层中编写原生SQL或存储过程调用。

**FR-FW-ARCH-06** 客户端与服务端的数据流向必须满足以下约束：客户端展示层产生的用户操作指令经客户端业务逻辑层校验后，通过网络代理序列化为网络消息发送至服务端；服务端业务逻辑层接收消息后执行权威规则计算，生成状态变更事件，经由网络代理广播至房间内的所有客户端；客户端接收状态更新后，由展示层渲染最终效果。数据流向为单向环形，任何节点不得跳过层级或逆向通信。

### 2.2 网络通信子系统

#### 2.2.1 传输层协议选型

**FR-FW-NET-01** 客户端与服务端的实时通信必须采用WebSocket长连接协议（RFC 6455），支持全双工、低延迟的消息传输。连接建立必须通过TLS 1.3加密，证书由框架网关服务统一管理。

**FR-FW-NET-02** HTTP/HTTPS协议仅用于辅助性接口，包括用户认证、配置获取、排行榜查询、静态资源下载四类场景，不得用于游戏实时消息的传输。

**FR-FW-NET-03** 网关服务必须支持WebSocket连接的负载均衡分发，基于玩家ID的哈希一致性算法将同一玩家的连接始终路由至同一逻辑服务实例，避免会话漂移。

#### 2.2.2 消息编解码格式规范

**FR-FW-NET-04** 所有WebSocket传输的网络消息必须采用统一协议格式，由协议头（Header）与协议体（Payload）两部分组成。协议体必须使用Protocol Buffers（Protobuf）v3编码。

| 字段名 | 类型 | 长度（字节） | 说明 | 示例值 |
|:---|:---|:---:|:---|:---|
| magic | uint16 | 2 | 协议魔数，固定值0x47 0x46（"GF"），用于快速识别有效协议包 | 0x4746 |
| version | uint8 | 1 | 协议版本号，当前版本为1 | 0x01 |
| msg_type | uint16 | 2 | 消息类型标识，定义于全局消息类型枚举表 | 0x1001（游戏操作） |
| seq_num | uint32 | 4 | 消息序列号，单调递增，用于消息去重与顺序校验 | 1048576 |
| timestamp | uint64 | 8 | 消息生成时间戳（毫秒级Unix时间） | 1715000000000 |
| payload_len | uint32 | 4 | Protobuf编码后的Payload字节长度 | 256 |
| payload | bytes | 变长 | Protobuf序列化的消息体，具体结构由msg_type决定 | — |

上表定义的消息格式中，协议头固定长度为21字节，采用大端字节序（Big-Endian）编码。msg_type字段的高字节表示消息大类（如0x01为系统类、0x10为游戏操作类、0x20为状态同步类），低字节表示具体消息子类型。序列号由发送方维护，每发送一条消息递增1，接收方通过序列号检测消息丢失与乱序。timestamp字段用于网络延迟计算与消息过期判定，超过30秒的消息将被丢弃。

#### 2.2.3 心跳检测与连接保活

**FR-FW-NET-05** 框架必须实现心跳检测机制以维持长连接活性。客户端必须每30秒向服务端发送一次心跳消息（Ping），服务端收到后必须在2秒内返回心跳响应（Pong）。

**FR-FW-NET-06** 连接断线判定策略采用超时计数器模式：若客户端连续3次未在5秒超时窗口内收到服务端Pong响应，则判定连接中断，触发断线重连流程。服务端若连续3次未收到客户端心跳，则主动关闭WebSocket连接并释放相关会话资源。

#### 2.2.4 消息路由与分发机制

**FR-FW-NET-07** 消息路由必须以房间ID（Room ID）作为隔离维度，同一房间内的消息仅分发给该房间的参与者。框架必须支持三种消息投递模式：单播（Unicast，定向发送至指定玩家）、组播（Multicast，发送至房间内指定子集）、广播（Broadcast，发送至房间内全部玩家）。

**FR-FW-NET-08** 网关服务必须维护路由表（Routing Table），记录房间ID到参与者连接句柄的映射关系。消息分发时，根据消息头部中的目标房间ID查询路由表，按投递模式选择目标连接列表，批量发送消息。

#### 2.2.5 断线重连与状态恢复

**FR-FW-NET-09** 框架必须实现重连令牌（Reconnect Token）机制。玩家连接建立时，服务端生成包含玩家ID、会话ID、时间戳、房间ID的加密令牌（JWT格式，RS256签名），下发至客户端。断线后重连时，客户端在WebSocket握手阶段携带该令牌，服务端验证后恢复原有会话上下文。

**FR-FW-NET-10** 房间服务必须在玩家断线后保留其状态快照（State Snapshot）至少300秒。状态快照包含玩家当前游戏状态、手牌/棋子信息、操作历史索引。玩家重连后，服务端对比客户端当前状态版本号与服务端权威版本号，若版本一致则发送增量更新，若版本偏差超过10个操作则发送完整状态快照。

### 2.3 游戏状态同步机制

#### 2.3.1 权威状态模型

**FR-FW-SYNC-01** 框架必须采用服务端权威（Server-Authoritative）状态模型：服务端作为唯一的状态权威源，所有游戏状态变更均由服务端业务逻辑层计算后下发，客户端仅负责接收状态更新并渲染，不得自主修改游戏核心状态。

**FR-FW-SYNC-02** 客户端可基于本地预测（Client-Side Prediction）机制在用户操作后立即更新本地视图以提升响应性，但必须在收到服务端权威状态更新后以服务端状态为准进行校正。

#### 2.3.2 状态同步策略

**FR-FW-SYNC-03** 框架必须根据游戏类型的特性差异采用不同的状态同步策略，棋类游戏与牌类游戏的同步方式存在本质区别，具体对比如下。

| 对比维度 | 棋类状态同步（如中国象棋、围棋） | 牌类混合同步（如斗地主、德州扑克） |
|:---|:---|:---|
| 同步模式 | 完整棋盘快照同步 | 操作指令+周期性校验 |
| 状态表示 | 二维棋盘矩阵（如9×10数组），每格记录棋子类型与归属 | 不可见区域仅同步哈希摘要，可见区域同步具体牌面 |
| 同步频率 | 每步操作后全量广播 | 操作指令实时广播+每5回合全量校验 |
| 数据量 | 单次约200-500字节 | 单次操作约20-100字节，校验包约500字节 |
| 防作弊机制 | 全量状态可任意客户端校验 | 服务端权威校验，客户端仅可见授权信息 |
| 断线恢复 | 直接下发当前完整棋盘 | 重发当前局操作日志+重新计算可见状态 |

棋类游戏采用状态同步的原因在于棋盘信息对所有参与者完全透明，每一步操作后棋盘状态可完整表示为一个二维矩阵，数据量可控且利于客户端校验。牌类游戏采用混合同步的原因在于手牌信息具有私有性，若采用全量同步将导致手牌泄露，因此仅同步操作指令（如出牌、跟注、弃牌），配合周期性全量校验确保客户端与服务端状态一致。不可见牌区（如牌堆、其他玩家手牌）仅同步哈希值，客户端可校验一致性但无法获取具体牌面。

#### 2.3.3 状态快照与增量同步协议

**FR-FW-SYNC-04** 框架状态同步必须采用双模式设计：关键帧（Key Frame）全量同步与操作指令（Delta）增量更新相结合。关键帧间隔默认为每10次操作或每60秒（以先到者为准）触发一次。

**FR-FW-SYNC-05** 增量更新消息必须包含以下字段：基准版本号（base_version）、目标版本号（target_version）、操作指令数组（operations，每条含操作类型、操作者、操作参数）、时间戳数组（timestamps，记录每条操作的精确执行时间）。

**FR-FW-SYNC-06** 当客户端版本号与服务端版本号偏差超过阈值（默认10个操作）时，服务端必须发送全量状态快照替代增量更新，确保客户端状态最终一致性。

#### 2.3.4 回放与观战的状态序列化

**FR-FW-SYNC-07** 框架必须定义统一的操作日志序列化格式，支持对局回放与观战功能。每条操作日志记录必须包含以下不可变字段：全局序列号（global_seq）、操作类型（op_type）、操作者ID（player_id）、操作参数（op_params，JSON格式）、服务端执行时间戳（server_ts，毫秒级Unix时间戳）、状态哈希（state_hash，操作执行后完整状态的SHA-256摘要）。

**FR-FW-SYNC-08** 操作日志序列必须以追加写（Append-Only）方式持久化至对象存储，文件格式采用按局分割的Line-Delimited JSON（LDJSON）结构，单局日志文件大小不得超过50MB。

### 2.4 数据持久化层

#### 2.4.1 持久化数据三级分类

**FR-FW-DB-01** 框架持久化数据必须按照访问热度和实时性要求分为三级：热数据（Redis缓存，TTL默认300秒）、温数据（关系型数据库，持久化存储）、冷数据（对象存储归档日志，长期保留）。热数据用于高频读取的会话状态与排行榜缓存，温数据用于用户资料、对局记录、财务流水等核心业务数据，冷数据用于操作日志回放、审计追踪与分析报表。

#### 2.4.2 数据库选型与分库分表

**FR-FW-DB-02** 温数据持久化必须采用支持水平分片的关系型数据库（如MySQL 8.0+或PostgreSQL 15+）。用户数据表必须按用户ID（user_id）取模分片，分片数量默认1024，路由规则为shard = hash(user_id) % 1024。

**FR-FW-DB-03** 房间数据表必须按房间ID（room_id）分片，分片算法采用一致性哈希（Consistent Hashing），虚拟节点数默认为160个，以支持房间数据的动态迁移与负载均衡。

**FR-FW-DB-04** 对局记录表必须采用复合分片策略：按对局结束时间进行范围分区（每月一个分区），同时按用户ID进行哈希分片，以优化时间范围查询与单用户查询性能。

#### 2.4.3 缓存一致性保障

**FR-FW-DB-05** 框架缓存层必须采用写穿（Write-Through）策略：所有数据写入操作先更新数据库，再同步更新或失效缓存。禁止绕过数据库直接写入缓存。

**FR-FW-DB-06** Redis缓存必须设置TTL过期时间，热数据默认TTL为300秒，排行榜数据默认TTL为60秒。框架必须实现主动失效机制，当数据库数据发生变更时，通过发布-订阅通道（Redis Pub/Sub）通知所有缓存节点使对应Key失效。

#### 2.4.4 数据迁移与版本兼容

**FR-FW-DB-07** 插件数据结构变更必须遵循向前兼容（Forward Compatible）原则：新版本的插件必须能够读取旧版本写入的数据，不得更改已存在字段的语义或数据类型。新增字段必须带有默认值或标记为可选（optional）。

**FR-FW-DB-08** 框架必须定义数据迁移脚本规范：迁移脚本以版本号命名（格式为V{主版本}_{次版本}_{补丁版本}__{描述}.sql），按顺序执行，每条迁移脚本必须在10秒内完成，超时自动回滚。迁移脚本的执行状态必须记录在数据库迁移日志表（schema_migration_log）中，包含脚本名称、执行时间、执行结果、校验和（SHA-256）。

### 2.5 插件生命周期管理

#### 2.5.1 插件加载与注册机制

**FR-FW-PLUGIN-01** 框架必须在启动时扫描配置的插件目录（默认路径为./plugins），识别符合命名约定的插件包文件（格式为{plugin_id}-{version}.plugin，本质为ZIP压缩包）。

**FR-FW-PLUGIN-02** 插件加载必须经过元数据校验阶段：框架解析插件包内的manifest.json文件，校验字段完整性（必须包含plugin_id、version、min_framework_version、game_types、dependencies），校验插件ID全局唯一性，校验依赖版本兼容性（采用语义化版本约束解析）。

**FR-FW-PLUGIN-03** 元数据校验通过后，框架必须将插件注册至插件注册表，将插件支持的游戏类型映射至游戏路由表，使对应游戏类型的对局请求可被正确路由至该插件实例。

#### 2.5.2 插件运行时隔离

**FR-FW-PLUGIN-04** 每个插件实例必须在独立的沙箱（Sandbox）环境中运行，沙箱必须限制以下资源配额：堆内存不超过512MB、CPU使用率不超过单核的50%、出站网络连接仅允许访问框架配置的白名单地址。

**FR-FW-PLUGIN-05** 框架必须实现插件异常熔断机制：若单个插件在60秒内崩溃超过3次，框架自动标记该插件为故障状态，停止为其分配新对局，已进行中的对局优雅迁移至备用实例或终止并赔偿。

#### 2.5.3 插件热更新与版本兼容

**FR-FW-PLUGIN-06** 框架必须支持插件热更新（Hot Update），采用蓝绿部署（Blue-Green Deployment）模式：新版本插件以绿色实例并行加载，框架逐步将新对局请求路由至绿色实例，待存量蓝色实例的对局全部结束后卸载旧版本。

**FR-FW-PLUGIN-07** 插件版本号必须采用语义化版本控制（SemVer），格式为MAJOR.MINOR.PATCH。MAJOR版本号变更表示存在不兼容的API修改，框架必须拒绝加载与当前MAJOR版本不匹配的依赖插件。

#### 2.5.4 插件间通信与事件总线

**FR-FW-PLUGIN-08** 框架必须提供跨插件事件总线（Event Bus），基于发布-订阅（Pub-Sub）模式实现。事件格式必须包含：事件类型（event_type，格式为{plugin_id}.{domain}.{action}）、事件源（source_plugin_id）、时间戳（timestamp）、载荷（payload，Protobuf Any类型）。

**FR-FW-PLUGIN-09** 插件间事件通信必须通过事件总线中转，禁止插件间的直接方法调用，以确保松耦合与隔离性。事件总线必须支持同步与异步两种投递模式，异步模式默认采用消息队列缓冲。

#### 2.5.5 插件卸载与资源回收

**FR-FW-PLUGIN-10** 框架必须实现插件的优雅关闭（Graceful Shutdown）流程：卸载请求发出后，框架首先停止向该插件路由新请求，等待已有对局完成或超时（最长300秒），然后调用插件的on_unload()钩子函数，最后释放插件占用的内存与句柄资源。

以下表格完整描述插件从加载到卸载的全生命周期状态转换。

| 阶段 | 状态名称 | 触发条件 | 执行动作 | 异常处理 |
|:---|:---|:---|:---|:---|
| 加载 | DISCOVERED | 框架扫描到插件包文件 | 解析文件路径与基础信息 | 文件损坏则记录日志并跳过 |
| 加载 | VALIDATING | 开始元数据校验 | 解析manifest.json，校验字段与依赖 | 校验失败则标记为INVALID并记录原因 |
| 加载 | REGISTERED | 校验通过 | 写入插件注册表，映射游戏类型路由 | 注册表写入失败则回滚至DISCOVERED |
| 加载 | ACTIVE | 完成初始化 | 调用on_load()钩子，分配运行时资源 | 初始化失败则标记为FAILED，触发卸载 |
| 运行 | RUNNING | 正在处理对局 | 接收游戏请求，执行游戏规则逻辑 | 运行时异常触发熔断计数器 |
| 运行 | DEGRADED | 熔断触发 | 停止分配新对局，存量对局继续运行 | 持续异常则强制迁移至TERMINATING |
| 更新 | UPDATING | 热更新请求 | 启动新版本绿色实例，逐步切换流量 | 更新失败则回滚至ACTIVE状态 |
| 卸载 | TERMINATING | 卸载请求或版本替换 | 停止新请求路由，等待存量对局结束 | 超时300秒则强制终止 |
| 卸载 | UNLOADED | 资源释放完成 | 调用on_unload()，回收内存与句柄 | 资源泄漏则记录告警日志 |

上表涵盖插件生命周期的9个状态与8个阶段转换。正常流程经历从DISCOVERED到ACTIVE再到RUNNING的加载路径，更新时通过UPDATING状态实现蓝绿切换，卸载时经过TERMINATING状态确保graceful shutdown。每个状态均定义明确的触发条件与异常处理策略，确保插件生命周期在任何环节出现错误时均有明确的回退或降级路径。

### 2.6 服务端核心服务

#### 2.6.1 网关服务

**FR-FW-SVC-01** 网关服务（Gateway Service）为客户端的统一入口，承担以下核心职责：WebSocket长连接管理（建立、维持、优雅关闭）、基于一致性哈希的负载均衡分发、TLS 1.3连接的SSL终结、基于令牌桶算法的DDoS请求速率限制（默认每IP每秒不超过100个连接请求）。网关服务不得处理任何游戏业务逻辑，仅负责连接管理与消息转发。

**FR-FW-SVC-02** 网关服务必须暴露管理接口GET /health，返回当前连接数、上游服务可用状态、近一分钟QPS指标，响应格式为JSON，HTTP状态码200表示健康、503表示过载。

#### 2.6.2 匹配服务

**FR-FW-SVC-03** 匹配服务（Matchmaking Service）必须实现基于ELO评分与等待时间的多维匹配算法。匹配评分函数定义为：match_score = |elo_a − elo_b| + k × |wait_a − wait_b|，其中k为时间权重系数（默认0.5）。系统优先匹配ELO差距小且等待时间相近的玩家。

**FR-FW-SVC-04** 匹配服务必须实现AI填充策略：当真人玩家等待时间超过配置阈值（默认60秒）时，自动匹配AI玩家填充对局。AI玩家由AI对战引擎服务动态生成，AI的ELO评分根据等待玩家水平动态调整。

**FR-FW-SVC-05** 匹配队列必须支持弹性伸缩：根据各游戏类型的排队长度自动调整匹配协程数量，每个游戏类型最小1个、最大100个匹配工作者，扩容步长为10，缩容冷却期为60秒。

#### 2.6.3 房间服务

**FR-FW-SVC-06** 房间服务（Room Service）必须实现房间状态机，包含以下状态与转换：CREATING（创建中）→ WAITING（等待玩家）→ PLAYING（游戏中）→ PAUSED（暂停，玩家断线触发）→ ENDED（结束）→ DESTROYED（销毁）。状态转换必须由服务端事件驱动，任何状态转换必须记录操作日志。

**FR-FW-SVC-07** 房间服务必须管理玩家生命周期：玩家进入房间时分配座位号与角色，断线时触发PAUSED状态或启动AI托管，重连时恢复座位状态，离开时释放资源。空房间在等待状态下持续300秒后自动销毁。

#### 2.6.4 排行榜与成就服务

**FR-FW-SVC-08** 排行榜服务（Leaderboard Service）必须支持多维排名计算：总胜率榜（按胜场/总场数排序）、ELO等级分榜（按ELO评分排序）、连胜榜（按最大连胜场次排序）。排行榜数据通过定时任务（每5分钟）从数据库聚合计算并写入Redis缓存。

**FR-FW-SVC-09** 成就服务（Achievement Service）必须实现事件监听模式：订阅游戏对局结束事件，根据成就定义规则（如累计胜场达到100场、连续登录7天等）判定成就达成，通过消息队列异步发放成就奖励，奖励结果写入用户账户流水。

#### 2.6.5 AI对战引擎服务

**FR-FW-SVC-10** AI对战引擎服务（AI Engine Service）必须管理AI进程池，采用预创建+动态扩容的混合模式。进程池默认预创建10个AI进程，每个进程最大并发处理1个对局，当对局请求超过池容量时按50%步长动态扩容，最大不超过100个进程。

**FR-FW-SVC-11** AI思考时间必须受限于配置阈值：简单难度不超过500毫秒、中等难度不超过2000毫秒、困难难度不超过5000毫秒。超时未返回决策时，AI引擎必须返回默认安全操作（如棋类随机合法走子、牌类跟注/过牌）。

**FR-FW-SVC-12** AI引擎必须实现结果缓存机制：相同的游戏状态（以状态哈希为键）在30秒内重复请求时，直接返回缓存决策，避免重复计算。缓存命中率不得低于80%，低于该阈值时触发告警。

## 3. 游戏插件接口规范

本章定义通用棋牌游戏框架的插件化技术契约，是所有具体游戏（德州扑克、斗地主、象棋、麻将等）接入框架的必遵规范。插件通过实现本章定义的接口，自动获得大厅展示、房间管理、AI托管、回放记录等全部框架级能力。所有接口采用面向对象抽象描述，实际实现时可翻译为具体编程语言的 `interface`、`protocol` 或 REST API 消息契约。

### 3.1 插件元数据与注册

#### 3.1.1 元数据结构 PluginMeta

框架启动时扫描所有已安装插件，调用注册方法获取元数据并存入游戏注册表。**FR-PLUGIN-01-01** 要求插件必须提供一个 `PluginMeta` 对象，字段规范如下表所示。

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `gameId` | `String` | 是 | 全局唯一标识符，仅含小写字母、数字和下划线，如 `"texas_holdem"` |
| `gameName` | `String` | 是 | 显示名称的国际化 Key，如 `"game_texas_name"`，由 `context.localize()` 解析 |
| `category` | `Enum` | 是 | 游戏分类，取值为 `mahjong`（麻将）、`card`（牌类）、`board`（棋类）、`other`（其他） |
| `minPlayers` | `Int` | 是 | 最少玩家人数，≥1；值为 1 时表示允许单人模式并由 AI 补位 |
| `maxPlayers` | `Int` | 是 | 最大玩家人数，须满足 `maxPlayers ≥ minPlayers` |
| `icon` | `AssetRef` | 是 | 大厅网格展示的图标资源引用，支持本地路径或远程 URL |
| `description` | `String` | 否 | 规则简介的国际化 Key，用于大厅详情面板展示 |
| `version` | `String` | 是 | 插件版本号，遵循 SemVer 格式（如 `"1.2.3"`），用于热更新兼容性校验 |
| `supportedModes` | `List<String>` | 是 | 支持的对局模式列表，如 `["cash", "tournament", "sit_and_go"]` |
| `currencySlug` | `String` | 否 | 本游戏专用积分币种代号，如 `"chips"`、`"beans"`；未指定时默认使用框架通用金币 |

`PluginMeta` 作为框架识别游戏插件的首要信息源，其 `gameId` 具有全局唯一性约束，重复注册将导致后加载的插件被忽略并写入警告日志。`category` 字段决定框架在大厅中分配的分类标签页归属，`supportedModes` 则影响房间创建界面可供选择的模式选项。`version` 字段在框架执行热更新时发挥关键作用：框架比对本地缓存版本与远程发布版本，仅当远程版本号更高时触发增量下载。

#### 3.1.2 注册入口函数 registerPlugin

**FR-PLUGIN-01-02** 规定插件须导出标准注册入口函数，其完整方法签名与规范如下。

**方法签名**：`registerPlugin(context: FrameworkContext): PluginMeta`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `context` | `FrameworkContext` | 框架运行时上下文对象，封装插件可用的全部框架能力 |

**返回值**：`PluginMeta` — 插件元数据对象，字段规范见 3.1.1。

**异常**：
- `IllegalArgumentException` — `context` 为 `null` 或框架版本不兼容时抛出
- `PluginRegistrationException` — 元数据校验失败（如必填字段缺失、`gameId` 格式非法）时抛出

`FrameworkContext` 作为插件与框架交互的唯一合法通道，暴露以下能力清单：日志记录（`context.logger`）、键值存储（`context.storage`）、网络请求（`context.http`）、事件总线（`context.eventBus`）、国际化解析（`context.localize(key)`）以及随机数服务（`context.rng`）。插件在 `registerPlugin` 执行期间可调用上述能力完成初始化配置，但不应在此阶段执行耗时操作（如预加载大量资源），以避免阻塞框架启动流程。

#### 3.1.3 插件卸载方法 unregister

**FR-PLUGIN-01-03** 要求插件提供 `unregister()` 方法，用于释放占用的内存、移除事件监听器、关闭未完成的网络请求。框架在以下时机调用此方法：插件热更新前卸载旧版本、玩家手动移除游戏、框架关闭时的资源清理。插件实现方须确保该方法幂等——多次调用不产生副作用。卸载完成后，插件的 `gameId` 从游戏注册表中移除，对应的大厅入口不再显示。

### 3.2 游戏参数设置接口

#### 3.2.1 声明式 UI 描述 Schema

当房主创建房间时，框架需动态渲染该游戏的专属设置面板。**FR-PLUGIN-02-01** 要求插件提供 `getSettingsSchema(): UISchema` 方法，返回一个平台无关的声明式 UI 描述。`UISchema` 支持五种字段类型：`number`（数值输入）、`radio`（单选）、`switch`（开关）、`slider`（滑块）、`select`（下拉选择）。

以下给出完整的 `UISchema` JSON 示例，包含 `number` 和 `radio` 两种字段类型：

```json
{
  "fields": [
    {
      "key": "small_blind",
      "type": "number",
      "label": "setting_small_blind",
      "default": 10,
      "min": 1,
      "max": 10000,
      "step": 10
    },
    {
      "key": "player_count",
      "type": "radio",
      "label": "setting_player_count",
      "options": [
        { "value": 2, "label": "players_2" },
        { "value": 6, "label": "players_6" },
        { "value": 9, "label": "players_9" }
      ],
      "default": 6
    },
    {
      "key": "allow_spectators",
      "type": "switch",
      "label": "setting_spectators",
      "default": false
    },
    {
      "key": "turn_timeout",
      "type": "slider",
      "label": "setting_timeout",
      "default": 30,
      "min": 15,
      "max": 120,
      "step": 15,
      "unit": "seconds"
    },
    {
      "key": "ai_difficulty",
      "type": "select",
      "label": "setting_ai_level",
      "options": [
        { "value": "off", "label": "ai_off" },
        { "value": "easy", "label": "ai_easy" },
        { "value": "hard", "label": "ai_hard" }
      ],
      "default": "easy"
    }
  ]
}
```

框架客户端负责将上述 Schema 渲染为平台原生 UI 组件。所有 `label` 字段均为国际化 Key，由框架在渲染时通过 `context.localize()` 解析为当前语言的显示文本。`key` 字段作为参数唯一标识，将连同玩家设定的值一并传递给后续的房间配置和游戏控制器。

#### 3.2.2 参数校验函数 validateSettings

**FR-PLUGIN-02-02** 要求插件提供参数校验方法，在房主点击"创建"时由框架调用。

**方法签名**：`validateSettings(settings: Map<String, Any>): ValidationResult`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `settings` | `Map<String, Any>` | 玩家设置参数键值对，Key 对应 Schema 中的 `key`，Value 为玩家输入值 |

**返回值**：`ValidationResult` 对象，包含以下字段：
- `isValid: Boolean` — 校验是否全部通过
- `errors: List<FieldError>` — 错误列表，每个 `FieldError` 包含 `field`（字段 Key）、`code`（错误码，如 `ERR_RANGE_EXCEEDED`）、`message`（国际化错误信息 Key）

**异常**：`IllegalArgumentException` — `settings` 为 `null` 时抛出。

若 `isValid` 为 `false`，框架将在对应输入控件旁渲染错误提示，阻止房间创建直至所有错误消除。插件应在此方法中执行完整的业务规则校验，例如确保 `"small_blind" ≤ "big_blind"`、玩家数量与所选模式兼容等。

### 3.3 游戏逻辑控制器接口

#### 3.3.1 控制器工厂方法

**FR-PLUGIN-03-01** 规定插件须提供控制器工厂方法，每次房间开始对局时由框架调用。

**方法签名**：`createGameController(roomConfig: RoomConfig): GameController`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `roomConfig` | `RoomConfig` | 房间完整配置，包含 `settings`（经校验的玩家设置参数）、`players`（玩家对象列表，含 ID 与座位号）、`aiConfig`（AI 补位配置，含难度与数量）、`mode`（对局模式标识）、`roomId`（房间唯一标识） |

**返回值**：`GameController` 实例，驱动该局游戏的全生命周期状态流转。

**异常**：`IllegalArgumentException` — `roomConfig` 为 `null` 或玩家数量不在 `[minPlayers, maxPlayers]` 范围内时抛出。

`RoomConfig` 中的 `settings` 字段即 3.2.1 中 Schema 定义经玩家配置后的键值对，`players` 列表按座位顺序排列，空座位由 `null` 占位以便 AI 后续补位。工厂方法须返回一个全新的控制器实例，确保每局游戏状态完全隔离。

#### 3.3.2 核心方法清单

**FR-PLUGIN-03-02** 要求 `GameController` 至少实现以下核心方法。

| 方法签名 | 返回值 | 说明 | 异常 |
|---|---|---|---|
| `start(): void` | — | 初始化游戏状态，触发开局流程（洗牌、发牌、确定庄家等），并通过事件总线广播 `onGameStarted` | `IllegalStateException` — 控制器已处于运行状态时调用 |
| `getCurrentState(): GameState` | `GameState` | 返回当前完整游戏状态快照，用于断线重连或观战者同步 | 无 |
| `getActivePlayer(): String \| null` | `String \| null` | 返回当前应行动的玩家 ID；无活跃玩家（如结算阶段）时返回 `null` | 无 |
| `submitAction(playerId: String, action: Action): void` | — | 接收玩家操作，由控制器验证合法性并执行状态转移；非法操作抛出异常 | `IllegalArgumentException` — 非当前活跃玩家提交；`InvalidActionException` — 操作不合法 |
| `getLegalActions(playerId: String): List<Action>` | `List<Action>` | 返回指定玩家当前可选的所有合法操作列表，供 UI 渲染操作按钮 | `IllegalArgumentException` — `playerId` 不在牌局中 |
| `endGame(): void` | — | 强制结束游戏（如房主解散房间），触发 `onGameFinished` 事件并释放资源 | 无 |

`GameState` 为插件自定义的游戏状态对象，须支持深拷贝以确保 `getCurrentState()` 返回的快照不会被外部修改污染控制器内部状态。`submitAction()` 是驱动游戏状态机的核心入口：控制器接收操作后执行规则校验、状态转移、胜负判定，并通过事件总线广播变更结果。`getLegalActions()` 返回的列表直接影响玩家操作面板的可用选项，UI 层仅渲染返回值中的操作类型。

#### 3.3.3 事件总线协议

**FR-PLUGIN-03-03** 规定控制器须通过 `FrameworkContext.eventBus` 广播状态变更事件，而**不得直接调用 UI 渲染函数**。框架大厅侧监听这些事件来驱动 UI 刷新、播放动画、记录日志。标准事件格式如下表所示。

| 事件名 | 触发时机 | Payload 结构 | 说明 |
|---|---|---|---|
| `onGameStarted` | `start()` 完成初始化后 | `{ gameId: String, roomId: String, players: Player[], dealerSeat: Int }` | 游戏正式开始，UI 应切换至牌桌界面 |
| `onStateUpdated` | 任何游戏状态变更后 | `{ gameId: String, stateDiff: Object, timestamp: Long }` | 通用状态更新，如公共牌翻开、玩家筹码变动；`stateDiff` 包含变更的增量数据 |
| `onPlayerTurn` | 轮到某玩家行动时 | `{ gameId: String, playerId: String, legalActions: Action[], timeoutSec: Int }` | 通知指定玩家操作，附带合法操作列表与倒计时秒数 |
| `onRoundEnd` | 单轮/单局牌结束后 | `{ gameId: String, roundResult: RoundResult, winners: String[], potDistribution: Map<String, Long> }` | 一轮结算，包含赢家列表与筹码分配明细 |
| `onGameFinished` | 整场游戏结束后 | `{ gameId: String, finalResult: FinalResult, rankings: PlayerRanking[] }` | 终局事件，包含最终排名与奖励分配；触发后控制器进入只读状态 |

事件总线采用发布-订阅模式，控制器作为发布方不感知订阅方数量。`onPlayerTurn` 中的 `timeoutSec` 与房间设置中的 `turn_timeout` 对应，框架倒计时服务据此启动计时器；超时未操作则触发框架统一超时处理逻辑（调用 `onTimeout()` 或切换 AI 托管）。`onGameFinished` 触发后，控制器不再接受新的玩家操作，`submitAction()` 调用将抛出 `IllegalStateException`。

### 3.4 AI 控制器接口

#### 3.4.1 AI 工厂方法

**FR-AI-01-01** 要求插件提供 AI 控制器工厂方法，支持不同难度级别。

**方法签名**：`createAIController(difficulty: String, gameController: GameController): AIController`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `difficulty` | `String` | AI 难度级别，框架预定义值为 `"easy"`、`"hard"`；插件可扩展自定义级别 |
| `gameController` | `GameController` | 当前对局的逻辑控制器实例，AI 通过其 `getCurrentState()` 和 `getLegalActions()` 获取决策所需信息 |

**返回值**：`AIController` 实例，每个被 AI 托管的座位对应一个独立实例。

**异常**：`IllegalArgumentException` — `difficulty` 为插件不支持的级别，或 `gameController` 为 `null` 时抛出。

`gameController` 的注入使 AI 能够读取完整的游戏状态并查询合法操作集合，但 AI 控制器**不得**直接调用 `submitAction()`——AI 决策须通过 3.4.2 定义的回调模式返回，由框架统一转发至控制器，以确保操作来源可追溯。

#### 3.4.2 AI 决策方法与超时约束

**FR-AI-01-02** 规定 AI 决策采用异步回调模式。

**方法签名**：`requestAction(playerId: String, gameState: GameState, legalActions: List<Action>, callback: (Action) => void): void`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `playerId` | `String` | 当前需 AI 代操作的玩家 ID |
| `gameState` | `GameState` | 当前游戏状态快照（与 `getCurrentState()` 返回值等价） |
| `legalActions` | `List<Action>` | 该玩家当前可选的合法操作列表 |
| `callback` | `(Action) => void` | 决策结果回调函数，AI 须通过此回调返回选定的 `Action` |

**返回值**：无（异步回调模式）。

**异常**：`TimeoutException` — AI 未在规定的超时时间内调用 `callback` 时由框架抛出。

框架调用 `requestAction()` 后启动计时器，超时时间取自房间设置中的 `turn_timeout`（通常比人类玩家略短或相等，确保对局流畅）。AI 必须在时限内通过 `callback` 返回一个存在于 `legalActions` 列表中的操作。超时未响应时，框架自动从 `legalActions` 中选取默认操作（通常为第一项）提交，并记录日志。

#### 3.4.3 AI 操作标识与托管状态展示

**FR-AI-01-03** 要求 AI 返回的操作须携带 `source: "ai"` 标记，以便框架在 UI 层面展示"托管中"图标。该标记在 `Action` 对象的元数据字段中设置（详见 3.6.1 的 `Action` JSON Schema）。当玩家主动开启托管或因超时/断线被系统自动接管时，框架在对应座位头像旁显示机器人图标，并向房间内其他玩家广播托管状态变更事件。托管状态下，AI 代打的每个操作在操作日志中均保留 `source: "ai"` 标记，确保回放时可区分人类操作与 AI 操作。

### 3.5 资源与渲染接口

#### 3.5.1 游戏视图组件

**FR-PLUGIN-05-01** 要求插件提供 `getGameView(): ViewComponent` 方法，返回游戏专属的主视图组件。框架在进入游戏房间后将其嵌入主界面的预留区域。该组件内部完全由插件控制，负责绘制牌面、棋子、操作按钮等游戏专属内容。框架提供安全区（Safe Area）Insets 信息，插件须确保关键 UI 元素（如操作按钮、玩家信息）不被系统状态栏、刘海或导航栏遮挡。

#### 3.5.2 异步资源加载

**FR-PLUGIN-05-02** 规定插件所有资源（图片纹理、音效、动画、字体）必须通过框架的 `AssetLoader` 异步加载。**方法签名**：`AssetLoader.load(assetRef: AssetRef): Promise<Resource>`。`AssetLoader` 提供统一缓存与更新机制：首次加载时从远程 CDN 或本地包体获取并缓存至磁盘；框架在执行热更新时比对资源清单的哈希值，仅下载变更文件。插件须在 `PluginMeta` 中声明资源清单（`assets` 字段，含资源标识、路径、CRC32 校验值），框架在游戏启动前或后台空闲时预加载高频资源。

#### 3.5.3 多分辨率适配

**FR-PLUGIN-05-03** 要求插件视图实现自适应布局，支持三种典型设备形态：手机竖屏（宽度 375dp–414dp）、平板横屏（宽度 768dp–1024dp）、PC 窗口（宽度 1024dp 以上）。框架提供基准设计尺寸（Reference Design Size，默认 1920×1080）和当前视口尺寸，插件据此计算缩放比例并调整布局。关键适配要求包括：牌桌区域保持居中且比例不变形，操作按钮在触屏设备上最小热区不小于 44dp，PC 端额外支持鼠标悬停提示与键盘快捷键。

### 3.6 回放与序列化接口

#### 3.6.1 操作序列化格式

**FR-PLUGIN-06-01** 要求每个 `Action` 对象支持序列化为通用 JSON 格式，包含操作类型、玩家 ID、数据及时间戳。以下给出完整的 `Action` JSON Schema 示例：

```json
{
  "type": "bet",
  "playerId": "player_12345",
  "data": {
    "amount": 100,
    "betType": "raise"
  },
  "source": "human",
  "timestamp": 1703001600000,
  "seqNum": 42,
  "metadata": {
    "latencyMs": 156,
    "clientVersion": "2.1.0"
  }
}
```

字段说明如下表所示。

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | `String` | 是 | 操作类型标识，如 `"bet"`、`"fold"`、`"move"`，由插件定义 |
| `playerId` | `String` | 是 | 执行操作的玩家唯一标识 |
| `data` | `Object` | 是 | 操作具体数据，结构因 `type` 而异，插件自定义 Schema |
| `source` | `String` | 是 | 操作来源，取值 `"human"`（人类玩家）或 `"ai"`（AI 托管） |
| `timestamp` | `Long` | 是 | 操作发生时的 Unix 毫秒时间戳，由服务端统一赋值 |
| `seqNum` | `Int` | 是 | 全局单调递增序列号，确保操作顺序严格一致 |
| `metadata` | `Object` | 否 | 附加元数据，如网络延迟、客户端版本，用于问题排查 |

框架负责存储这些序列化后的操作日志，每条日志附加服务端接收时间戳以校准时序。`seqNum` 在回放重建时发挥关键作用：控制器按序列号顺序重放操作，忽略乱序到达的日志条目。

#### 3.6.2 回放重建方法

**FR-PLUGIN-06-02** 要求插件提供从日志重建控制器的方法，用于回放器功能。

**方法签名**：`replayFromLog(initialState: GameState, actions: List<Action>): GameController`

| 参数名 | 类型 | 说明 |
|---|---|---|
| `initialState` | `GameState` | 对局初始状态快照（开局时的完整状态） |
| `actions` | `List<Action>` | 按 `seqNum` 排序的操作序列 |

**返回值**：`GameController` 的只读回放实例。

**异常**：`IllegalArgumentException` — `actions` 未按 `seqNum` 升序排列，或 `initialState` 为 `null` 时抛出。

回放控制器处于严格只读模式：`submitAction()` 调用始终抛出 `UnsupportedOperationException`，`getLegalActions()` 返回空列表。回放仅允许顺序前进，框架通过调用 `stepForward(): void` 方法（回放控制器特有）逐条执行操作并广播对应事件，驱动 UI 重绘历史画面。回放过程中不触发超时逻辑，不生成新的 `Action` 序列号。

### 3.7 国际化支持

#### 3.7.1 文本资源外挂规范

**FR-PLUGIN-07-01** 要求插件中所有面向用户的文本（按钮标签、提示信息、规则说明、设置项名称）必须使用字符串 Key 并在独立语言文件中提供默认英文翻译。框架根据当前系统语言加载对应翻译表，插件通过 `context.localize(key)` 获取最终显示文本。

文本资源的组织遵循以下约定：每个插件在包体内包含 `i18n/` 目录，其中 `en.json` 为必须提供的默认英文翻译文件，其他语言文件（如 `zh.json`、`es.json`）可选提供。翻译文件采用扁平 Key 结构，例如：

```json
{
  "game_texas_name": "Texas Hold'em",
  "setting_small_blind": "Small Blind",
  "setting_player_count": "Number of Players",
  "action_fold": "Fold",
  "action_call": "Call",
  "action_raise": "Raise",
  "error_invalid_bet": "Bet amount must be between {min} and {max}"
}
```

`context.localize(key)` 支持插值参数传递，如 `context.localize("error_invalid_bet", { min: 10, max: 1000 })` 返回 `"Bet amount must be between 10 and 1000"`。若当前语言缺失某 Key 的翻译，框架自动回退至默认英文；若英文亦缺失，则返回 Key 本身并记录警告日志。插件更新时不应修改已有 Key 的语义，以确保已缓存的翻译文件兼容性；新增文本须使用新的唯一 Key。

## 4. 通用功能需求

通用功能需求定义框架层面向所有游戏插件提供的基础服务能力。这些能力通过框架运行时统一实现，游戏插件通过第3章定义的接口规范调用，无需在插件内部重复开发。本章涵盖大厅展示、用户与资产、房间管理、匹配调度、社交互动、成就任务、排行统计以及跨游戏通用能力共八个功能域。

### 4.1 游戏大厅

#### 4.1.1 游戏列表平铺展示

游戏大厅作为玩家启动应用后的首要入口，必须以直观的平铺布局展示所有已注册的游戏插件。系统须支持网格（Grid）与列表（List）两种布局模式的切换，网格模式下每个游戏项占据固定尺寸的卡片区域，展示游戏图标、名称及最近一局对局时间戳；列表模式下每项横向展开，额外显示该游戏当前在线人数与一句简要规则描述。**FR-LOBBY-01** 游戏卡片展示顺序必须遵循以下优先级规则：最近24小时内有对局记录的游戏排在首位，按对局结束时间降序排列；其余游戏按安装时间倒序排列；被玩家主动隐藏的游戏不显示于主界面。

#### 4.1.2 搜索与筛选

**FR-LOBBY-02** 系统须在大厅顶部提供搜索栏，支持按游戏名称关键词进行实时过滤，输入响应延迟不得超过300毫秒，匹配范围覆盖游戏名称与拼音首字母缩写。**FR-LOBBY-03** 搜索栏下方必须提供分类标签页，枚举值为：全部、牌类、棋类、麻将。标签切换时游戏列表须即时重新过滤，无需刷新整个页面。**FR-LOBBY-04** 系统须支持长按游戏图标弹出操作菜单，提供"隐藏此游戏"选项；被隐藏的游戏可在"设置-已隐藏的游戏"列表中恢复显示，恢复后立即重新出现在大厅对应位置。

#### 4.1.3 公告与活动

**FR-LOBBY-05** 大厅顶部必须配置可折叠的横幅轮播区域，用于展示运营公告与限时活动。轮播间隔固定为5秒，支持玩家手动滑动切换。**FR-LOBBY-06** 每条横幅须关联一个详情页面URL或路由标识，点击后跳转至对应的活动详情页或弹出模态框展示完整信息。**FR-LOBBY-07** 对于具有截止时间的活动，横幅右下角必须显示动态倒计时，格式为"HH:mm:ss"，每秒刷新一次，倒计时归零后横幅自动下架。

#### 4.1.4 在线状态

**FR-LOBBY-08** 每个游戏卡片必须实时显示当前在线人数，数据来源为服务端推送的房间级聚合统计，更新频率为每30秒一次。**FR-LOBBY-09** 大厅右上角必须提供服务器健康状态指示器，枚举值为：正常（绿色）、繁忙（黄色）、维护中（红色）。状态指示器的数据来源为心跳接口返回的服务端负载指标，当负载超过阈值时自动切换为繁忙状态。

### 4.2 用户与资产体系

#### 4.2.1 统一账号体系

**FR-USER-01** 系统必须提供统一的账号认证体系，支持三种登录方式：手机号+短信验证码、邮箱+密码、第三方OAuth（含Apple/Google/Facebook）。所有登录方式最终须绑定至同一个平台用户ID（platform_user_id），确保跨设备数据一致。**FR-USER-02** 系统须支持访客模式，访客可在不登录的情况下体验受限功能；访客模式必须提供明确的"转正"入口，转正时须将访客期间的全部本地数据（包括虚拟资产、战绩记录）迁移至正式账号。**FR-USER-03** 用户的核心身份数据（昵称、头像URL、等级、经验值）须在服务端集中存储，任何客户端修改必须通过服务端校验后生效，确保多端数据同步。

#### 4.2.2 多币种虚拟钱包

**FR-USER-04** 系统须实现多币种虚拟钱包体系，包含一个通用币种与多个游戏专属币种。通用币种为"金币"（gold），适用于所有游戏的低注房间入场门槛抵扣。每个游戏插件在注册时（参见**FR-PLUGIN-01-01**）可声明一个专属币种代号（如德州扑克的"chips"、麻将的"huanledou"），用于该游戏内部的对局结算。**FR-USER-05** 各币种余额的变更操作必须由服务端钱包服务统一执行，插件仅上报操作指令（如"玩家A赢得本局，应增加100筹码"），不可直接修改资产数值。钱包服务在处理变更前必须校验操作签名与防重放令牌（nonce）。

#### 4.2.3 用户资料管理

**FR-USER-06** 用户资料页面必须支持昵称、头像、个性签名的查看与修改。**FR-USER-07** 昵称修改必须受以下限制：单次修改消耗一张"改名卡"道具或等值金币；昵称长度限制为2至12个字符；禁止包含系统保留关键词与特殊符号；修改后30天内不可再次变更。**FR-USER-08** 头像修改支持从系统预设头像库选择或上传自定义图片，自定义图片须经过服务端审核，审核期间对其他玩家显示原头像。**FR-USER-09** 玩家等级由经验值（XP）累计决定，等级计算公式为 `level = floor(sqrt(XP / 100)) + 1`，升级所需经验值呈非线性递增。经验值通过对局结算、任务完成、每日签到三种途径获得。

#### 4.2.4 资产安全

**FR-USER-10** 所有虚拟资产变更请求必须经过服务端校验，校验项包括：请求来源的会话令牌有效性、操作金额是否超出当前余额、操作是否匹配当前游戏状态（如仅在结算阶段允许增减筹码）、防重放nonce是否已被使用。**FR-USER-11** 游戏插件客户端仅负责收集玩家操作意图并上报至服务端，插件代码不得包含任何直接修改资产余额的逻辑。框架须在加载插件时对插件二进制进行静态扫描，检测并拒绝包含敏感资产操作API调用的插件包。

### 4.3 房间系统

#### 4.3.1 房间创建

**FR-ROOM-01** 玩家在大厅选择游戏后点击"创建房间"，系统必须调起该游戏插件通过 `getSettingsSchema()` 接口提供的参数设置面板（参见**FR-PLUGIN-02-01**）。**FR-ROOM-02** 房主确认参数后，系统调用插件的 `validateSettings()` 方法进行参数合法性校验；校验通过后，服务端生成唯一房间码（格式为6位字母数字混合，如"A3B7K9"），创建房间记录并返回创建成功响应。**FR-ROOM-03** 房间记录须包含以下字段：房间ID、房间码、游戏类型ID、房主用户ID、当前状态、创建时间戳、最大玩家数、当前玩家列表（含座位号）、观战者列表、房间参数配置（JSON格式）。

#### 4.3.2 房间状态机

房间生命周期由框架层统一维护的状态机驱动，所有游戏插件共享此状态机模型。下表定义完整的状态转换规则：

| 当前状态 | 触发事件 | 下一状态 | 转换条件与说明 |
|----------|----------|----------|----------------|
| 等待中（Waiting） | 房主点击"开始游戏" | 准备中（Preparing） | 房间内玩家数达到游戏最小人数要求，且所有玩家标记为"已准备" |
| 等待中（Waiting） | 最后一名玩家离开 | 已解散（Dissolved） | 房间仅剩房主一人且房主选择解散，或所有普通玩家离开后房主离线超60秒 |
| 准备中（Preparing） | 插件调用 `start()` 完成初始化 | 游戏中（Playing） | 插件控制器成功完成洗牌/发牌等开局流程，广播 `onGameStarted` 事件 |
| 游戏中（Playing） | 插件广播 `onGameFinished` | 已结束（Finished） | 当前局游戏正常结束（含胜负判定），进入结算展示阶段 |
| 游戏中（Playing） | 房主点击"强制解散" | 已解散（Dissolved） | 房主拥有强制解散权限，解散后房间内所有玩家被踢回大厅 |
| 游戏中（Playing） | 所有人类玩家断线超超时阈值 | 已结束（Finished） | 超时阈值由房间参数设定（默认120秒），超时后AI接管并完成当前局 |
| 已结束（Finished） | 房主点击"再来一局" | 等待中（Waiting） | 保留房间配置与玩家列表，重置准备状态，未确认玩家自动踢出 |
| 已结束（Finished） | 所有玩家离开 | 已解散（Dissolved） | 结算完成后玩家逐一离开，最后一人离开后房间进入已解散状态 |

**FR-ROOM-04** 房间状态变更必须由服务端房间管理器作为唯一权威源发起，任何客户端状态显示均须以服务端推送的状态同步消息为准。**FR-ROOM-05** 状态机转换过程中，服务端必须向房间内所有玩家（含观战者）广播状态变更事件，事件载荷须包含新状态值、触发事件类型、触发者用户ID及时间戳。

#### 4.3.3 玩家管理

**FR-ROOM-06** 玩家加入房间须通过以下三种途径之一：输入正确房间码、从大厅房间列表点击进入、通过快速匹配自动分配。加入时服务端须校验房间当前人数是否达到上限及玩家资产是否满足门槛。**FR-ROOM-07** 玩家主动离开房间时，若当前处于"等待中"或"已结束"状态，离开立即生效；若处于"游戏中"状态，玩家离开后将由AI自动接管（参见**FR-AI-03**），玩家本人需等待当前局结束后方可真正离开。**FR-ROOM-08** 房主拥有踢出权限，可对处于"等待中"状态的房间内任意非房主玩家执行踢出操作；踢出后该玩家收到通知并被退回大厅，房间内其他玩家同步更新玩家列表。**FR-ROOM-09** 房主离线超过60秒后，房主身份自动转移至房间内最早加入且当前在线的玩家。

#### 4.3.4 观战模式

**FR-ROOM-10** 房间参数中"允许观战"开关为开启时，其他玩家可选择以观战者身份进入房间，观战者不占用玩家座位，不影响游戏进行。**FR-ROOM-11** 观战者视角必须至少延迟15秒相对实际对局，以防止利用观战进行信息传递作弊。**FR-ROOM-12** 观战者可在多个玩家视角之间自由切换，切换操作仅影响本地显示，不向其他观战者或玩家广播。**FR-ROOM-13** 观战者的聊天消息仅对其他观战者可见，玩家不可见观战频道内容；观战者不可发送语音消息。

#### 4.3.5 房间内互动

**FR-ROOM-14** 所有房间必须提供文字聊天功能，聊天频道分为"房间"（所有玩家与观战者可见）和"系统"（仅框架广播的系统消息）。**FR-ROOM-15** 系统须提供预设表情与快捷短语库，至少包含20个常用表情图标与30条预设短语（如"快点吧，我等得花都谢了"、"合作愉快"、"你的牌打得真好"），玩家点击即可发送。**FR-ROOM-16** 房间须支持语音快捷消息功能，玩家在设置中预先录制或选择系统提供的语音片段，在游戏中一键发送；语音消息在接收端自动播放，播放期间其他语音消息排队等待。

### 4.4 匹配系统

#### 4.4.1 快速匹配

**FR-MATCH-01** 快速匹配流程必须遵循以下步骤执行：步骤一，玩家在游戏子界面点击"快速开始"按钮，客户端向匹配服务提交匹配请求，请求载荷包含游戏类型ID、玩家用户ID、当前资产余额、玩家ELO等级分及偏好参数（如底注范围）；步骤二，匹配服务将请求放入该游戏类型对应的匹配队列，首先进行资产门槛筛选，剔除余额不满足任何可用房间最低底注要求的请求；步骤三，在通过资产筛选的请求中，匹配服务按ELO等级分进行区间匹配，初始匹配区间为 +/- 50分；步骤四，若匹配队列中存在满足条件的候选对手，匹配服务立即组建房间并将所有匹配玩家推送至该房间；步骤五，若未找到合适对手，匹配区间每10秒自动扩大25分，同时等待计时器累加；步骤六，当总等待时间超过配置的弹性上限（默认60秒）时，触发AI填充机制。

**FR-MATCH-02** 匹配服务必须支持按游戏模式进行分组隔离，不同模式（如德州扑克的现金桌与锦标赛）的玩家不可互相匹配。**FR-MATCH-03** 匹配服务须维护一个全局匹配状态映射表，记录每个玩家当前匹配状态（空闲/匹配中/已匹配），防止同一玩家在多个匹配队列中重复出现。

#### 4.4.2 AI填充机制

**FR-MATCH-04** 当匹配超时（默认60秒）且当前候选人数不足游戏最小人数要求时，系统必须自动使用AI玩家填充空缺座位。**FR-MATCH-05** AI填充的数量为 `max(0, minPlayers - currentPlayers)`，AI难度继承房间设置中房主指定的难度级别（参见**FR-AI-02**）。**FR-MATCH-06** AI玩家使用系统生成的临时用户ID（前缀为"ai_"），昵称格式为"AI玩家X"，头像使用系统预设AI头像，AI填充事件须广播给房间内所有人类玩家。

#### 4.4.3 匹配取消

**FR-MATCH-07** 玩家在匹配等待期间可随时主动点击"取消匹配"按钮取消匹配请求，取消后匹配服务立即从队列中移除该请求，客户端退回游戏子界面。**FR-MATCH-08** 匹配成功组建房间后，系统须向所有匹配到的玩家推送"匹配成功"通知，通知包含房间码与10秒倒计时；倒计时内玩家须点击"确认加入"按钮，超时未确认视为拒绝。**FR-MATCH-09** 若任一名匹配成功的玩家点击"拒绝"或超时未确认，该玩家退回匹配队列或大厅，其余已确认玩家须重新等待匹配补充；若已确认玩家数仍满足游戏最小人数，房间可立即开始。

### 4.5 社交系统

#### 4.5.1 好友系统

**FR-SOCIAL-01** 系统必须提供独立于具体游戏的好友管理体系。玩家可通过搜索昵称或用户ID查找其他玩家，发送好友请求；被请求方收到推送通知后可选择接受或拒绝。**FR-SOCIAL-02** 好友列表须显示每位好友的在线状态（在线/游戏中/离线）及当前所在游戏类型，状态更新通过服务端实时推送。**FR-SOCIAL-03** 系统须提供好友推荐功能，推荐来源包括：最近三局对局中的对手/队友、同地区活跃用户、共同好友关联用户。**FR-SOCIAL-04** 玩家可随时删除好友，删除后双向解除关系，双方不可再查看对方在线状态与详细战绩。

#### 4.5.2 聊天系统

**FR-SOCIAL-05** 系统必须提供三层聊天频道架构，各频道的消息存储策略与可见范围如下表所示：

| 频道类型 | 消息可见范围 | 消息存储策略 | 离线推送 | 消息保留时长 |
|----------|-------------|-------------|----------|-------------|
| 世界频道 | 所有在线玩家 | 服务端持久化，最近100条 | 不支持 | 72小时 |
| 房间频道 | 同房间玩家与观战者 | 仅内存缓存，不持久化 | 不支持 | 房间解散后清除 |
| 私聊频道 | 发送方与接收方 | 服务端持久化，全量存储 | 支持 | 永久（可手动删除） |

**FR-SOCIAL-06** 世界频道消息须在服务端按时间顺序广播至所有在线玩家，新上线的玩家自动接收最近100条历史消息。**FR-SOCIAL-07** 私聊频道须支持离线消息存储，当接收方不在线时消息存入离线信箱，接收方上线后按时间顺序推送；离线信箱容量上限为200条，超出后最早的未读消息被丢弃。**FR-SOCIAL-08** 所有聊天消息必须经过服务端敏感词过滤，命中敏感词的消息替换为"***"后投递。

#### 4.5.3 举报与黑名单

**FR-SOCIAL-09** 系统必须提供举报功能，玩家可举报其他玩家的违规行为（辱骂、作弊、消极游戏等），举报时须选择举报类别并可选填写补充说明。**FR-SOCIAL-10** 举报提交后生成客服工单，工单流转至后台审核系统，审核结果在72小时内通过系统消息反馈给举报方。**FR-SOCIAL-11** 玩家可将其他玩家加入黑名单，拉黑后双方不可互相发送私聊消息、不可加入同一房间、不可查看对方详细资料；黑名单列表在服务端存储，容量上限为100人。

### 4.6 成就与任务系统

#### 4.6.1 跨游戏成就

**FR-ACHV-01** 系统必须定义一套全局跨游戏成就体系，这些成就的触发条件不依赖于特定游戏类型，由框架层统一监听与判定。下表列出必须实现的基础跨游戏成就类型：

| 成就ID | 成就名称 | 触发条件 | 奖励内容 | 等级递进 |
|--------|----------|----------|----------|----------|
| ACHV_FIRST_WIN | 初出茅庐 | 在任意游戏中取得首次对局胜利 | 1000金币、头像框 | 无 |
| ACHV_SOCIAL_10 | 社交达人 | 累计添加10名好友 | 2000金币、称号 | I(5人)→II(10人)→III(50人) |
| ACHV_ALL_ROUNDER | 全能玩家 | 累计在5款不同的游戏中完成对局 | 5000金币、动态头像 | I(3款)→II(5款)→III(10款) |
| ACHV_WIN_STREAK | 连胜将军 | 在任意游戏中达成指定场次的连续胜利 | 按连胜长度递增金币 | I(3场)→II(5场)→III(10场) |
| ACHV_TOTAL_GAMES | 百战老兵 | 累计完成对局总场次达到指定数量 | 称号、金币 | I(100场)→II(500场)→III(2000场) |
| ACHV_WEALTHY | 腰缠万贯 | 累计持有金币数量达到指定峰值 | 动态头像框 | I(10万)→II(100万)→III(1000万) |

**FR-ACHV-02** 跨游戏成就的进度数据必须在服务端集中维护，进度更新由框架的事件总线驱动。框架监听游戏结束事件（`onGameFinished`）、好友关系变更事件等通用事件，当事件参数满足成就触发条件时自动推进对应成就进度。

#### 4.6.2 游戏专属成就

**FR-ACHV-03** 每个游戏插件在注册时可向框架提交专属成就列表，列表中的每项成就须包含成就ID、名称、多语言key、触发条件表达式、奖励配置及图标资源引用。**FR-ACHV-04** 框架负责为每个插件维护独立的成就进度命名空间，防止不同游戏的成就ID冲突；成就ID在框架内部自动附加游戏类型前缀（如 `texas_royal_flush_win` 内部存储为 `texas_holdem:texas_royal_flush_win`）。**FR-ACHV-05** 游戏专属成就的触发由插件主动上报。插件在对局中检测到满足成就条件的事件后，通过框架API `reportAchievementProgress(achievementId, increment)` 上报进度增量，框架校验成就ID有效性后累加进度并在达成目标时发放奖励。

#### 4.6.3 任务体系

**FR-ACHV-06** 系统必须提供三层任务结构：日任务（每日0点刷新，上限3个同时活跃）、周任务（每周一0点刷新，上限2个同时活跃）、长期任务（无时间限制，由成就系统统一管理）。**FR-ACHV-07** 任务对象须包含以下字段：任务ID、任务名称、任务描述、任务类型（日/周/长期）、完成条件表达式、当前进度、目标进度、奖励列表、过期时间戳、状态（未接取/进行中/已完成/已领取）。**FR-ACHV-08** 任务奖励发放遵循"完成即锁定，主动领取到账"的原则：任务条件满足后状态变更为"已完成"，奖励进入待领取列表；玩家须在任务界面主动点击"领取"按钮后，奖励才通过钱包服务发放至账户。**FR-ACHV-09** 日任务与周任务的生成必须从任务池中按权重随机抽取，任务池由运营后台配置，支持按玩家等级段进行差异化投放。

### 4.7 排行榜与统计

#### 4.7.1 多维度排行榜

**FR-LEAD-01** 系统必须提供以下维度的排行榜数据结构：时间维度（日榜、周榜、月榜、总榜）与范围维度（好友榜、地区榜、全服榜）。两种维度可组合，如"本周好友榜"。**FR-LEAD-02** 排行榜数据以排行榜记录（LeaderboardEntry）为基本单元，结构包含：排名、用户ID、昵称、头像URL、排名数值（如赢取金币数）、排名变化趋势（上升/下降/持平）。**FR-LEAD-03** 日榜在每日0点根据前一日数据结算并生成快照，周榜在每周一0点结算，月榜在每月1日0点结算。快照生成后，历史排行榜数据变为只读状态。**FR-LEAD-04** 玩家在任何排行榜界面中必须能够定位到自己的排名位置，系统自动滚动至玩家所在行并高亮显示。

#### 4.7.2 个人战绩

**FR-LEAD-05** 系统必须为每位玩家维护个人战绩统计，数据来源为每场对局结束后的结算数据持久化。**FR-LEAD-06** 战绩统计指标须至少包含：总对局场次、胜场数、胜率（`winRate = wins / totalGames * 100%`）、当前连胜场次、历史最高连胜、各游戏类型的细分战绩。对于牌类游戏，额外记录最佳手牌（按牌型强度等级排序后的历史最高）。**FR-LEAD-07** 胜率计算保留一位小数，当总场次为零时显示"-"。连胜统计中，平局或无效局（如对手全部逃跑）不中断连胜也不计入连胜场次。

#### 4.7.3 防刷分机制

**FR-LEAD-08** 系统必须在排行榜数据入库前进行异常检测，检测项包括：单场收益超出该游戏类型历史最大值3倍以上、同一局内多个账号的胜负模式高度关联（疑似串通）、短时间内（1小时内）同一组玩家的对局频次超过阈值。**FR-LEAD-09** 命中异常检测规则的数据记录须标记为"待审核"状态，不纳入排行榜排名计算；审核通过后恢复正常排名，审核拒绝则从排行榜中剔除相关数据并对涉事账号执行清零处罚。**FR-LEAD-10** 全服排行榜前100名玩家的排名数据须经过二次校验，校验逻辑包括多维度数据交叉比对与行为模式分析。

### 4.8 跨游戏通用能力

#### 4.8.1 通用计时器

**FR-COMMON-01** 框架必须提供统一的计时器服务（TimerService），供所有游戏插件调用以管理玩家思考时间与局时限制。计时器服务接口须包含以下方法：`startTimer(playerId, timeoutSeconds, callback)` 启动倒计时、`extendTime(playerId, extraSeconds)` 延长指定玩家的剩余时间、`getRemainingTime(playerId)` 查询剩余秒数、`cancelTimer(playerId)` 取消计时器。**FR-COMMON-02** 玩家操作超时后，框架必须自动调用插件控制器注册的 `onTimeout(playerId)` 回调方法，由插件决定超时后的具体处理逻辑（如自动弃牌、自动出最小牌、切换AI托管等）。**FR-COMMON-03** 计时器服务必须支持客户端与服务端双端倒计时同步：服务端作为权威源每秒推送剩余时间，客户端据此校正本地倒计时显示，网络抖动导致的偏差不得超过1秒。

#### 4.8.2 标准操作流程

框架定义悔棋、求和、认输三种标准操作流程的标准协议，所有棋类游戏插件可选择性接入，牌类游戏插件至少须实现认输协议。

**FR-COMMON-04** 悔棋协议（Undo Protocol）适用于棋类游戏。流程如下：请求方点击"悔棋"按钮，向服务端发送 `UNDO_REQUEST` 消息；服务端将请求转发给对手方；对手方收到弹窗提示，可选择"同意"或"拒绝"；若对手方在15秒内同意，服务端执行状态回退至最近一个合法历史状态（回退步数由插件定义，通常为1步），并广播 `UNDO_EXECUTED` 事件通知双方；若对手方拒绝或超时未响应，悔棋请求自动失效。每局每方悔棋次数不得超过房间参数设定的上限（默认3次）。

**FR-COMMON-05** 求和协议（Draw Protocol）适用于棋类游戏。流程如下：请求方发送 `DRAW_REQUEST` 消息；对手方收到弹窗提示，可选择"同意"或"拒绝"；若对手方在15秒内同意，本局立即以和局结束，双方不计胜负，结算时各自收回本局下注；若对手方拒绝或超时，对局继续进行。每局每方求和请求仅限一次。

**FR-COMMON-06** 认输协议（Surrender Protocol）适用于所有游戏类型。流程如下：玩家点击"认输"按钮后弹出二次确认弹窗（防止误触）；玩家确认后，发送 `SURRENDER` 消息至服务端；服务端立即判定该玩家为负，触发本局结算流程；结算时认输方承担全部下注损失（牌类游戏）或判定为负（棋类游戏）。认输操作不可撤销。

**FR-COMMON-07** 以上三种标准操作的消息结构须统一格式：`{ actionType: string, requesterId: string, timestamp: number, targetPlayerId?: string }`。插件在收到服务端广播的操作事件后，负责更新本地游戏状态与UI表现。

#### 4.8.3 回放系统

**FR-COMMON-08** 框架必须提供通用的对局回放功能。每场对局结束后，服务端以标准化格式存储完整的操作序列，每条记录包含：操作序号、操作类型、执行者用户ID、操作数据（JSON格式）、服务端时间戳、当前游戏状态哈希（用于完整性校验）。**FR-COMMON-09** 回放UI由框架提供统一组件，至少包含以下控件：播放按钮、暂停按钮、步进按钮（前进/后退一步）、进度条拖拽、倍速切换（0.5x/1x/2x/4x）、退出按钮。**FR-COMMON-10** 回放时框架负责操作序列的顺序调度，插件负责根据当前操作索引渲染对应的游戏画面。回放控制器通过调用插件的 `replayFromLog()` 接口（参见**FR-PLUGIN-06-02**）初始化回放专用控制器实例，该实例仅接受步进指令，不接受玩家操作输入。

#### 4.8.4 多语言本地化

**FR-COMMON-11** 框架必须支持运行时的语言切换能力，玩家可在设置页面选择界面语言，切换后立即生效无需重启应用。首期支持语言至少包括：简体中文、繁体中文、英语、日语、韩语、西班牙语。**FR-COMMON-12** 框架层所有通用UI元素（大厅菜单、房间界面、聊天系统、设置页面等）的文本须通过框架本地化服务 `context.localize(key)` 获取。**FR-COMMON-13** 每个游戏插件须独立打包语言资源文件，文件命名遵循 `{pluginId}_{locale}.json` 的约定。框架根据当前语言设置与插件ID自动加载对应语言包；若目标语言包不存在，回退至插件声明的默认语言（通常为英语）。**FR-COMMON-14** 插件语言包加载采用懒加载策略，仅在玩家首次进入该游戏时才从服务端下载对应语言资源，避免大厅启动时加载全部插件的全部语言文件。

## 5. 五子棋插件设计需求

五子棋（Gomoku）插件是通用棋牌游戏框架的参考实现，遵循框架插件接口规范，通过 registerPlugin() 注册并提供完整的 GameController 实现。本节定义该插件在游戏规则、数据模型、状态管理、AI 引擎和界面布局方面的全部技术需求。

### 5.1 游戏规则定义

#### 5.1.1 棋盘规格与坐标系

棋盘采用直角坐标系，原点位于左上角。横轴为列，以大写字母 A 至 O（15 路）或 A 至 S（19 路）标注；纵轴为行，以阿拉伯数字 1 至 15 或 1 至 19 标注，从左至右递增。坐标书写格式为列号在前、行号在后，例如 H8 表示第 H 列、第 8 行的交叉点。棋盘规格必须在游戏创建时通过 getSettingsSchema() 提供的配置面板选定，且一局游戏内不可变更。天元点（15 路棋盘为 H8，19 路棋盘为 J10）必须以特殊标记（如小圆点）在界面上标示，作为开局参考点。**FR-GOM-RULE-01**：棋盘必须支持 15×15 和 19×19 两种规格，坐标系列号使用 A-O/S、行号使用 1-15/1-19 的标注方式，天元点在开局前可见。**FR-GOM-RULE-02**：坐标系必须在所有向玩家展示的界面元素（包括落子提示、历史记录、AI 分析标记）中保持一致，内部数组索引用 (row, col) 表示，row 对应行号减 1，col 对应列号的 ASCII 偏移量（如 col=0 对应 A）。

#### 5.1.2 禁手规则

禁手规则仅适用于黑方，作为平衡先行优势的补偿机制。规则为可配置项，通过 getSettingsSchema() 返回的面板提供独立开关。当禁手规则启用时，黑方若形成禁手棋型，系统必须立即判定该落子为非法操作，阻止其执行并提示具体禁手类型。白方无禁手限制。

| 禁手类型 | 定义 | 检测条件 | 影响 |
|----------|------|----------|------|
| 三三禁手 | 黑方一子落下后，同时形成两个或以上的"活三"棋型 | 在四个方向上分别识别活三（两端均开放的连续三子），统计独立活三数量 ≥ 2 | 该落子位置被禁止，黑方必须另选他处 |
| 四四禁手 | 黑方一子落下后，同时形成两个或以上的"四"棋型 | 统计冲四（一端开放）与活四（两端开放）的总数 ≥ 2 | 同上，落子被禁止 |
| 长连禁手 | 黑方一子落下后，形成连续六子或以上的连线 | 在八个方向上扫描连续同色子数 > 5 | 该落子直接导致黑方判负 |

禁手检测算法在落子合法性校验阶段执行，顺序为：先检验落子位置是否为空，再检验是否构成禁手（若规则启用），最后更新棋盘状态并触发胜负判定。长连禁手因直接导致判负，优先级最高，检测逻辑必须在三三和四四之前完成判定。**FR-GOM-RULE-03**：禁手规则必须支持独立开关配置，启用时仅对黑方生效，白方不受约束。**FR-GOM-RULE-04**：长连禁手必须触发即时判负，三三和四四禁手必须阻止非法落子并返回错误提示。

#### 5.1.3 胜负判定

胜负判定通过八方向扫描算法实现。以落子点为中心，在水平（左右）、垂直（上下）、主对角线（左上-右下）、副对角线（右上-左下）四个轴共八个方向上，分别向两侧延伸计数连续同色棋子数量。若任一轴向上连续同色棋子总数（含落子点）达到 5，且该轴两端不存在"长连"（即总数恰好为 5，而非 6 或以上），则判定该颜色方获胜。**FR-GOM-RULE-05**：五连珠胜负判定必须在每次合法落子后立即执行，扫描范围为以落子点为中心的八方向射线，检测目标为恰好连续 5 子。**FR-GOM-RULE-06**：超时判负机制依赖框架层提供的计时器，当某方累计思考时间超过配置值时，系统自动调用 endGame() 并设置胜方为对方。**FR-GOM-RULE-07**：认输操作由玩家主动发起，经框架确认后直接进入 ENDED 状态，胜方为对方。

#### 5.1.4 开局配置

**FR-GOM-RULE-08**：游戏默认采用自由开局，黑方第一手必须落于天元点，随后双方自由落子。开局配置通过 getSettingsSchema() 暴露"交换执子权"选项，启用时允许白方在黑方第一手后选择是否交换黑白身份，以增加策略深度。

### 5.2 数据模型

#### 5.2.1 棋盘状态

棋盘状态使用二维整数数组 `board[row][col]` 表示，三个取值为：0 表示空位，1 表示黑子，2 表示白子。数组维度在初始化时由选定的棋盘规格（15 或 19）决定，默认全部元素为 0。**FR-GOM-DATA-01**：棋盘状态数组必须在内存中以连续二维结构存储，支持 O(1) 时间复杂度的随机访问。落子合法性校验函数接收目标坐标 (x, y)，返回布尔值：当且仅当坐标在棋盘范围内且 `board[y][x] == 0` 时返回 true，否则返回 false 并附带错误码（OUT_OF_BOUNDS 或 POSITION_OCCUPIED）。

#### 5.2.2 落子历史栈

落子历史采用 LIFO 栈结构实现悔棋功能。栈中每个元素为结构体 `MoveRecord`，包含字段：`position`（坐标对，列号字符串如 "H" 和行号整数）、`color`（枚举值 BLACK/WHITE）、`timestamp`（Unix 时间戳，毫秒精度）、`moveNumber`（正整数，从 1 开始递增）。**FR-GOM-DATA-02**：历史栈必须支持 push（落子时自动压栈）和 pop（悔棋时出栈）操作，push 和 pop 的时间复杂度均为 O(1)。悔棋操作弹出栈顶记录后，同步将对应棋盘位置重置为 0，并从游戏状态机中回退当前执子方。

#### 5.2.3 执子分配

**FR-GOM-DATA-03**：游戏严格遵循黑先行规则，开局时由座位编号较小的玩家执黑，另一方执白。执子方切换逻辑由游戏状态机驱动，每次合法落子后自动轮换。内部以枚举值 `ActivePlayer.BLACK` 和 `ActivePlayer.WHITE` 标记当前回合方，通过 XOR 运算或状态切换函数实现轮换。

### 5.3 游戏状态机

#### 5.3.1 状态定义与转换

五子棋游戏的状态机包含四个核心状态：WAITING（等待开始）、BLACK_TURN（黑方回合）、WHITE_TURN（白方回合）、ENDED（游戏结束）。状态转换遵循严格条件：WAITING → BLACK_TURN 在 start() 被调用且双方玩家就绪后触发；BLACK_TURN → WHITE_TURN 在黑方完成合法落子且无五连后触发；WHITE_TURN → BLACK_TURN 在白方完成合法落子且无五连后触发；任一 TURN 状态 → ENDED 在五连检测命中、长连禁手触发、超时判负、认输或和棋达成时触发。**FR-GOM-STATE-01**：状态转换必须通过 EventBus 广播状态变更事件，事件负载携带旧状态、新状态和触发原因（如 MOVE_PLACED、TIMEOUT、RESIGN、DRAW_AGREED）。

#### 5.3.2 和棋判定

**FR-GOM-STATE-02**：和棋有两种触发路径：其一为棋盘满子检测，当历史栈长度等于总交叉点数（225 或 361）且最后一步未形成五连时，自动触发和棋并进入 ENDED 状态；其二为双方玩家同意和棋，任一方发起求和请求后，对方在限定时间内（默认 30 秒）确认即达成和棋，若拒绝或超时则游戏继续。

### 5.4 玩家操作接口

#### 5.4.1 落子操作

落子操作通过 submitAction() 接口提交，请求参数结构为：`{ playerId: string, actionType: "PLACE_STONE", payload: { x: number, y: number } }`，其中 x 为列索引（0-based），y 为行索引（0-based）。接口执行顺序为：(1) 校验 playerId 与当前 activePlayer 一致；(2) 校验坐标在棋盘范围内；(3) 校验目标位置为空；(4) 若禁手规则启用，校验黑方不落禁手；(5) 更新棋盘数组并压入历史栈；(6) 执行八方向五连检测；(7) 切换执子方或进入 ENDED。**FR-GOM-ACTION-01**：落子接口的响应体必须包含字段 `success`（布尔值）、`boardState`（二维数组快照）、`lastMove`（坐标对）、`gameStatus`（当前状态枚举）、`winner`（若游戏结束则为 BLACK/WHITE/DRAW，否则 null），以及 `errorCode`（失败时如 NOT_YOUR_TURN、POSITION_OCCUPIED、FORBIDDEN_MOVE）。

#### 5.4.2 悔棋操作

悔棋采用请求-确认模式。**FR-GOM-ACTION-02**：悔棋流程为：发起方调用 submitAction() 提交 actionType 为 "REQUEST_UNDO" 的请求 → 系统向对弈方发送确认事件 → 对弈方在限定时间内确认 → 系统从历史栈弹出两条记录（撤销双方最近各一步） → 棋盘回退至对应状态 → 当前回合方回退为被撤销最后一步的执子方 → 通过 EventBus 广播 UNDO_EXECUTED 事件。若对弈方拒绝或超时未响应，悔棋请求自动失效。

#### 5.4.3 认输与求和

**FR-GOM-ACTION-03**：认输操作由 submitAction() 接收 actionType "RESIGN"，无需对方确认，执行后即时进入 ENDED 状态，胜方为对方。求和操作通过 actionType "REQUEST_DRAW" 发起，对方玩家收到 DRAW_REQUESTED 事件后可选择接受（actionType "ACCEPT_DRAW"）或拒绝（actionType "DECLINE_DRAW"），接受则进入 ENDED 状态且结果为和棋，拒绝则游戏继续当前回合。

### 5.5 AI设计

#### 5.5.1 算法框架

AI 引擎采用 Minimax 算法结合 Alpha-Beta 剪枝作为决策核心。评估函数由两部分构成：棋型识别评分与位置评分。棋型识别模块扫描当前棋盘所有空位，模拟落子后识别形成的各类棋型并按权重表累加评分；位置评分模块基于棋盘几何中心（天元区域）给予靠近中心的落子点额外加分，鼓励 AI 争夺中央控制权。Minimax 树在每个决策节点上展开所有合法落子，叶节点调用评估函数返回评分值，内部节点交替取最大/最小值向上回溯。Alpha-Beta 剪枝在标准 Minimax 基础上维护 α（当前最大值）和 β（当前最小值）两个边界，当某分支的评分超出边界时立即剪除，降低搜索空间。**FR-GOM-AI-01**：AI 引擎必须实现 Minimax + Alpha-Beta 剪枝框架，评估函数由棋型评分权重表驱动，搜索深度由难度等级参数动态控制。

#### 5.5.2 棋型评分权重

棋型评分体系基于连子数量与开放端数量两个维度建立。AI 评估函数在棋盘上对每个空位模拟落子后，沿四个轴向扫描并识别以下棋型，按对应权重计入总分：

| 棋型名称 | 描述 | 权重值 |
|----------|------|--------|
| 活四 | 连续四子且两端均为空位，下一手必胜 | 100000 |
| 冲四 | 连续四子仅一端开放，需对方阻挡 | 10000 |
| 活三 | 连续三子两端均开放，可发展为活四 | 1000 |
| 眠三 | 连续三子仅一端开放，发展潜力受限 | 100 |
| 活二 | 连续二子两端均开放，有发展为活三的潜力 | 10 |
| 位置分 | 基于距天元点曼哈顿距离的加成 | 1-5 |

评估函数的最终得分为己方棋型总分减去对方棋型总分（使用相同权重表模拟对方最佳落子），再加上位置分修正。**FR-GOM-AI-02**：棋型识别必须覆盖四个轴向的所有空位扫描，权重值必须可配置以支持后续调优。当 AI 识别到己方可形成活四时，必须优先选择该位置作为落子点。

#### 5.5.3 难度分级

AI 难度通过搜索深度、开局库和评估扰动三个参数的组合实现：

| 难度等级 | 搜索深度 | 开局策略 | 评估扰动 | 响应时间上限 |
|----------|----------|----------|----------|--------------|
| 简单 | 2 | 随机选择中心区域合法落子 | ±20% 随机评分扰动 | 3 秒 |
| 中等 | 4 | 无特殊开局 | 无扰动 | 5 秒 |
| 困难 | 6 | 内置 50 步标准开局库 | 无扰动 | 10 秒 |

**FR-GOM-AI-03**：AI 难度参数必须通过 getSettingsSchema() 暴露为下拉选择项，三个等级分别映射到上述深度和策略配置。困难模式在开局前 50 步优先查询开局库返回预设应手，超出库覆盖范围后 fallback 至深度 6 的搜索。所有难度等级的 AI 落子均通过 submitAction() 接口提交，与真人玩家的操作路径完全一致。

### 5.6 界面布局

#### 5.6.1 棋盘区域

棋盘区域占据主界面约 60% 的可用面积，居中放置。**FR-GOM-UI-01**：棋盘必须采用木纹纹理背景或纯色背景配合深色网格线，线宽 1-2 像素，交叉点间距自适应容器尺寸确保均匀分布。15 路棋盘显示 15×15 交叉点，19 路棋盘显示 19×19 交叉点。坐标轴标注位于棋盘外侧，列号（A-O/S）置于顶部，行号（1-15/1-19）置于左侧，字体大小与棋盘尺寸成比例。天元点及星位（如 15 路棋盘的 D4、D12、L4、L12）以 5-8 像素直径的小圆点标记。

#### 5.6.2 落子指示

**FR-GOM-UI-02**：鼠标悬停在棋盘空位上时，必须显示半透明的落子预览（黑子或白子轮廓，依据当前回合方决定）。落子成功后，触发框架音效系统的落子事件（如木质敲击声）。最后一步落子位置必须以红色方框（边长约为交叉点间距的 80%，线宽 2 像素）高亮标注，持续到下一步落子时移至上新位置。

#### 5.6.3 信息面板

信息面板位于主界面右侧或下方，占剩余约 40% 面积。**FR-GOM-UI-03**：面板必须包含以下信息区：当前回合方标识（以黑白圆形图标加文字"黑方回合"/"白方回合"显示）、双方累计用时（格式 MM:SS，每秒刷新）、双方已下步数统计（整数计数）、悔棋/认输/求和操作按钮。当某方玩家思考时间接近上限时（剩余 10%），该方用时数字必须以红色闪烁提示。

## 6. 黑白棋插件设计需求

黑白棋（Reversi/Othello）插件是基于本章所定义的规则引擎与AI策略体系构建的可插拔游戏模块。该插件遵循第3章定义的 `PluginMeta` 接口规范注册至框架，其中 gameId="reversi"，category="棋类"，minPlayers=2，maxPlayers=2，支持人机对战与人人对战两种模式。本章从规则定义、数据模型、状态机、玩家操作接口、AI设计与界面布局六个维度，完整定义黑白棋插件的功能需求与技术规格。

### 6.1 游戏规则定义

#### 6.1.1 棋盘规格

黑白棋采用8行8列的标准方格棋盘，行列编号沿用国际象棋坐标体系：横向（列）以字母a至h标识，纵向（行）以数字1至8标识。**FR-REV-RULE-01** 棋盘数据模型必须定义为 `8×8` 二维数组，数组索引 `[row][col]` 中 row 取值范围为 0 到 7（对应数字坐标 1 到 8），col 取值范围为 0 到 7（对应字母坐标 a 到 h）。棋盘初始状态下，中心四个格子（d4、e4、d5、e5，对应数组索引 `[3][3]`、`[3][4]`、`[4][3]`、`[4][4]`）必须按对角关系预置两枚黑子与两枚白子：d4 与 e5 为黑子（值为1），e4 与 d5 为白子（值为2）。

#### 6.1.2 落子合法性判定

**FR-REV-RULE-02** 落子合法性判定必须满足以下两个条件：第一，目标位置必须在棋盘范围内且当前为空（值为0）；第二，落子后在至少一个方向上能够夹住对方连续棋子序列，且该序列的另一端紧邻己方棋子。"夹住"定义为：从落子位置出发，沿某一方向前进，首先遇到连续一个或多个对方棋子，紧接着遇到一个己方棋子。若上述条件之一不满足，该落子位置即为非法，拒绝执行并返回错误码 `INVALID_MOVE`。

#### 6.1.3 翻转算法

**FR-REV-RULE-03** 翻转算法必须以落子位置为原点，沿八个方向进行射线扫描。八个方向的坐标增量（dx, dy）定义如下表所示，其中 dx 表示列偏移（正值向右），dy 表示行偏移（正值向下）：

| 方向编号 | 方向名称 | dx（列增量） | dy（行增量） |
|----------|----------|-------------|-------------|
| 0 | 东（右） | +1 | 0 |
| 1 | 西（左） | -1 | 0 |
| 2 | 南（下） | 0 | +1 |
| 3 | 北（上） | 0 | -1 |
| 4 | 东南（右下） | +1 | +1 |
| 5 | 西南（左下） | -1 | +1 |
| 6 | 东北（右上） | +1 | -1 |
| 7 | 西北（左上） | -1 | -1 |

扫描时，每个方向从落子点沿增量方向逐步推进。若第一个相邻格子为对方棋子，则继续沿该方向前进；若遇到己方棋子，则将此前经过的全部对方棋子标记为待翻转；若遇到空位或超出边界，则该方向上无翻转。**FR-REV-RULE-04** 所有待翻转棋子的翻转操作必须在一次原子性更新中完成，翻转序列须记录为有序数组供动画回放与撤销使用。

#### 6.1.4 终局判定

**FR-REV-RULE-05** 终局判定必须满足以下任一条件：条件一，棋盘64格全部被填满（无空位）；条件二，当前回合方无合法落子，且对方在尝试回合后也无可行落子（双方连续Pass）。终局时，系统统计双方棋子数量，数量多者获胜。**FR-REV-RULE-06** 若双方棋子数量相同（各32枚），则判定为平局；平局时双方不计胜负场，各自获得基础经验值奖励。

### 6.2 数据模型

#### 6.2.1 棋盘状态

**FR-REV-DATA-01** 棋盘状态数据结构定义为一个包含64个整数字段的 `BoardState` 对象，使用 `CellValue` 枚举表示每个格子的当前值：`EMPTY = 0`（空位）、`BLACK = 1`（黑子）、`WHITE = 2`（白子）。棋盘状态对象必须额外维护以下元数据字段：`currentPlayer`（当前回合方，枚举值1或2）、`moveCount`（已执行步数，从0开始）、`lastMove`（上一步落子坐标，无则为null）、`scoreBlack`（黑方当前棋子数）、`scoreWhite`（白方当前棋子数）。

#### 6.2.2 合法落子位掩码

**FR-REV-DATA-02** 每回合开始前，系统必须预计算当前回合方的所有合法落子位置，生成 `LegalMoveMask` 位掩码对象。`LegalMoveMask` 是一个64位整数（或8×8布尔数组），每个比特位（或数组元素）对应棋盘上一个格子，`true` 表示该位置为合法落子点。**FR-REV-DATA-03** 合法落子位掩码的计算必须与棋盘状态同步更新，在每次翻转完成后立即重新生成；掩码数据通过 `onStateUpdated` 事件广播至客户端，客户端据此渲染可落子位置高亮提示。

#### 6.2.3 翻转链追踪

**FR-REV-DATA-04** 每次落子操作必须生成一个 `FlipChain` 翻转链记录对象，用于精确追踪该步所触发的全部翻转序列。`FlipChain` 对象包含以下字段：`movePosition`（触发翻转的落子坐标，如 "f5"）、`movePlayer`（落子方，1或2）、`flipSequences`（翻转序列数组，每个元素为一个方向的翻转坐标列表）、`timestamp`（操作时间戳）。**FR-REV-DATA-05** 翻转链记录必须压入操作历史栈 `moveHistory`，支持悔棋操作通过弹出栈顶记录并逆向执行翻转恢复先前状态，同时支持逐枚动画回放。

### 6.3 游戏状态机

#### 6.3.1 状态定义与转换图

**FR-REV-STATE-01** 黑白棋插件必须维护独立的内部游戏状态机，状态定义与转换规则如下表所示：

| 当前状态 | 触发事件 | 下一状态 | 转换条件与说明 |
|----------|----------|----------|----------------|
| WAITING | 双方玩家准备就绪 | BLACK_TURN | 游戏初始化完成，轮到黑方落子 |
| BLACK_TURN | 黑方执行合法落子 | WHITE_TURN | 落子合法且翻转完成，切换至白方回合 |
| BLACK_TURN | 黑方无合法落子 | PASSED | 黑方无可行落子，自动Pass，通知双方 |
| WHITE_TURN | 白方执行合法落子 | BLACK_TURN | 落子合法且翻转完成，切换至黑方回合 |
| WHITE_TURN | 白方无合法落子 | PASSED | 白方无可行落子，自动Pass，通知双方 |
| PASSED | 对方有合法落子 | 对方TURN | Pass后对方可以落子，切换至对方回合 |
| PASSED | 对方也无合法落子 | ENDED | 双方连续Pass，触发终局判定 |
| BLACK_TURN / WHITE_TURN | 棋盘满64子 | ENDED | 棋盘无空位，立即触发终局判定 |
| ENDED | 玩家点击再来一局 | WAITING | 重置棋盘至初始四子状态，重新开局 |

状态机由服务端权威维护，每次状态变更须通过 `onStateUpdated` 事件向客户端广播。**FR-REV-STATE-02** 客户端必须根据服务端广播的状态值渲染对应的UI提示（如当前回合指示、Pass通知、终局结算面板），不可基于本地推测擅自变更状态显示。

#### 6.3.2 强制Pass

**FR-REV-STATE-03** 当当前回合方无合法落子时，系统必须自动触发强制Pass，不允许玩家手动选择落子位置。**FR-REV-STATE-04** 强制Pass触发时，系统须向房间内所有玩家（含观战者）广播 `onPlayerPassed` 事件，事件载荷包含被Pass方颜色（black/white）与原因说明（`no_legal_moves`）；同时状态机切换至PASSED状态，等待切换至对方回合。

### 6.4 玩家操作接口

#### 6.4.1 落子操作

**FR-REV-ACTION-01** 落子操作必须遵循以下完整流程：步骤一，客户端通过 `submitAction` 接口提交落子请求，请求载荷包含坐标（如 `{row: 5, col: 4}` 或 `"f5"`）与玩家ID；步骤二，服务端校验坐标有效性，确认目标位置在棋盘范围内且为空；步骤三，服务端调用翻转算法检测该位置是否至少在一个方向上能夹住对方棋子；步骤四，若合法性检测通过，执行翻转操作，更新棋盘状态，重新计算合法落子位掩码，增加步数计数器；步骤五，生成翻转链记录并压入历史栈；步骤六，检查终局条件，若满足则切换至ENDED状态并广播 `onGameFinished` 事件，否则切换至对方回合并广播 `onStateUpdated` 事件。

**FR-REV-ACTION-02** 若落子合法性检测失败，系统必须返回错误响应 `ILLEGAL_MOVE`，错误信息须包含失败原因（如 `no_flip_in_any_direction`），客户端收到错误后禁止更新本地棋盘显示。

#### 6.4.2 Pass操作

**FR-REV-ACTION-03** Pass操作仅在当前回合方无任何合法落子时由系统自动触发，玩家不可主动选择Pass。**FR-REV-ACTION-04** 系统在检测到无合法落子时，须在客户端显示Pass提示弹窗（持续2秒），提示文案为"无可行落子，自动跳过回合"，弹窗消失后自动切换至对方回合。

#### 6.4.3 悔棋操作

**FR-REV-ACTION-05** 悔棋操作允许撤销最近一次的落子及其触发的全部翻转。执行流程为：步骤一，从 `moveHistory` 栈顶弹出最近的 `FlipChain` 记录；步骤二，将落子位置恢复为空（0）；步骤三，遍历 `flipSequences` 中的全部翻转坐标，将每个被翻转的棋子恢复为翻转前的颜色（即重新翻回对方颜色）；步骤四，恢复上一步的回合方、棋子计数与合法落子位掩码；步骤五，广播 `onStateUpdated` 事件通知所有客户端同步恢复后的状态。**FR-REV-ACTION-06** 悔棋操作仅限人机对战模式使用，人人对战模式下必须禁用悔棋功能以防止争议。

### 6.5 AI设计

#### 6.5.1 贪心+位置权重策略

**FR-REV-AI-01** AI评估函数必须采用基于位置权分的贪心策略，为棋盘每个格子分配固定的位置权重值，AI在选择落子时计算每个合法落子位置执行后己方获得的评估总分（包括落子位置权重与翻转棋子数的加权和），选择总分最高的位置。**FR-REV-AI-02** 位置权重评分表以8×8矩阵形式定义，矩阵中的每个值与棋盘格子一一对应：

|  | a | b | c | d | e | f | g | h |
|--|---|---|---|---|---|---|---|---|
| **1** | +100 | -20 | +20 | +5 | +5 | +20 | -20 | +100 |
| **2** | -20 | -40 | -5 | -5 | -5 | -5 | -40 | -20 |
| **3** | +20 | -5 | +15 | +3 | +3 | +15 | -5 | +20 |
| **4** | +5 | -5 | +3 | +1 | +1 | +3 | -5 | +5 |
| **5** | +5 | -5 | +3 | +1 | +1 | +3 | -5 | +5 |
| **6** | +20 | -5 | +15 | +3 | +3 | +15 | -5 | +20 |
| **7** | -20 | -40 | -5 | -5 | -5 | -5 | -40 | -20 |
| **8** | +100 | -20 | +20 | +5 | +5 | +20 | -20 | +100 |

该权重矩阵基于黑白棋位置战略价值的经典评估模型设计。角位（a1、h1、a8、h8）权重为+100，因其一旦占据便不可被翻转，是最具战略价值的位置。C位（b1、g1、b8、g8）紧邻角位但无法自保，权重为-20，属于高风险区域。X位（b2、g2、b7、g7）是对角邻近角位的陷阱位置，权重低至-40，落子后极易被对方抢占角位。边位非角（c1、f1、c8、f8、a3、h3等）权重为+20，相对安全且稳定。中心区域权重较低（+1至+15），主要用于鼓励AI在开局阶段向中心拓展。

#### 6.5.2 终局穷举切换

**FR-REV-AI-03** 当棋盘上剩余空位数量小于等于12时，AI必须切换为完全搜索模式（Minimax算法配合Alpha-Beta剪枝），搜索深度固定为剩余空位数，确保终局阶段决策的最优性。**FR-REV-AI-04** 完全搜索的评估函数在终局阶段简化为双方棋子数量差（己方棋子数减去对方棋子数），不再使用位置权重，因为终局时所有位置翻转风险已不存在，唯一目标是最大化己方最终棋子数。

#### 6.5.3 动态难度

**FR-REV-AI-05** AI难度分为三级，各等级的策略配置与评估维度如下表所示：

| 难度等级 | 评估策略 | 前瞻深度 | 额外评估维度 | 终局穷举阈值 |
|----------|----------|----------|-------------|-------------|
| 简单 | 纯贪心 | 1步（仅当前回合） | 无 | 剩余≤6子时启用 |
| 中等 | 贪心+位置权重 | 3步前瞻（Minimax深度=3） | 无 | 剩余≤10子时启用 |
| 困难 | 贪心+位置权重+Mobility | 5步前瞻（Minimax深度=5） | Mobility差异评估 | 剩余≤12子时启用 |

**FR-REV-AI-06** 困难模式的Mobility评估必须计算双方合法落子数的差异值 `mobilityScore = myLegalMoves - opponentLegalMoves`，并将该差异值以权重系数0.5叠加至总分中。Mobility评估的核心思想是：限制对方的可选行动（低Mobility）同时保持自身的行动灵活性（高Mobility），从而在战略层面获得主动权。**FR-REV-AI-07** 难度选择通过 `createAIController(difficulty, gameController)` 接口的 `difficulty` 参数传入，枚举值为 `EASY`、`MEDIUM`、`HARD`，AI工厂方法根据参数值返回配置对应策略的AI控制器实例。

### 6.6 界面布局

#### 6.6.1 棋盘渲染

**FR-REV-UI-01** 棋盘渲染区域必须采用深绿色毡面质感背景（色值 `#1a7a3e` 至 `#228B45` 的径向渐变），棋格线以浅色细线（`#8FBC8F`，线宽1px）绘制，形成8×8的网格。**FR-REV-UI-02** 可落子位置必须以半透明白色圆形（opacity=0.4，直径为格子的60%）叠加显示于对应格子中央，帮助玩家快速识别合法落子点；此提示功能可通过配置项 `showLegalHints`（布尔值）开启或关闭，默认开启。**FR-REV-UI-03** 棋子渲染为圆盘形状，黑子使用径向渐变从 `#333333` 到 `#000000`，白子使用从 `#FFFFFF` 到 `#DDDDDD`，棋子直径为格子的80%，边缘保留1px的 `#888888` 阴影以增强立体感。

#### 6.6.2 计分面板

**FR-REV-UI-04** 棋盘正上方必须配置计分面板，左侧显示黑方棋子数与圆形黑子图标，右侧显示白方棋子数与圆形白子图标，中间显示当前回合指示器（高亮当前回合方的图标边框）。**FR-REV-UI-05** 计分面板必须包含水平进度条，总长度为64（满盘棋子数），黑方与白方的已占份额以相反方向从中心向两侧延伸显示，进度条上方标注双方数量差及领先提示文字（如"黑方领先6子"）。**FR-REV-UI-06** 当一方棋子数达到33枚且超过对方时，进度条须显示金色闪烁动画提示该方已确立不可逆优势。

#### 6.6.3 翻转动画

**FR-REV-UI-07** 落子触发的翻转动画必须以逐枚方式顺序播放，每枚棋子的翻转动画持续时间为0.3秒，使用CSS 3D旋转效果模拟棋子从正面翻至背面的物理过程。**FR-REV-UI-08** 翻转序列的播放顺序必须遵循 `FlipChain.flipSequences` 中的方向分组，同一方向上的翻转从内向外依次执行（先翻转紧邻落子点的棋子，再逐步延伸至远端），不同方向之间按方向编号0至7的顺序并行或微延迟交错启动，以增强视觉层次感。**FR-REV-UI-09** 翻转动画播放期间，必须同步触发音效（翻转音效时长0.15秒），且客户端在动画完全结束前必须屏蔽新的落子操作，防止在状态过渡期间提交重复请求。

## 7. 围棋插件设计需求

### 7.1 游戏规则定义

#### 7.1.1 棋盘规格

围棋插件必须支持三种标准棋盘规格。**FR-GO-RULE-01** 棋盘尺寸可配置为9×9（初学者用）、13×13（中级练习用）与19×19（正式对局用），尺寸在游戏初始化时由配置参数确定且对局过程中不可变更。**FR-GO-RULE-02** 坐标系采用字母列标（A–T，跳过I）与数字行标（1–N）的组合形式，例如19路棋盘的左上角为A1，右下角为T19。坐标原点A1固定位于棋盘左上角，行号向下递增，列标向右递增。

#### 7.1.2 气与提子

**FR-GO-RULE-03** 棋子之"气"定义为与该棋子四邻连通（上下左右，不含对角线）的所有空点集合。属于同一玩家的连通棋子构成一个**连通块**（Group），连通块的气为该块内所有棋子四邻空点的并集。**FR-GO-RULE-04** 连通块的气数计算算法采用并查集（Disjoint Set Union, DSU）结合广度优先搜索（BFS）实现：落子时若邻接同色棋子，则通过并查集的`union`操作合并相关连通块；合并后从新连通块根节点出发执行BFS，遍历块内所有棋子的四邻格，统计不重复空点数量得到气数。并查集需维护父指针数组`parent[board_size]`与秩数组`rank[board_size]`，按秩合并确保均摊时间复杂度接近O(α(N))。**FR-GO-RULE-05** 无气状态判定触发提子：当某连通块的气数降为零时，该块内所有棋子从棋盘移除，由对方获得等量提子数。

**FR-GO-RULE-06** 自杀禁着：若落子后该棋子所在连通块的气数为零且未造成对方棋子无气，则该落子为非法自杀着法，必须拒绝执行并返回错误。但允许"反提"情形——即落子同时使对方连通块无气而被提，则自身连通块在提子后获得气的情形合法。

#### 7.1.3 打劫规则

**FR-GO-RULE-07** 劫争状态必须记录上一步被提子的坐标与提子颜色。**FR-GO-RULE-08** 采用简单劫规则时，禁止玩家在对方提子后立即于同一坐标回提（热劫禁止）。**FR-GO-RULE-09** 配置项启用Superko（全局同形重复）时，系统必须拒绝导致全局盘面状态重复的任何着法，盘面状态以Zobrist哈希值标识，维护历史局面哈希集合进行O(1)时间复杂度查重。

下表列出打劫规则的类型定义与处理逻辑。

| 规则类型 | 适用条件 | 违规检测方式 | 违规处理 |
|:---|:---|:---|:---|
| 简单劫（Ko） | 对方上一步提子1子 | 检查落子坐标==上一步被提坐标 | 拒绝落子，返回打劫违规错误 |
| 打二还一（Snapback） | 落子提2+子后立即被回提 | 检查全局盘面Zobrist哈希 | 允许（不构成劫） |
| Superko（Situational） | 配置项启用后全程有效 | 查询历史局面哈希集合 | 拒绝落子，返回同形重复错误 |
| Superko（Positional） | 忽略行棋方仅比较盘面 | 查询历史盘面哈希集合 | 拒绝落子，返回同形重复错误 |

打劫规则体系的选择在游戏初始化时通过配置参数`ko_rule`确定，可选值为`simple`（默认）与`superko`。采用Superko时需在内存中维护局面哈希集合，每步落子后将当前盘面Zobrist哈希加入集合；若配置为Positional Superko，则不计入执子方信息，仅比较物理盘面状态。

#### 7.1.4 终局与数子

**FR-GO-RULE-10** 终局由连续两次虚手（Pass）触发，进入数子阶段。**FR-GO-RULE-11** 数子阶段支持两种规则体系：中国数子法（数子法，贴目应用于子数）与日本数目法（数目法，贴目应用于目数）。**FR-GO-RULE-12** 终局时双方玩家可对死子标记存在争议，争议解决机制包括继续对弈与系统仲裁两种路径。若争议超时未解决，系统将未标记为死的连通块全部视为活子。

#### 7.1.5 贴目与让子

**FR-GO-RULE-13** 贴目值（Komi）支持按规则体系配置，标准对局默认贴目值如下表所示。让子对局中，黑方先行的贴目值需按让子数减半折算（让2子贴0目，让3子贴-2.5目，以此类推）。**FR-GO-RULE-14** 让子数配置范围为0–9子，让子坐标按棋盘规格自动摆放在标准星位：9路无让子星位，13路让子摆于(4,4)/(4,10)/(10,4)/(10,10)/(7,7)及其组合，19路让子摆于(4,4)/(4,16)/(16,4)/(16,16)/(10,10)及其组合。

| 规则体系 | 标准贴目值 | 让子范围 | 让子贴目折算规则 |
|:---|:---|:---|:---|
| 中国规则（数子法） | 3.75子（即7.5目） | 0–9子 | 贴目 = 3.75 – 让子数/2 |
| 日本规则（数目法） | 6.5目 | 0–9子 | 贴目 = 6.5 – 让子数 |
| 韩国规则（数目法） | 6.5目 | 0–9子 | 贴目 = 6.5 – 让子数 |
| 自定义规则 | 0–20目可调 | 0–9子 | 用户手动输入 |

贴目配置通过`komi`与`handicap`两个独立配置项传入。当`handicap`大于0时，`komi`值按上表公式自动覆盖。让子坐标根据`handicap`值从标准星位列表中按索引顺序选取，星位列表在代码中以常量数组定义，支持9/13/19三种规格分别查表。

### 7.2 数据模型

#### 7.2.1 棋盘状态

**FR-GO-DATA-01** 棋盘状态采用三维数据结构维护：二维数组`board[size][size]`存储每交叉点状态（EMPTY=0, BLACK=1, WHITE=2）；并查集`dsu`维护连通块关系，每个非空点对应一个并查集节点；每个连通块根节点关联一个`GroupInfo`对象，记录块内棋子数、气的位图（Bitmap）或坐标集合、以及 liberties_count 缓存值。**FR-GO-DATA-02** 落子后并查集合并操作流程：将新落子点初始化为独立连通块，遍历其四邻同色棋子，对每对连通块执行`union(root_a, root_b)`，合并后更新根节点的`GroupInfo`为两块的并集；随后从新连通块根出发BFS四邻空点，重建气集合。提子时遍历被提连通块内所有棋子，逐一将其从`board`清除，并更新相邻异色连通块的气集合（将清除后的空点加入邻块气集合）。

#### 7.2.2 劫争记录

**FR-GO-DATA-03** 劫争记录数据结构`KoRecord`包含字段：`last_captured_pos`（上一步被提子坐标，Point类型或null）、`last_captured_color`（被提子颜色枚举）、`ko_point`（当前禁着点坐标或null）、`position_hashes`（历史局面Zobrist哈希集合，仅Superko启用时维护）。**FR-GO-DATA-04** 打劫违规检测逻辑：落子前检查目标坐标是否等于`ko_point`，若相等则拒绝；Superko模式下额外检查当前落子后盘面哈希是否存在于`position_hashes`集合中。

#### 7.2.3 死子与领地

**FR-GO-DATA-05** 终局死子标记数据结构`DeadStoneMarker`包含：标记映射表`dead_map[size][size]`（布尔值，true表示该位置棋子被标记为死子）、双方已确认标志`black_agreed`与`white_agreed`、提交计数`submission_count`。**FR-GO-DATA-06** 领地计数采用洪水填充算法（Flood Fill）：从未被棋子占据的空点出发，沿四邻空格扩散，若扩散区域仅接触单一颜色棋子则该区域归属该方领地；若同时接触双方棋子则为公气（中性点），不计入任何一方。

#### 7.2.4 SGF支持

**FR-GO-DATA-07** 棋谱序列化与反序列化接口必须兼容Smart Game Format（SGF）FF[4]标准。序列化时输出符合SGF语法的文本，包含属性标识`GM[1]`（围棋游戏编号）、`SZ[尺寸]`、`KM[贴目]`、`HA[让子数]`及落子序列`B[坐标]/W[坐标]`。坐标编码规则：列标A–T（跳I）转小写字母，行标1–N直接映射为a–s。**FR-GO-DATA-08** 序列化示例：一手19路对局的SGF片段格式为`(;GM[1]FF[4]SZ[19]KM[7.5]HA[0];B[qd];W[dd];B[pq];W[dp];B[pass];W[dc])`，其中`pass`用空属性值表示为`B[]`或`W[]`。反序列化接口需解析该格式并还原为内部棋盘状态与落子序列。

### 7.3 游戏状态机

#### 7.3.1 状态定义

**FR-GO-STATE-01** 围棋插件状态机定义如下状态及转换关系：`WAITING`（等待对局开始）→ `BLACK_MOVE`（黑方行棋）↔ `WHITE_MOVE`（白方行棋）；任意行棋方可通过Pass转入`BLACK_PASS`或`WHITE_PASS`；连续两次Pass后进入`SCORING`（数子阶段）；数子争议解决或双方确认后进入`ENDED`（对局结束）。状态转换必须由状态机引擎统一管理，每次转换触发对应事件通知。

#### 7.3.2 双虚手终局

**FR-GO-STATE-02** 系统维护连续Pass计数器`consecutive_pass`，初始值为0。当任一方执行Pass操作时`consecutive_pass`加1；当任一方执行非Pass落子时`consecutive_pass`重置为0。**FR-GO-STATE-03** 当`consecutive_pass`达到2时，状态机自动从`WHITE_PASS`（或`BLACK_PASS`）转换至`SCORING`状态，触发`onEnterScoring`事件通知双方客户端进入终局数子阶段。

#### 7.3.3 终局争议

**FR-GO-STATE-04** 进入`SCORING`状态后，双方玩家可标记死子。若双方标记结果一致（`black_agreed && white_agreed`为true），则直接计算胜负并转入`ENDED`。**FR-GO-STATE-05** 若标记存在分歧，任一方可发起"继续对弈"请求，状态机从`SCORING`退回至`BLACK_MOVE`（由上一Pass方的对方先行），恢复对弈。**FR-GO-STATE-06** 若争议在配置的超时时间内（默认5分钟）未解决且无人请求继续对弈，系统启动仲裁：将所有非明确标记为死的连通块视为活子，强制完成计分并转入`ENDED`。

### 7.4 玩家操作接口

#### 7.4.1 落子操作

**FR-GO-ACTION-01** 落子操作接口`placeStone(playerId, point)`执行以下有序校验链：(1) 坐标边界校验——检查`point.x`与`point.y`在[0, size)范围内，越界返回`INVALID_COORDINATE`；(2) 已有棋子检测——检查`board[point]`非空则返回`OCCUPIED`；(3) 自杀检测——模拟落子后计算新连通块气数，若为零且未造成对方提子则返回`SUICIDE`；(4) 打劫检测——检查目标坐标是否为当前禁着点，若是则返回`KO_VIOLATION`；(5) Superko检测——若启用Superko，模拟落子后计算盘面Zobrist哈希，若存在于历史集合则返回`SUPERKO_VIOLATION`；(6) 全部通过后执行落子——更新棋盘、合并连通块、执行提子、更新劫争记录、切换行棋方。步骤(3)与步骤(4)的顺序不可调换，因打劫检测依赖自杀检测后的连通块状态。

#### 7.4.2 虚手操作

**FR-GO-ACTION-02** 虚手（Pass）接口`passMove(playerId)`执行流程：校验当前行棋方为该玩家→增加`consecutive_pass`计数→切换行棋方→发送Pass通知至对方→若`consecutive_pass==2`则触发终局进入`SCORING`状态。Pass操作无需棋盘状态变更，但必须记录于棋谱序列与SGF输出中。

#### 7.4.3 死子标记

**FR-GO-ACTION-03** 死子标记接口`markDead(point, playerId)`仅在`SCORING`状态可用。玩家点击某连通块内任意一子，系统将该整个连通块的`dead_map`标记取反（切换死/活状态）。**FR-GO-ACTION-04** 双方确认机制：每方拥有独立的确认标志，当任一方点击"确认"按钮后设置对应标志；仅当`black_agreed && white_agreed`同时满足时，系统执行最终计分。**FR-GO-ACTION-05** 任一方在对方未确认前可修改死子标记，修改时自动重置双方确认标志为false。

### 7.5 AI设计

#### 7.5.1 MCTS框架

**FR-GO-AI-01** 围棋AI核心引擎采用蒙特卡洛树搜索（Monte Carlo Tree Search, MCTS）算法，包含四个标准阶段：选择（Selection，基于UCT公式遍历树）、扩展（Expansion，添加新节点）、模拟（Simulation，随机 rollout 至终局）、反向传播（Backpropagation，更新路径上节点统计值）。**FR-GO-AI-02** UCT公式定义为：`UCT(v_i) = Q(v_i)/N(v_i) + C * sqrt(2*ln(N(parent))/N(v_i))`，其中Q为累计胜率估计，N为访问次数，C为探索常数。**FR-GO-AI-03** MCTS参数配置如下表所示，参数通过`MCTSConfig`对象传入引擎。

| 参数名称 | 默认值 | 可调范围 | 说明 |
|:---|:---|:---|:---|
| simulation_count | 800 | 100–50000 | 单次决策的模拟总次数 |
| uct_constant_C | 1.414 | 0.5–5.0 | UCT公式探索项系数，sqrt(2)为理论最优 |
| parallel_threads | 4 | 1–16 | 并行模拟线程数，CPU核心数-1为推荐值 |
| virtual_loss | 3 | 1–10 | 并行冲突避免虚拟损失值 |
| rollout_limit | 450 | 100–1000 | 单次模拟最大步数限制，超过则以当前局面近似评估 |
| resign_threshold | 0.05 | 0.01–0.20 | 认输阈值，胜率低于此值时AI主动认输 |
| dirichlet_alpha | 0.3 | 0.1–1.0 | 根节点先验噪声Dirichlet分布α参数 |
| dirichlet_epsilon | 0.25 | 0.0–0.5 | 先验噪声混合比例 |

MCTS引擎采用虚拟损失（Virtual Loss）机制支持并行模拟：多线程同时从根节点选择不同子节点展开模拟，在节点访问计数上暂加虚拟损失以避免多线程选择同一节点，模拟完成后反向传播时扣除虚拟损失并更新真实统计值。该机制在多核CPU环境下可提升3–5倍吞吐。

#### 7.5.2 神经网络集成

**FR-GO-AI-04** 高级难度模式下，MCTS的模拟阶段由神经网络替代纯随机Rollout。系统集成分为两部分：策略网络（Policy Network）输入当前盘面特征，输出362维（19×19+Pass）走子概率向量，为MCTS树搜索提供先验概率P(s,a)；价值网络（Value Network）输入当前盘面特征，输出标量估值v∈[-1,1]，表示当前局面下先手胜率估计。**FR-GO-AI-05** 神经网络输入特征平面包含：当前玩家棋子位置、对方棋子位置、空点位置、己方合法着法掩码、上一步落子位置、以及距离边界等辅助特征，共17个二进制特征平面。网络推理通过ONNX Runtime框架执行，支持CPU/GPU后端切换。

#### 7.5.3 难度分级

**FR-GO-AI-06** AI引擎支持四级难度配置，难度切换通过`difficulty`配置参数实现：

- **入门**（`difficulty=0`）：完全随机走子加基础贪心启发——优先选取可提子的着法，无提子时随机选取合法点。
- **初级**（`difficulty=1`）：MCTS 100次模拟，UCT常数C=1.414，单线程执行，无神经网络指导，Rollout采用随机落子至200步上限。
- **中级**（`difficulty=2`）：MCTS 1000次模拟，4线程并行，启用虚拟损失机制=3，Rollout步数上限450步。
- **高级**（`difficulty=3`）：MCTS 1600次模拟，8线程并行，集成策略网络提供先验概率，价值网络替代终点评估，Dirichlet噪声α=0.3/ε=0.25注入根节点以保障着法多样性。

## 8. 象棋插件设计需求

中国象棋（Xiangqi）插件基于框架的插件接口规范实现，通过 `PluginMeta` 注册：`gameId="xiangqi"`，`category="棋类"`，`minPlayers=2`，`maxPlayers=2`。插件配置项包括思考时间（单位秒，默认 30）、AI 难度（简单/中等/困难/大师）、是否启用开局库（布尔值）、和棋规则配置（自然限着步数/长将阈值）。以下分六个维度展开设计需求。

### 8.1 游戏规则定义

#### 8.1.1 棋盘布局

象棋棋盘由 9 条纵线（路）与 10 条横线（线）交叉组成，共 90 个交叉点。棋盘中央以"楚河汉界"文字分隔为上下两半。红方（下方）纵线自右向左编号 1–9，黑方（上方）纵线自右向左编号 1–9，横线自红方底线向上编号 1–10。九宫区域为各方底线中心 3×3 的格子范围，将（帅）与士（仕）的活动被限制于此区域内。

**FR-XQ-RULE-01** 系统必须支持以坐标 `(file, rank)` 标识每个交叉点，其中 `file ∈ [1,9]`，`rank ∈ [1,10]`，并能在渲染层正确绘制楚河汉界与九宫斜线。

#### 8.1.2 棋子走法矩阵

中国象棋共 32 枚棋子，分红黑两方，每方 16 枚。七种棋子的走法规则汇总于表 8-1。

**表 8-1 棋子走法规则矩阵**

| 棋子 | 颜色标识 | 每方数量 | 走法描述 | 限制条件 | 特殊规则 |
|:---|:---|:---|:---|:---|:---|
| 将（帅） | 红K/黑k | 1 | 九宫内纵横移动 1 格 | 不得离开九宫 | 双方将帅不得照面（同一纵线无遮挡） |
| 士（仕） | 红A/黑a | 2 | 九宫内斜线移动 1 格 | 不得离开九宫 | 无 |
| 象（相） | 红B/黑b | 2 | 走"田"字（斜向 2 格） | 不能过河 | 象心（田字中心点）被占时不能行走 |
| 马 | 红N/黑n | 2 | 走"日"字（1 格直+1 格斜） | 棋盘边界内 | 马前方向相邻点有子时蹩脚 |
| 车 | 红R/黑r | 2 | 直线任意格（横或纵） | 路径不得有子阻挡 | 无 |
| 炮 | 红C/黑c | 2 | 直线移动同车；吃子时需隔 1 子 | 移动时路径无障碍 | 炮打隔山：吃子必须恰好跳过一个棋子 |
| 兵（卒） | 红P/黑p | 5 | 过河前向前走 1 格；过河后可横走或向前走 1 格 | 不得后退 | 过河指越过楚河汉界进入对方半区 |

表 8-1 所列走法为每个棋子生成合法走法的核心依据。将帅的"照面"规则需在生成走法后做额外校验：若移动己方将帅后双方处于同一纵线且中间无其他棋子，则该走法非法。马的日字走法产生 8 个候选方向，需依次通过蹩脚检测与边界检测进行过滤。炮的移动逻辑分为两种模式：不吃子时与车完全相同；吃子时目标位置必须有对方棋子，且起点与目标点之间恰好存在一枚棋子（炮架）。

**FR-XQ-RULE-02** 系统必须依据表 8-1 的走法描述、限制条件和特殊规则，为当前局面下的指定棋子生成完整的合法走法集合。

#### 8.1.3 特殊规则判定算法

蹩马脚与塞象眼是中国象棋中两类典型的几何阻塞规则，其判定算法直接影响合法走法的正确性。

**蹩马脚判定**。马走日字，设马当前位置为 `(fx, fy)`，候选目标位置为 `(tx, ty)`。计算偏移量 `dx = tx - fx`，`dy = ty - fy`。马腿位置（阻塞点）为 `(fx + sign(dx), fy + sign(dy))`，即日字拐角处的相邻点。若该位置存在任何棋子（无论敌我），则此方向的走法被蹩住。伪代码如下：

```
function isHorseBlocked(board, fx, fy, tx, ty):
    dx = tx - fx; dy = ty - fy
    if |dx| == 2 and |dy| == 1:
        legX = fx + sign(dx); legY = fy
    else if |dx| == 1 and |dy| == 2:
        legX = fx; legY = fy + sign(dy)
    else:
        return true  // 非日字走法，视为阻塞
    return board[legX][legY] != EMPTY
```

**塞象眼判定**。象走田字，设象当前位置为 `(fx, fy)`，目标位置为 `(tx, ty)`。象心（田字中心）坐标为 `((fx + tx) / 2, (fy + ty) / 2)`。若该位置存在任何棋子，则此象步被塞住。伪代码如下：

```
function isElephantBlocked(board, fx, fy, tx, ty):
    eyeX = (fx + tx) / 2; eyeY = (fy + ty) / 2
    return board[eyeX][eyeY] != EMPTY
```

**FR-XQ-RULE-03** 系统在生成马和象的合法走法时，必须调用上述蹩马脚与塞象眼判定算法，将被阻塞的方向从候选走法中剔除。

**炮打隔山判定**。炮吃子时，设起点 `(fx, fy)` 与目标点 `(tx, ty)` 在同一横线或纵线上。令 `count` 为两点之间（不含端点）的棋子数量。`count == 1` 且目标位置为对方棋子时，该走法合法；否则不合法。

**FR-XQ-RULE-04** 系统在验证炮吃子走法时，必须确保起点与目标点之间恰好存在一枚棋子作为炮架，且目标位置为对方棋子。

#### 8.1.4 将军与应将

**将军状态检测** 的实现方式为：在走法生成完毕后，遍历对方所有棋子的合法走法，检查是否存在任意走法的目标位置为己方将帅所在坐标。若存在，则当前己方处于被将军状态。

**应将合法性** 要求玩家必须选择能够解除将军状态的走法。系统将某个候选走法标记为应将合法的条件是：模拟执行该走法后，调用将军状态检测函数，确认己方不再被将军。

**FR-XQ-RULE-05** 当一方处于被将军状态时，系统必须过滤走法集合，仅向玩家展示能将解除将军状态的应将合法走法；若应将合法走法集合为空，则判定该方被将死。

#### 8.1.5 胜负判定

胜负判定的触发条件包括：（1）将死——一方被将军且无任何应将合法走法；（2）困毙——一方未被将军但无任何合法走法；（3）超时判负——某方累计用时超过配置时限；（4）认输——玩家主动认输；（5）和棋——包括自然限着（双方连续 50 步无吃子）、三次重复局面、长将（连续将军超过阈值）、双方同意和棋。

**FR-XQ-RULE-06** 系统必须在每次走法执行后调用胜负判定流程，按上述优先级顺序检测将死、困毙、超时、和棋等终止条件。

### 8.2 数据模型

#### 8.2.1 棋盘编码

棋盘状态采用类 FEN（Forsyth-Edwards Notation）字符串编码，格式定义为：

```
< piece_placement > < active_color > < halfmove_clock > < fullmove_number >
```

其中 `piece_placement` 按 rank 10 到 rank 1 的顺序排列，同 rank 内按 file 1 到 file 9 的顺序，空位以连续数字表示；`active_color` 为 `r`（红方）或 `b`（黑方）；`halfmove_clock` 为自然限着计数器；`fullmove_number` 为完整回合数。每枚棋子记录其类型（K/A/B/N/R/C/P）、颜色（红/黑）、位置坐标及存活状态。

**FR-XQ-DATA-01** 系统必须支持类 FEN 字符串与内部棋盘状态的双向转换，转换函数须处理所有 32 枚棋子的类型、颜色、位置和存活状态。

#### 8.2.2 走法历史

每步记录为一个结构体 `MoveRecord`，字段定义如下：`piece`（移动棋子类型与颜色）、`from`（起始坐标）、`to`（目标坐标）、`captured`（被吃子类型，无吃子则为 null）、`isCheck`（是否将军）、`timestamp`（落子时间戳）。历史记录以栈结构存储，支持一步或多步撤销。

**FR-XQ-DATA-02** 系统必须为每步走法生成 `MoveRecord` 记录并压入历史栈，悔棋时按后进先出顺序回退棋盘状态并恢复被吃子。

#### 8.2.3 重复局面检测

采用 Zobrist Hash 技术实现高效的重复局面检测。初始化阶段为每种棋子类型在每个交叉点位置生成 64 位随机哈希值，另为当前行棋方生成一个独立的随机值。局面哈希值为所有在位棋子的对应随机值按位异或的结果。每次走法执行后，将新局面的哈希值存入历史哈希表（`Map<hash, count>`）。

**FR-XQ-DATA-03** 系统必须在每次走法执行后计算并记录当前局面的 Zobrist Hash，当同一哈希值在历史表中出现次数达到 3 时，自动触发三次重复和棋判定。

#### 8.2.4 长将检测

长将检测通过连续将军计数器实现。计数器规则如下：若某步走法为将军，则对应方的连续将军计数器加 1；若某步走法不是将军，则该计数器重置为 0。当一方的连续将军计数器达到阈值 12 时，判定该方长将负。

**FR-XQ-DATA-04** 系统必须为红方和黑方分别维护连续将军计数器，阈值可配置（默认 12），当任一方计数器达到阈值时触发长将判负。

### 8.3 游戏状态机

#### 8.3.1 状态定义

象棋游戏的状态转换遵循有限状态机模型，状态集合定义为：`WAITING`（等待玩家加入）、`RED_TURN`（红方行棋）、`BLACK_TURN`（黑方行棋）、`CHECK`（将军状态，嵌套于红/黑回合中）、`ENDED`（对局结束）。状态转换规则为：`WAITING` → 双方就位后 → `RED_TURN` → 红方落子后 → `BLACK_TURN` → 黑方落子后 → `RED_TURN`，循环往复；任一方落子触发将军时进入 `CHECK` 子状态；触发终止条件时进入 `ENDED`。

**FR-XQ-STATE-01** 系统必须实现上述状态机，确保状态转换的原子性，同一时间仅有一个激活状态。

#### 8.3.2 和棋判定

和棋判定在每次走法执行后进行检查，判定条件按优先级排列如下：双方连续 50 步无吃子（`halfmove_clock >= 50`）；同一局面 Zobrist Hash 出现 3 次；一方连续将军达到阈值（默认 12 次）；双方玩家同时点击同意和棋按钮。任一条件满足即判定和棋。

**FR-XQ-STATE-02** 系统必须在每次走法执行后按上述优先级顺序检查和棋条件，和棋判定触发后立即进入 `ENDED` 状态并记录和棋原因。

### 8.4 玩家操作接口

#### 8.4.1 选子落子

玩家操作采用两步交互模型：第一步点击己方棋子，系统以金色边框高亮选中棋子，并在所有合法落子点位置显示绿色圆点提示；第二步点击目标位置（带绿色圆点的合法落子点），系统执行走法。若第一步选中无合法走法的棋子，系统以灰色边框提示不可移动。

**FR-XQ-ACTION-01** 系统必须实现两步选子落子交互模型，并在玩家选中棋子后实时计算并显示所有合法落子点的绿色提示圆点。

#### 8.4.2 走法校验

走法校验采用三级流水线：（1）棋子规则校验——依据表 8-1 的走法矩阵验证是否符合目标棋子的移动规则；（2）棋盘边界校验——确认目标坐标在 [1,9]×[1,10] 范围内；（3）不送将校验——模拟走法后检测己方是否被将军，若被将军则该走法非法。

**FR-XQ-ACTION-02** 系统必须在走法执行前完成三级校验，任何一级校验失败则拒绝该走法并向玩家展示原因提示。

#### 8.4.3 悔棋操作

悔棋功能支持一步撤销和多步撤销两种模式。执行悔棋时，从历史栈中弹出指定数量的 `MoveRecord`，按逆序逐条恢复：将被移动棋子从目标坐标移回起始坐标，若存在被吃子则将其恢复至目标坐标并置为存活状态，同时恢复将军计数器和自然限着计数器。

**FR-XQ-ACTION-03** 系统必须支持任意步数的悔棋操作，悔棋后棋盘状态、历史记录和各项计数器必须与回退到的历史节点完全一致。

#### 8.4.4 求和操作

求和流程采用请求-响应模型：一方点击求和按钮后，系统向对方弹出求和确认对话框；对方可选择同意（触发和棋结束）或拒绝（关闭对话框，对局继续）；若对方在 15 秒内未响应，系统自动视为拒绝。

**FR-XQ-ACTION-04** 系统必须实现带超时机制的求和流程，超时时间可配置（默认 15 秒），超时未响应自动关闭对话框并视为拒绝。

### 8.5 AI 设计

#### 8.5.1 局面评估函数

AI 的走法选择依赖于局面评估函数，评估值由子力价值与位置加分两部分构成。子力价值为基础分，反映各棋子的绝对战斗力；位置分（Piece-Square Table）为附加分，反映棋子在不同位置的战略价值。

**表 8-2 局面评估权重表**

| 棋子 | 子力价值（基础分） | 位置分范围 | 评估说明 |
|:---|:---|:---|:---|
| 车 | 500 | 0–20 | 通路车（纵线无敌方棋子阻挡）额外加 20 分 |
| 马 | 300 | -10–15 | 中心位置加分高，边线位置扣分；蹩马脚风险位置扣分 |
| 炮 | 300 | 0–15 | 与对方棋子对齐时加分（打击威胁）；空头炮额外加 15 分 |
| 士 | 100 | 固定 | 仅限于九宫斜线位置，无额外位置浮动 |
| 象 | 100 | 固定 | 仅限于己方半区田字格位置，过河位置无效 |
| 兵（未过河） | 50 | 0–5 | 每向前走一步加 2 分 |
| 兵（已过河） | 80 | 0–20 | 过河后价值提升，逼近九宫位置大幅加分 |

表 8-2 的评估权重为 AI 难度分级提供统一的评分基准。局面评估函数的计算公式为：`Score = Σ(己方棋子子力价值 + 位置分) - Σ(对方棋子子力价值 + 位置分)`。红方计算时取正值，黑方计算时取负值。

**FR-XQ-AI-01** AI 引擎的局面评估函数必须依据表 8-2 的权重计算双方局面分差，评估结果作为极小化极大（Minimax）搜索的叶子节点分值。

#### 8.5.2 开局库

开局库用于在搜索深度外提供人类经验积累的高质量开局走法。存储结构采用局面哈希索引表，每条记录格式为：

```
OpeningEntry {
    zobristHash: uint64,      // 局面 Zobrist 哈希值
    move: Move,               // 推荐走法（起止坐标+棋子）
    weight: uint8,            // 权重（1-100，出现频率越高权重越大）
    name: string,             // 开局名称（如"中炮对屏风马"）
    nextPositions: uint64[]   // 后续常见局面哈希引用
}
```

开局库存储为 JSON 文件，加载时解析为 `Map<uint64, OpeningEntry[]>`。查询接口为 `queryOpening(zobristHash: uint64): Move?`，返回加权随机选择的一条推荐走法；若无匹配则返回 null，AI 回退至搜索模式。

**FR-XQ-AI-02** 系统必须支持开局库 JSON 文件的加载与查询，查询接口按权重概率随机返回推荐走法，无匹配时自动回退至 Minimax 搜索。

#### 8.5.3 残局库

残局库覆盖关键残局局面的预计算结果，包括单车胜单士、马兵胜单士、炮兵胜单象等。残局记录格式为：`(attackersMask, defendersMask, result, minMoves)`，其中 `result` 为 WIN/DRAW/LOSE，`minMoves` 为最短胜利用法数。查询接口为 `queryEndgame(boardHash: uint64): EndgameResult?`。

**FR-XQ-AI-03** 系统残局库必须在棋子总数 ≤ 5 时激活查询，命中时直接返回预计算结果指导 AI 走法选择。

#### 8.5.4 难度分级

AI 提供四级难度，配置参数如下表：

| 难度等级 | 搜索深度 | 开局库 | 残局库 | 随机因子 | 评估精度 |
|:---|:---|:---|:---|:---|:---|
| 简单 | 2 | 否 | 否 | 30% 随机偏移 | 仅子力价值 |
| 中等 | 4 | 否 | 否 | 10% 随机偏移 | 子力价值 + 位置分 |
| 困难 | 6 | 是 | 否 | 无 | 完整评估函数 |
| 大师 | 8 | 是 | 是 | 无 | 完整评估函数 + Alpha-Beta 剪枝 + 迭代加深 |

**FR-XQ-AI-04** AI 引擎必须支持四级难度动态切换，难度变更在下一回合生效，不影响当前回合的走法计算。

### 8.6 界面布局

#### 8.6.1 棋盘渲染

棋盘渲染层负责绘制 9×10 交叉点网格、楚河汉界文字标识、九宫斜线、以及 32 枚棋子。棋子以圆形底座承载汉字形式渲染，红方棋子文字为红色，黑方棋子文字为黑色（或深色），圆形底座为浅木纹或浅灰底色。

**FR-XQ-UI-01** 棋盘渲染必须完整展示 9×10 交叉网格、楚河汉界文字、九宫斜线，棋子以圆形底座加汉字形式渲染，红黑双方颜色区分清晰。

#### 8.6.2 选子提示

交互反馈包含三层视觉提示：选中棋子时以金色边框（2px 实线）高亮；合法落子点以绿色圆点（半径为交叉点间距的 25%）标记；将军状态时将军方棋子以红色边框闪烁（频率 2Hz，持续 2 秒）提示。

**FR-XQ-UI-02** 系统必须在选子后 200ms 内完成合法落子点提示的渲染，将军闪烁效果必须覆盖将军方将帅棋子。

#### 8.6.3 信息面板

信息面板位于棋盘右侧或下方，显示以下信息：当前回合方标识（红方/黑方色块 + 文字）；双方累计用时（格式 MM:SS）；双方被吃子展示区（按棋子类型分组，以灰色半透明图标表示）；当前总步数计数器。面板布局应确保所有信息在单屏内可见，无需滚动。

**FR-XQ-UI-03** 信息面板必须实时更新回合方、用时、被吃子和步数信息，时间更新精度为 1 秒，步数在红黑双方各走一步后累加 1。

## 9. 麻将插件设计需求

### 9.1 游戏规则定义

#### 9.1.1 牌张组成

麻将牌张分为数牌与字牌两大类。**数牌**(Suited Tiles)包含万子(Man)、筒子(Pin)、条子(Sou)三种花色，每种花色各1至9点各4张；**字牌**(Honor Tiles)包含东南西北四风牌与白发中三元牌各4张。标准配置共136张，启用花牌(春/夏/秋/冬/梅/兰/竹/菊)后扩展至144张。牌张组成的具体规格见表9-1。

**表9-1 麻将牌张组成规格表**

| 花色类别 | 具体牌张 | 单种张数 | 牌种数 | 小计 |
|:---|:---|:---|:---|:---|
| 万子(Wan) | 一万至九万 | 4 | 9 | 36 |
| 筒子(Pin) | 一筒至九筒 | 4 | 9 | 36 |
| 条子(Sou) | 一条至九条 | 4 | 9 | 36 |
| 风牌(Wind) | 东、南、西、北 | 4 | 4 | 16 |
| 三元牌(Dragon) | 白、发、中 | 4 | 3 | 12 |
| 花牌(Flower) | 春夏秋冬梅兰竹菊(可选) | 1 | 8 | 8(可选) |
| **合计** | — | — | **43** | **136 / 144** |

表9-1中的牌张数量以"牌种数×单种张数"计算。风牌与三元牌合称**字牌**(Honor Tiles)，不计入序数牌连续组合。花牌在136张标准配置中不启用，补花动作仅在144张模式下触发。每种牌张通过全局唯一标识符进行实例追踪，区分同一牌种的不同物理拷贝。

**FR-MJ-RULE-01** 插件必须支持136张标准牌组和144张花牌牌组两种配置，通过PluginMeta配置项`tileSetSize`控制。

#### 9.1.2 核心操作

麻将的核心操作围绕手牌的获取与弃置展开。**摸牌**(Draw)指从牌山(牌墙)顶端取一张牌加入手牌；**打牌**(Discard)指从手牌中选择一张置入弃牌河；**吃**(Chow)指获取上家刚打出的牌，与手中两张连续序数牌组成顺子(如4-5-6)；**碰**(Pong)指获取任一家打出的牌，与手中两张相同牌组成刻子；**杠**(Kong)指获取或自摸形成四张相同的牌组；**胡**(Win)指手牌组合满足法定牌型要求。各操作的触发条件与约束见后续章节。

**FR-MJ-RULE-02** 吃操作仅限吃上家刚打出的牌，且必须组成同花色连续三张顺子；碰与杠操作可对任一家打出的牌触发。

#### 9.1.3 胡牌算法

胡牌判定的核心为验证手牌能否分解为合法牌型组合。基础牌型为**平胡**(Regular Hand)，由四组面子(Meld，含顺子或刻子)加一对将(Head，两张相同牌)组成。插件支持多种番种(Yaku)配置，具体牌型与判定条件见表9-2。

**表9-2 胡牌牌型与番数配置表**

| 牌型名称 | 结构要求 | 基础番数 | 说明 |
|:---|:---|:---|:---|
| 平胡 | 四面子+一对将 | 1 | 标准牌型，含顺子/刻子组合 |
| 七对 | 七个对子 | 2 | 特殊牌型，不可含任何面子 |
| 十三幺 | 全部幺九牌与字牌各一张+其中一张成对 | 役满 | 国士无双，需13种特定牌 |
| 全不靠 | 由148张中按特定组合不靠牌组成 | 役满 | 需特殊规则集支持 |
| 碰碰胡 | 四刻子+一对将 | 3 | 全为刻子/杠，无顺子 |
| 混一色 | 同一数牌+字牌组成 | 3 | 数牌仅限一种花色 |
| 清一色 | 同一数牌组成 | 6 | 不含任何字牌 |
| 混老头 | 全由幺九牌与字牌组成 | 2 | 可组成顺子或刻子 |
| 清老头 | 全由幺九牌组成 | 役满 | 仅1/9数牌，无刻子限制 |

**FR-MJ-RULE-03** 胡牌判定算法必须支持表9-2中所有番种的自动识别，番数可通过PluginMeta配置项`yakuConfig`覆盖默认数值。

**FR-MJ-RULE-04** 计分规则采用底分×2^番数倍数体系。设底分为`baseScore`，番数为`fan`，则得点计算公式为`score = baseScore × 2^fan`。封顶番数由配置项`maxFan`控制，达到封顶时不再累加计分。

#### 9.1.4 流局处理

**流局**(Exhaustive Draw)指牌山摸完时仍无人胡牌的情形。流局时执行**查花庄**：已处于**听牌**(Tenpai，距胡牌仅差一张)状态的玩家从未听牌玩家处获得约定赔付。赔付金额按 PluginMeta 配置项`notenPenalty`确定，通常为1000点/未听牌者。若四家全部听牌或全部未听牌，则不发生赔付流转。

**FR-MJ-RULE-05** 流局时系统必须自动计算所有玩家的听牌状态，执行查花庄赔付，并在结算面板显示各家的听牌/未听牌标识。

### 9.2 数据模型

#### 9.2.1 牌张对象

每张麻将牌以结构化对象表示，包含三个核心字段：`suit`(花色，枚举值Wan/Pin/Sou/Wind/Dragon/Flower)、`rank`(点数，数牌1-9，风牌0-3，三元牌0-2)、`uniqueId`(全局唯一标识符，整数递增)。牌张对象的不可变性由构造函数保证，避免游戏中途被意外篡改。

**FR-MJ-DATA-01** 牌张对象必须包含suit、rank、uniqueId三个字段，其中uniqueId在整局游戏中全局唯一，用于追踪每张牌的流转轨迹。

#### 9.2.2 牌山管理

牌山(Tile Wall)采用数组结构存储，初始顺序通过**Fisher-Yates洗牌算法**生成。算法输入为43种牌×4张的展开数组，迭代从末尾至开头，每次随机交换当前索引与前方某一索引的元素，时间复杂度O(n)，空间复杂度O(1)。发牌顺序为：从庄家开始逆时针发牌，每人每次4张，重复3轮后各发1张，庄家额外获得第14张(岭上开花牌)，最终形成13×4+1的初始牌山消耗。

**FR-MJ-DATA-02** 洗牌算法必须使用加密安全的伪随机数生成器(CSPRNG)，确保每局发牌序列的不可预测性。

#### 9.2.3 手牌分区

每个玩家的手牌空间划分为三个逻辑区域。**手牌区**(Concealed Hand)存放未公开的牌张，按花色分组、点数升序排列展示；**副露区**(Open Melds)存放已通过吃碰杠公开的牌组，每张副露牌标注操作类型(吃/碰/明杠/加杠)；**弃牌区**(Discard River)按打出先后顺序排列，四列矩阵式展示以支持牌河分析。三区状态在游戏进程中持续更新，副露区牌张从手牌区移出后不可再参与手牌内部组合。

**FR-MJ-DATA-03** 手牌区必须保持自动排序状态，排序优先级为：花色(万→筒→条→字) > 点数升序。

#### 9.2.4 听牌检测

**听牌**(Tenpai)状态指手牌距离完整胡牌牌型仅差一张牌的状态。插件必须在每次摸牌或打牌后自动计算当前玩家的听牌状态，输出**待牌**(Waiting Tiles)列表，即能使手牌完成胡牌的所有牌张集合。**振听**(Furiten)是一种限制状态，分为两种子类型：**舍牌振听**指玩家曾经打出过待牌列表中的某张牌(即使该牌当前已不再需要)；**自摸振听**指玩家拒绝对家放铳(未选择胡牌)后，该待牌进入振听状态。振听状态下玩家仅可自摸胡牌，不可荣和他家放铳。

**FR-MJ-DATA-04** 听牌检测算法必须实时计算待牌列表，并维护舍牌振听与自摸振听两种标记状态。

**向听数**(Shanten Number)是听牌检测的核心指标，定义为"距离听牌所需的最少替换牌张数"。算法采用递归分解思路：将手牌递归拆分为面子/将头/搭子(两张趋近面子形态的牌组)，统计无法组成有效牌型的剩余单牌数量，取所有拆分方案中的最小值。向听数0即为听牌状态，-1即为胡牌状态。

### 9.3 游戏状态机

#### 9.3.1 状态定义

麻将游戏状态机包含七个核心状态：**WAITING**(等待玩家)→**DEAL**(发牌)→**DRAW**(摸牌)→**DISCARD**(打牌)→**RESPOND**(响应操作)→**WIN**(胡牌结算)→**EXHAUST**(流局)→**ENDED**(终局)。状态转换遵循严格时序：DEAL完成后进入首局庄家的DRAW状态；每次DRAW后进入当前玩家的DISCARD；DISCARD后向所有其他玩家广播RESPOND窗口，等待吃碰杠胡操作请求；若无人响应则流转至下家的DRAW；WIN或EXHAUST状态下执行结算后检查是否满足终局条件(某玩家点数归零或完成设定圈数)，满足则进入ENDED，否则重新DEAL。

**FR-MJ-STATE-01** 状态机必须确保DRAW→DISCARD→RESPOND的循环在每轮正常流转中完整执行，不允许跳过中间状态。

#### 9.3.2 操作优先级

当一家打牌后，多家可能同时满足操作条件，此时依据固定优先级裁决。**截断规则**规定：胡牌优先级最高，若有玩家声明胡牌，则其他所有操作请求自动失效；其次为杠，再次为碰，最低为吃。若多家同时声明同优先级操作(如两家都碰)，则按行动顺序(打牌者下家→对家→上家)赋予优先权。

**表9-3 操作优先级裁决表**

| 操作类型 | 优先级 | 适用条件 | 截断行为 |
|:---|:---|:---|:---|
| 胡(荣和) | 1(最高) | 手牌加上所打牌满足胡牌条件 | 阻断所有其他操作 |
| 杠(大明杠) | 2 | 手牌已有三张相同牌 | 阻断碰与吃 |
| 碰 | 3 | 手牌已有两张相同牌 | 阻断吃 |
| 吃 | 4(最低) | 上家打牌且可组成顺子 | 无阻断权 |

**FR-MJ-STATE-02** 操作裁决必须遵循表9-3的优先级顺序，同优先级时按打牌者的下家→对家→上家顺序赋予优先权。

#### 9.3.3 抢杠处理

**加杠**(Added Kong)指玩家已碰出三张牌的明刻子，在摸到第四张相同牌时追加杠的动作。加杠具有被**抢杠胡**的风险：若第四张牌恰好是他家的待牌，则该家可立即声明胡牌，优先于杠的完成。**抢杠**的处理流程为：(1)玩家声明加杠；(2)系统广播加杠意图，开启他家响应窗口；(3)若有玩家声明胡牌，则胡牌生效，加杠失败，打出牌计入放铳；(4)若无胡牌响应，则加杠完成，执行岭上摸牌。

**FR-MJ-STATE-03** 加杠操作必须先广播意图并等待他家响应，胡牌声明优先于杠的完成。

### 9.4 玩家操作接口

#### 9.4.1 打牌操作

玩家从手牌区选择一张牌执行打牌操作。操作流程为：手牌选择→弃牌堆追加→清空当前玩家响应窗口→触发下家摸牌或他家RESPOND状态。若玩家在配置的思考时间内未执行打牌，系统自动打出最后摸入的牌(模切)。超时时间由PluginMeta配置项`thinkTimeMs`控制，默认15000毫秒。

**FR-MJ-ACTION-01** 打牌接口签名必须为`discard(tileId: string): DiscardResult`，参数为选中牌张的uniqueId，返回结果包含打出的牌对象与下一状态标识。

#### 9.4.2 吃碰杠操作

当一家打牌后，满足条件的玩家界面弹出操作请求窗口，窗口内显示所有可执行的牌型组合(如吃操作时列出所有可能的顺子组合)。玩家可选择某一组合确认执行，或点击取消按钮放弃操作。超时未选择视为自动放弃。操作窗口的展示时长与打牌操作共用`thinkTimeMs`配置。

**FR-MJ-ACTION-02** 操作请求窗口必须列出所有合法的牌型组合，每个组合标注所需手牌与目标牌的对应关系。

#### 9.4.3 暗杠与加杠

**暗杠**(Concealed Kong)指玩家手中持有四张相同牌时执行的杠操作，四张牌全部明置于副露区但背面朝上，不公开具体牌张直至终局。暗杠完成后从牌山尾部摸取一张**岭上牌**(Replacement Tile)补充手牌。**加杠**(Added Kong)指对已碰明刻子追加第四张牌，第四张牌明置于副露区，同时翻出新宝牌指示牌。暗杠与加杠的接口签名为`kong(type: 'concealed' | 'added', meldIndex?: number): KongResult`，加杠时需提供副露区刻子的索引。

**FR-MJ-ACTION-03** 暗杠的四张牌在副露区必须以背面朝上方式展示，加杠时必须在第四张明置后自动触发宝牌翻开展示。

### 9.5 AI设计

#### 9.5.1 打牌决策

AI的打牌决策基于手牌估值体系，核心为向听数计算与**搭子效率**(Tile Efficiency)分析。搭子效率评估每张牌在手牌中的组合贡献度：孤立牌(无相邻同花色牌)效率最低，边缘搭子(如1-2、8-9)效率较低，中央搭子(如4-5、5-6)效率最高。AI优先保留搭子效率高的牌，弃置效率低的牌。

**表9-4 AI手牌估值因素表**

| 估值因素 | 权重范围 | 说明 |
|:---|:---|:---|
| 向听数贡献 | 0.40 | 打出后向听数增加量的负值 |
| 搭子效率 | 0.25 | 该牌作为搭子构件的灵活度 |
| 牌河分析 | 0.15 | 剩余未现张数与河底可见张数比值 |
| 宝牌相关 | 0.10 | 是否为宝牌或宝牌邻张 |
| 役种前景 | 0.10 | 对目标役种(如清一色)的结构贡献 |

表9-4中的权重配置适用于困难难度AI，中等与简单难度通过调整权重或忽略部分因素实现差异化。AI在每次需要打牌时综合五项因素计算每张牌的保留价值，选择价值最低的牌弃置。

**FR-MJ-AI-01** 困难难度AI必须实现表9-4中全部五项估值因素的加权计算，实时更新牌河可见张数以调整估值。

#### 9.5.2 防守策略

AI的防守策略基于**牌河读牌**(Suji Reading)与危险度评估。**早巡**(游戏前期)外筋牌危险度较低，**晚巡**(游戏后期)生张(此前无人打过的牌)危险度显著上升。AI维护一张动态危险度矩阵，记录每张牌的放铳风险估计值。当检测到某家已听牌迹象(如突然切出好牌、副露加速)时，AI切换至防守模式，优先打出已通过牌河确认安全的牌。

**FR-MJ-AI-02** 困难难度AI必须实现基于巡目进度与对手副露模式的动态危险度评估，支持攻守模式自动切换。

#### 9.5.3 难度分级

插件提供三级AI难度配置。**简单**(Easy)难度采用纯随机策略，在所有合法打牌选项中均匀随机选择。**中等**(Medium)难度在向听数优化基础上叠加基础防守，优先选择不增加向听数的牌打出，同时规避明显的生张。**困难**(Hard)难度启用完整的牌河读牌与攻守平衡判断，根据局势动态调整进攻与防守倾向，模拟人类高阶玩家的决策逻辑。

**FR-MJ-AI-03** AI难度配置通过PluginMeta配置项`aiDifficulty`控制，可选值为`easy`/`medium`/`hard`，必须在游戏初始化时确定且不可中途变更。

### 9.6 界面布局

#### 9.6.1 四方布局

麻将界面采用固定四边形布局。自家手牌位于屏幕底部，以**扇形排列**(手牌两端略向外倾斜)展示，牌张正面朝上且清晰可读。左家手牌位于屏幕左侧，对家位于顶部，右家位于右侧，其他三家手牌均以背面朝上方式展示，仅显示张数。当前轮到行动的玩家座位边框以高亮色(金色描边)标识，弃牌流程中高亮边框持续闪烁直至操作完成。

**FR-MJ-UI-01** 四方布局必须固定对应屏幕四边，当前行动玩家的座位高亮标识必须清晰可见，高亮色值由主题系统统一提供。

#### 9.6.2 副露与弃牌

副露区位于每位玩家手牌上方，已公开的牌组以横向排列展示，每组副露左下角标注操作类型缩写(チ/ポン/カン分别代表吃/碰/杠)。弃牌区位于屏幕中央偏上区域，四家弃牌按各自方向分四列矩阵排列，每列从上到下、从右到左排列，便于玩家快速扫描牌河信息。

**FR-MJ-UI-02** 副露区必须在每组牌张旁标注操作类型标识，弃牌区必须按玩家方位分四列独立排列，最近打出的牌以轻微放大效果突出显示。

#### 9.6.3 信息面板

信息面板位于屏幕右上角，显示当前局的关键信息：宝牌指示牌(横向排列，已翻开的宝牌正面朝上)、本场数(连续庄家次数)/供托(积棒与立直棒点数累计)。听牌提示以非侵入式方式在自家手牌上方显示待牌图标。胡牌时界面中央播放番种名称与得点数的动画效果，动画持续2000毫秒后自动进入结算面板。

**FR-MJ-UI-03** 信息面板必须实时更新宝牌状态与场况数据，胡牌动画必须依次展示番种列表、番数合计、得点数三个信息层。

## 10. 斗地主插件设计需求

斗地主插件（gameId="doudizhu"）实现三人扑克牌游戏，属于"牌类"分类，固定3人参与（minPlayers=3, maxPlayers=3），通过叫分确定地主后进入对抗阶段。插件支持底分、思考时间、AI难度、是否允许不出（Pass）四项配置参数。

### 10.1 游戏规则定义

#### 10.1.1 牌张组成与玩家配置

**FR-DDZ-RULE-01** 斗地主必须使用54张扑克牌，包含52张标准牌（四种花色×13个牌点）与2张王牌（小王、大王）。四种花色为黑桃（Spade）、红桃（Heart）、方块（Diamond）、梅花（Club），不影响牌力比较，仅用于区分同点数的牌张。牌点从大到小依次为：大王 > 小王 > 2 > A > K > Q > J > 10 > 9 > 8 > 7 > 6 > 5 > 4 > 3。

**FR-DDZ-RULE-02** 游戏固定为3人配置，通过随机发牌每人获得17张手牌，剩余3张作为底牌。底牌在叫分阶段结束后归并至地主手牌，地主最终持有20张牌。

#### 10.1.2 叫分抢地主流程

**FR-DDZ-RULE-03** 叫分阶段由系统随机指定首家叫分玩家，按顺时针方向依次进行。每位玩家可选择叫1分、2分、3分或选择Pass（不叫）。叫分分数表示玩家对自身牌力的自信程度，分数越高牌力评估越强。若某玩家叫出3分，叫分流程立即终止，该玩家直接成为地主。若一轮结束后最高叫分低于3分，则叫分继续进行，已Pass的玩家不再参与后续叫分。

**FR-DDZ-RULE-04** 叫分阶段的完整流程遵循以下步骤：步骤1，系统随机指定首家叫分者；步骤2，当前玩家选择叫分（1/2/3）或Pass，若叫3分则跳转步骤5；步骤3，顺时针转移到下一位未Pass的玩家；步骤4，若仍有未叫分玩家则返回步骤2，否则比较已叫分玩家的最高分数；步骤5，最高分者为地主，若三家均Pass则重新发牌。3分立即终止规则的设计目的是避免牌力极强的玩家被低分抢地主，同时加快游戏节奏。

#### 10.1.3 牌型大小比较

**FR-DDZ-RULE-05** 斗地主支持十种牌型，牌型大小比较遵循"同类型比较点数、不同类型仅炸弹可压制"的核心原则。具体牌型定义及大小规则如下表所示。

| 牌型名称 | 牌张构成 | 示例 | 大小比较规则 |
|:---|:---|:---|:---|
| 单张 | 任意1张牌 | 黑桃5 | 按牌点比较，大王最大、3最小 |
| 对子 | 相同牌点的2张牌 | 红桃8+方块8 | 按牌点比较，需牌点大于上家对子 |
| 三张 | 相同牌点的3张牌 | 梅花K+红桃K+方块K | 按牌点比较 |
| 三带一 | 三张+任意1张单牌 | QQQ+5 | 比较三张部分的牌点，附带牌不参与比较 |
| 三带二 | 三张+任意1个对子 | JJJ+88 | 比较三张部分的牌点，附带对子不参与比较 |
| 顺子 | 5张及以上连续牌点 | 34567、10JQKA | 比较最大牌点，A可作为顺子顶端但不可循环（如KQJA2不合法） |
| 连对 | 3个及以上连续对子 | 334455、7788991010 | 比较最大对子的牌点，要求连续对子数量相同 |
| 飞机 | 2个及以上连续三张 | JJJQQQ、777888999 | 比较最大三张的牌点，带牌数量需匹配飞机组数 |
| 炸弹 | 相同牌点的4张牌 | AAAA | 按牌点比较，任意炸弹大于所有非炸弹牌型 |
| 王炸 | 小王+大王 | 小王+大王 | 最大牌型，可压任何其他牌型 |

牌型大小比较遵循三条核心规则。第一，相同牌型之间按牌点比较大小，顺子、连对、飞机还需确保牌张数量一致。第二，炸弹可以压制所有非炸弹牌型，不同炸弹之间按牌点比较。第三，王炸为全局最大牌型，可压制包括炸弹在内的任何牌型。三带一与三带二属于不同类型，不可互相比大小；飞机带翅膀时，翅膀可为单牌或对子形式，但需与飞机组数匹配（如JJJQQQ可带两张单牌或两个对子）。

#### 10.1.4 出牌规则

**FR-DDZ-RULE-06** 每局首出由地主开始，地主获得底牌后拥有首出权。首家出牌时可选择任意合法牌型。后续玩家出牌时必须满足两个条件：出牌牌型与当前最大出牌的牌型一致（或出炸弹/王炸），且牌力严格大于当前最大出牌。玩家也可选择Pass放弃本轮出牌。当连续两位玩家Pass后，最后出牌者获得下一轮首出权，可自由选择牌型。

#### 10.1.5 胜负与计分

**FR-DDZ-RULE-07** 胜负判定采用以下规则：地主一方先出完所有手牌则地主获胜；农民任一方先出完所有手牌则两名农民共同获胜。计分公式为：胜者得分 = 底分 × 叫分分数 × 倍数。其中底分由房间配置决定，叫分分数为叫分阶段确定的1/2/3分，倍数包含炸弹倍率（每个炸弹×2）与春天/反春天倍率（×2）。负者等分扣除相应分数。

### 10.2 数据模型

#### 10.2.1 牌张对象设计

**FR-DDZ-DATA-01** 每张牌的数据结构包含三个字段：rank（牌点，枚举值为3-10、J、Q、K、A、2、小王、大王）、suit（花色，枚举值为黑桃/红桃/方块/梅花，王牌花色为特殊标记JOKER）、type（牌型分类辅助字段，用于快速归类手牌中的单张/对子/三张/炸弹）。牌张对象需实现Comparable接口，支持基于rank的全局大小比较。

#### 10.2.2 发牌与底牌管理

**FR-DDZ-DATA-02** 发牌算法采用Fisher-Yates洗牌后对54张牌进行均匀分配，玩家0/1/2各获得17张，最后3张存入底牌池。底牌池在叫分结束前处于锁定状态，地主确定后将3张底牌合并至地主手牌数组，并触发手牌重排序。底牌支持明置（所有玩家可见）与暗置（仅地主可见）两种显示模式，由房间配置决定。

#### 10.2.3 手牌与出牌区管理

**FR-DDZ-DATA-03** 每位玩家维护两个牌张数组：handCards（当前手牌，初始17张，地主为20张）与playedCards（本局已出牌记录，用于结算统计）。出牌操作时，系统从handCards中移除指定牌张并追加至出牌区的本轮记录。手牌管理模块需实现牌型自动识别功能，输入一组牌张后输出其可能构成的所有合法牌型列表，用于出牌提示与AI决策。

#### 10.2.4 回合记录追踪

**FR-DDZ-DATA-04** 每轮出牌维护回合记录对象，包含以下字段：currentHighestCards（当前最大出牌牌张数组）、currentHighestPlayer（当前最大出牌玩家ID）、currentPattern（当前牌型枚举）、consecutivePasses（连续Pass计数，达到2时触发回合重置）、roundStarter（本轮首出玩家ID）。回合记录在每次出牌或Pass操作后更新，用于合法性校验与状态判断。

### 10.3 游戏状态机

#### 10.3.1 状态定义与转换

**FR-DDZ-STATE-01** 斗地主游戏状态机包含六个核心状态，状态转换遵循严格的有向路径。WAITING（等待状态）为初始状态，3名玩家全部就位后由系统触发进入DEAL（发牌状态），执行洗牌与发牌逻辑后自动进入BID（叫分状态）。BID状态下玩家轮流叫分，地主确定后进入LANDLORD（地主确认状态），底牌归并地主后转入PLAY（出牌状态）。PLAY状态下进行多轮出牌对抗，直至任意一方手牌出完触发ENDED（结束状态），结算完成后可选择返回WAITING开启新局。

#### 10.3.2 春天与反春天判定

**FR-DDZ-STATE-02** 春天判定规则定义如下：地主出完所有20张手牌的过程中，两名农民均未出过任何一张牌（即地主连续首轮至末轮持续出牌，农民始终Pass），此时触发春天效果，结算倍数额外乘以2。反春天判定规则为：任一农民出完所有17张手牌的过程中，地主未出过任何一张牌，此时触发反春天效果，结算倍数同样额外乘以2。春天与反春天的检测依赖于对PLAY状态下每轮出牌玩家记录的完整追踪，系统在状态转入ENDED时扫描出牌历史进行判定。春天倍数与炸弹倍数采用累乘方式计算，而非累加。

### 10.4 玩家操作接口

#### 10.4.1 叫分操作

**FR-DDZ-ACTION-01** 叫分操作接口接收参数为bidValue（枚举值：0表示Pass、1/2/3表示叫分分数）与playerId（玩家ID）。接口执行以下校验：当前状态必须为BID、playerId必须为当前待叫分玩家、已Pass玩家不可再次叫分。若bidValue为3分，立即设置地主并终止叫分流程；否则记录当前最高叫分与对应玩家，转移至下一位未Pass玩家。若所有未Pass玩家叫分完毕且无3分者，最高分者中标为地主。

#### 10.4.2 出牌操作

**FR-DDZ-ACTION-02** 出牌操作接口接收参数为cards（牌张数组）与playerId。校验流程分为两步：第一步验证cards在playerId手牌中存在且构成合法牌型，牌型识别算法遍历cards的rank频次分布，通过模式匹配判断是否符合十种牌型之一的构成规则；第二步验证cards是否大于当前最大出牌——当本轮为首出时无需比较，否则要求牌型匹配（或cards为炸弹/王炸）且牌点大于currentHighestCards。校验通过后从手牌扣除cards，更新回合记录，并广播出牌动画事件。

牌型识别算法的核心逻辑基于牌点频次统计。算法输入为一组牌张，输出为识别的牌型及比较键值。具体流程为：统计各rank出现次数生成频次映射，根据频次模式判断牌型——单张（最大频次1且总张数1）、对子（最大频次2且总张数2）、三张（最大频次3且总张数3）、三带一（频次模式3+1）、三带二（频次模式3+2）、炸弹（最大频次4）、顺子（连续5+张各频次1）、连对（连续3+对子各频次2）、飞机（连续2+三张各频次3）、王炸（大小王组合）。

#### 10.4.3 不出操作

**FR-DDZ-ACTION-03** Pass操作接口接收playerId参数，校验条件为：当前状态为PLAY、playerId为当前待出牌玩家、该玩家不是本轮首出玩家（首家不可Pass）。Pass操作将consecutivePasses计数加1，若计数达到2则重置回合（currentHighestCards清空、consecutivePasses归零、roundStarter设为最后出牌玩家），否则转移至下家。Pass操作同样广播事件供界面更新。

### 10.5 AI设计

#### 10.5.1 牌力评估模型

**FR-DDZ-AI-01** AI牌力评估模块综合考虑以下因素计算手牌综合得分，评估因素及权重如下表所示。

| 评估因素 | 计算方法 | 权重说明 |
|:---|:---|:---|
| 大牌控制率 | 手牌中A/2/王/炸弹的数量占总大牌比例 | 核心指标，决定叫分上限与角色定位 |
| 跑牌概率 | 基于手牌结构估算几轮内出完的概率 | 影响出牌策略激进或保守程度 |
| 牌型完整度 | 已有牌型种类数与可直接走顺子/连对的数量 | 牌型越多出牌灵活性越高 |
| 单牌负担 | 无法组成任何牌型的单张牌数量 | 单牌过多需保留小牌垫牌 |
| 炸弹威慑力 | 手牌炸弹数量及牌点等级 | 炸弹影响叫分决策与出牌节奏 |

牌力评估的输出为0-100分的综合评分，评分阈值划分如下：80分以上建议叫3分抢地主，60-80分建议叫2分，40-60分建议叫1分，40分以下建议Pass。评分在每轮出牌后根据手牌变化动态重新计算，AI据此调整出牌策略。

#### 10.5.2 农民配合策略

**FR-DDZ-AI-02** 双农民模式下AI需实现主攻/助攻角色判断与配合机制。角色判定逻辑为：两名农民各自计算牌力评分，评分较高者承担主攻角色（优先跑牌），评分较低者承担助攻角色（配合拦截地主）。主攻策略以快速减少手牌为目标，优先出顺子、连对等长牌型；助攻策略以拦截地主出牌为目标，保留大牌（2、A、炸弹）压制地主关键牌型。信号传递机制通过出牌行为间接实现：助攻农民出小牌暗示主攻农民可放心跑牌，助攻农民出大牌拦截暗示主攻农民暂时不要拆牌型。

#### 10.5.3 难度分级设计

**FR-DDZ-AI-03** AI系统提供三个难度等级。简单难度AI随机选择任意合法出牌（含Pass），不考虑牌力比较与配合策略，叫分阶段随机叫1分或Pass。中等难度AI采用牌型优先策略，优先出完整牌型（顺子、连对、飞机），保留单牌最后出；叫分阶段基于固定评分阈值决策；农民之间实现基础配合（不互相顶牌）。困难难度AI实现完整记牌功能（追踪已出牌推算剩余牌分布），基于动态牌力评估规划最优出牌序列，农民之间实现最优配合（主攻/助攻角色动态切换、主动传递跑牌信号）。

### 10.6 界面布局

#### 10.6.1 三方座位布局

**FR-DDZ-UI-01** 界面采用经典三方布局：自家手牌位于屏幕底部横向排列，牌面朝上完整显示；左侧与右侧农民以头像+剩余牌数方式简化显示，农民手牌以牌背形式竖向叠放。地主玩家头像旁显示皇冠图标，农民头像旁显示锄头图标，明确标识身份。三名玩家出牌区域分别位于底部中央、左侧中央、右侧中央，出牌时牌面在对应区域横向展开显示。

#### 10.6.2 底牌与倍数显示

**FR-DDZ-UI-02** 底牌区域位于屏幕中央偏上位置，叫分阶段以牌背暗置显示3张底牌；地主确定后切换为牌面明置，并以动画形式飞入地主手牌区。屏幕顶部中央显示当前倍数信息，包含底分、叫分、炸弹数量、春天状态的实时乘积值，每次炸弹或春天触发时倍数数字以缩放动画高亮更新。

#### 10.6.3 出牌提示与交互

**FR-DDZ-UI-03** 自家出牌阶段，系统在手牌区以高亮边框标记推荐出牌组合，推荐逻辑基于牌型识别算法输出的大于当前最大出牌的最小牌型方案。界面中央显示牌型自动识别文字提示（如"三条J带一对8"）。操作按钮区域包含"出牌"按钮（需选中合法牌型后激活）、"不出/Pass"按钮（首家不可点时置灰）、"提示"按钮（循环显示可用出牌方案）。出牌倒计时以环形进度条围绕头像显示，超时未操作则自动执行Pass或AI代打（由配置决定）。

## 11. 德州扑克插件设计需求

德州扑克（Texas Hold'em）插件基于框架的 `GameController` 接口实现标准无限制德州扑克（No-Limit Texas Hold'em）逻辑。插件元数据 `PluginMeta` 的 `gameId` 为 `"texas_holdem"`，`category` 为 `"牌类"`，`minPlayers` 为 2，`maxPlayers` 为 9，`supportedModes` 包含 `"cash"` 和 `"tournament"` 两种对局模式。现金桌（Cash Game）模式下玩家可随时离座，筹码按实际盈亏结算；锦标赛（Tournament）模式下所有玩家以等额买入获得统一筹码量，淘汰至最后一名或达成约定名次时结束。插件注册时须通过 `getSettingsSchema()` 暴露以下配置项：小盲注（Small Blind）与大盲注（Big Blind，默认比例为 1:2）、买入下限/上限（Buy-in Min/Max，现金桌通常为 20~100 倍大盲注）、思考时间（15/30/45/60 秒可选）、AI 难度（easy/medium/hard）、对局模式（cash/tournament）。

### 11.1 游戏规则定义

#### 11.1.1 基本配置

**FR-POKER-RULE-01** 牌张：德州扑克使用一副 52 张标准扑克牌，不包含大小王。每局结束后已发牌张须重新洗牌，洗牌算法须调用框架提供的认证 RNG 接口（NFR-FAIR-01）。**FR-POKER-RULE-02** 人数：牌桌支持 2~9 名玩家同时就座，其中 2 人模式称为单挑（Heads Up）。**FR-POKER-RULE-03** 盲注与庄家：每局指定一名庄家（Dealer），其位置以标记表示。庄家左侧第一位为小盲注位（Small Blind，SB），第二位为大盲注位（Big Blind，BB）。每局结束后庄家位按顺时针方向移动一格，SB 与 BB 随之同步轮换。在单挑模式下，庄家同时充当小盲注位，大盲注位为其左侧唯一对手。

#### 11.1.2 牌局阶段

**FR-POKER-RULE-04** 牌局完整流程由以下五个阶段构成：Preflop（翻牌前）、Flop（翻牌圈）、Turn（转牌圈）、River（河牌圈）、Showdown（摊牌）。各阶段的触发条件与执行动作如下表所示。

| 阶段名 | 触发条件 | 执行动作 | 下一阶段 |
|--------|----------|----------|----------|
| Preflop | 所有玩家就位、SB/BB 强制下注完成 | 每位玩家发 2 张暗牌（Hole Cards）；从 BB 左侧玩家开始第一轮行动 | Flop |
| Flop | Preflop 下注轮闭合（所有活跃玩家下注额相等或全押） | 桌面中央翻开 3 张公共牌（Community Cards）；开始新一轮下注 | Turn |
| Turn | Flop 下注轮闭合 | 翻开第 4 张公共牌；开始新一轮下注 | River |
| River | Turn 下注轮闭合 | 翻开第 5 张公共牌；开始最后一轮下注 | Showdown |
| Showdown | River 下注轮闭合 | 所有剩余玩家亮牌比较；按牌型大小分配底池 | ENDED |

进入下一阶段前，当前下注轮的闭合条件必须严格满足：所有仍然活跃（未弃牌）的玩家中，每人本轮投入额相等，或已处于全押（All-in）状态且无法再行动。此外，若仅剩一名未弃牌玩家，则该玩家直接赢得底池，牌局立即跳至 ENDED 状态，不再发出后续公共牌。

#### 11.1.3 下注动作

**FR-POKER-RULE-05** 系统须支持德州扑克的全部六种标准下注动作。各动作的语义、可用条件与执行效果如下表所示。

| 动作名 | 语义 | 可用条件 | 执行效果 |
|--------|------|----------|----------|
| Fold | 弃牌 |  always available（除非已 All-in 或牌局已结束） | 玩家手牌作废，不再参与本局，已下注筹码不予退还 |
| Check | 让牌/过牌 | 当前最大下注额等于该玩家本轮已下注额 | 不下注，将行动权传递给下一位玩家 |
| Call | 跟注 | 当前最大下注额大于该玩家本轮已下注额 | 补足差额至当前最大下注额，进入底池 |
| Bet | 下注 | 当前轮无人下过注（所有玩家均为 Check 状态或首轮 BB 后） | 投入指定筹码成为当前轮最大下注额，开启下注轮 |
| Raise | 加注 | 已有人下注，且玩家筹码足以满足最小加注额 | 在当前最大下注额基础上增加至少最小加注额 |
| All-in | 全押 | 玩家剩余筹码不足以 Call 或达到最小加注额 | 将所有剩余筹码投入底池，本轮不再被要求追加下注 |

在单挑模式下，SB 玩家在 Preflop 阶段的行动顺序优先于 BB，此顺序差异须在 `getLegalActions()` 的实现中体现。Bet 动作与 Raise 动作的区分取决于当前轮是否已有玩家下注：Bet 是首轮下注，Raise 是在已有下注基础上追加。

#### 11.1.4 下注轮闭合

**FR-POKER-RULE-06** 下注轮闭合须同时满足以下两个条件之一：（1）除全押玩家外，所有活跃玩家本轮累计下注额严格相等；（2）仅剩一名未弃牌玩家。每轮下注以顺时针方向进行，从当前阶段规定的起始位置开始，至最后一名做出非被动行动（Bet/Raise/All-in 导致他人需响应）的玩家右侧第一位玩家行动后结束。若一轮完整循环中无人做出主动加注，则轮次自动闭合。

#### 11.1.5 边池计算

**FR-POKER-RULE-07** 当两名及以上玩家在同一轮或不同轮次执行 All-in 且其下注额不相等时，系统必须创建边池（Side Pot）。边池的分配遵循"赢者得其所应得"原则：每个边池仅由参与形成该边池的玩家竞争。

边池创建的核心算法逻辑如下。首先，收集所有未弃牌玩家的本轮总下注额，按从小到大的顺序排序。初始化一个空边池列表和一个累计变量 `running_total = 0`。遍历排序后的玩家下注额（去重处理），对于每个不同的下注额度 `wager`：计算该层级对应的有效下注为 `wager - running_total`；遍历所有下注额大于等于 `wager` 的玩家，将其纳入当前边池的竞争者集合；创建边池对象，记录该边池的金额（有效下注 × 竞争者数量）和归属玩家集合；将 `running_total` 更新为 `wager`。遍历完成后，主池（Main Pot）对应最小下注额层级的边池，其余为边池 1、边池 2……按创建顺序编号。摊牌阶段，先以完整 7 张牌（2 张底牌 + 5 张公共牌）从每个边池的竞争者集合中选出最大牌型持有者分配该边池；若出现平局，则该边池由平局者均分（遇奇数筹码时按座位号近庄家位者优先分配余数）。

#### 11.1.6 牌型比较

**FR-POKER-RULE-08** 牌型比较以每名未弃牌玩家的 2 张底牌与 5 张公共牌中任选 5 张组成最佳牌型。系统须支持完整的 10 级牌型比较，同级牌型之间按踢脚（Kicker）机制决胜。

| 排名 | 牌型名（中文/英文） | 示例 | 比较规则 |
|------|---------------------|------|----------|
| 1 | 皇家同花顺 Royal Flush | A♠ K♠ Q♠ J♠ 10♠ | 同为皇家同花顺时平局 |
| 2 | 同花顺 Straight Flush | 9♥ 8♥ 7♥ 6♥ 5♥ | 比较最大牌面点数，相同则平局 |
| 3 | 四条 Four of a Kind | K♦ K♥ K♠ K♣ 3♦ | 先比较四条点数，再比较单张踢脚 |
| 4 | 葫芦 Full House | Q♠ Q♥ Q♦ 8♣ 8♠ | 先比较三条点数，再比较对子点数 |
| 5 | 同花 Flush | A♦ J♦ 8♦ 5♦ 2♦ | 从最大牌开始逐张比较点数 |
| 6 | 顺子 Straight | 8♠ 7♥ 6♦ 5♣ 4♠ | 比较最大牌面点数；A-2-3-4-5 为最小顺子 |
| 7 | 三条 Three of a Kind | 7♣ 7♦ 7♠ K♥ 4♦ | 先比较三条点数，再依次比较两张踢脚 |
| 8 | 两对 Two Pair | A♣ A♦ 9♠ 9♥ 5♣ | 先比较大对子，再比较小对子，最后比较踢脚 |
| 9 | 一对 One Pair | 10♠ 10♥ A♦ 7♣ 3♠ | 先比较对子点数，再依次比较三张踢脚 |
| 10 | 高牌 High Card | A♦ Q♠ 8♥ 5♣ 2♦ | 从最大牌开始逐张比较点数 |

牌面点数排序（从大到小）为：A > K > Q > J > 10 > 9 > 8 > 7 > 6 > 5 > 4 > 3 > 2。顺子中的 A 仅在 10-J-Q-K-A 组合中视为最大点，在 A-2-3-4-5 组合中视为最小点，不可绕回（如 Q-K-A-2-3 不构成顺子）。踢脚比较时，牌面点数优先于花色；仅当牌面点数完全相同时才视为平局，花色不用于决胜。

### 11.2 数据模型

#### 11.2.1 牌张对象

**FR-POKER-DATA-01** `Card` 对象须包含以下字段：`rank`（枚举值：2~10、J、Q、K、A）、`suit`（枚举值：♠ Spades、♥ Hearts、♦ Diamonds、♣ Clubs）、`faceUp`（布尔值，true 为明牌、false 为暗牌）。`Card` 须提供序列化方法，支持在网络传输与回放日志中重建。

#### 11.2.2 牌桌状态

**FR-POKER-DATA-02** `TableState` 对象维护牌桌实时数据，包含：底池总额（`mainPot`，所有边池金额之和）、边池列表（`sidePots`，按 11.1.5 算法生成的 `SidePot` 数组，每项含金额和竞争者 ID 集合）、当前轮最大下注额（`currentBet`）、最小加注额（`minRaise`，按 11.3.3 公式动态计算）、庄家位座位号（`dealerPosition`）、小盲注位座位号（`smallBlindPosition`）、大盲注位座位号（`bigBlindPosition`）、当前行动位座位号（`activePosition`）。

#### 11.2.3 手牌与公共牌

**FR-POKER-DATA-03** 每名玩家的底牌（Hole Cards）以 `faceUp = false` 状态存储，仅该玩家本人与回放系统（需授权）可见。**FR-POKER-DATA-04** 公共牌（Community Cards）共 5 张，分三阶段揭示：Flop 阶段将第 1~3 张的 `faceUp` 设为 true，Turn 阶段将第 4 张设为 true，River 阶段将第 5 张设为 true。`TableState` 中须维护 `communityCards: Card[5]` 数组和当前已揭示数量 `communityRevealed`（取值 0/3/4/5）。

#### 11.2.4 下注历史

**FR-POKER-DATA-05** 系统须为每局牌维护完整的下注历史 `BettingHistory`，记录每一轮下注中每位玩家的动作序列。每条记录包含：轮次标识（Preflop/Flop/Turn/River）、玩家 ID、动作类型（Fold/Check/Call/Bet/Raise/All-in）、金额、时间戳。**FR-POKER-DATA-06** 下注历史须支持持久化存储和回放重建，回放时可通过 `replayFromLog()` 按时间顺序逐步重放，完整还原牌局过程。

### 11.3 游戏状态机

#### 11.3.1 阶段转换

**FR-POKER-STATE-01** 德州扑克插件的状态机继承框架通用状态机，细化为以下状态转换序列：WAITING（等待玩家加入）→ PREFLOP（发底牌、第一轮下注）→ FLOP（翻 3 张公共牌）→ TURN（翻第 4 张）→ RIVER（翻第 5 张）→ SHOWDOWN（摊牌比牌）→ ENDED（本局结束）。WAITING 状态下牌桌等待至少 2 名玩家就座，达到最低人数且所有玩家标记"准备"后自动进入 PREFLOP。**FR-POKER-STATE-02** 从 PREFLOP 到 FLOP、FLOP 到 TURN、TURN 到 RIVER 的转换条件均为当前下注轮闭合（参见 11.1.4）。**FR-POKER-STATE-03** RIVER 到 SHOWDOWN 的转换下注轮闭合后触发；SHOWDOWN 到 ENDED 在底池分配完成后触发。若在任意阶段仅剩一名未弃牌玩家，状态直接跳转至 ENDED，跳过所有后续阶段。

#### 11.3.2 All-in 摊牌

**FR-POKER-STATE-04** 当所有未弃牌玩家均处于 All-in 状态时，若当前公共牌尚未完全发出，牌局须提前进入 SHOWDOWN 阶段。**FR-POKER-STATE-05** 在 All-in 摊牌模式下，已发出的公共牌保持可见，剩余未发公共牌由系统自动依次发出（依次经过 FLOP_COMPLETE、TURN_COMPLETE、RIVER_COMPLETE 中间状态，无需下注轮），发完后立即比较所有玩家的最佳牌型。**FR-POKER-STATE-06** All-in 摊牌时，所有玩家的底牌自动亮牌（`faceUp` 强制设为 true），不再提供主动亮牌/弃牌选项。

#### 11.3.3 最小加注额动态计算

**FR-POKER-STATE-07** 最小加注额的计算公式为：`minRaise = currentBet + (currentBet - previousBet)`，其中 `currentBet` 为当前轮最大下注额，`previousBet` 为当前轮次中前一次最大下注额（即本轮最后一次 Raise 之前的下注额）。在 Preflop 轮中，`previousBet` 初始值为大盲注额（Big Blind）；在 Flop/Turn/River 轮中，`previousBet` 初始值为 0（因为首轮允许 Bet）。例如，若 BB = 20，某玩家将下注额从 20 Raise 至 50，则 `currentBet = 50`、`previousBet = 20`，此时其他玩家若要 Raise，最小加注额为 `50 + (50 - 20) = 80`，即至少须下注至 80。该公式确保每次加注的幅度不小于上一次加注的幅度，防止微小加注干扰牌局节奏。

### 11.4 玩家操作接口

#### 11.4.1 操作可用性

**FR-POKER-ACTION-01** `getLegalActions(playerId)` 方法须根据以下因素动态计算该玩家当前可选操作列表：玩家剩余筹码量、当前轮最大下注额与该玩家本轮已下注额的差值、牌局当前阶段、该位置的行动历史。若玩家剩余筹码 ≤ 跟注所需差额，则 Call 不可用，All-in 成为唯一可行选择（若不愿 Fold）。**FR-POKER-ACTION-02** 当当前轮最大下注额为 0 时（如 Flop 圈所有前位玩家均 Check），玩家可选操作集合中须包含 Check 与 Bet，不含 Call 与 Raise。**FR-POKER-ACTION-03** 操作提交后，`submitAction()` 须对操作合法性进行二次校验，非法操作（如不足最小加注额的 Raise）须拒绝并返回错误码，不修改游戏状态。

#### 11.4.2 超时处理

**FR-POKER-ACTION-04** 系统须支持可配置的超时处理策略。当玩家在设定思考时间内未提交操作时，自动行为规则如下：若当前无可跟注额（即 Check 可用），自动执行 Check；若存在可跟注额但玩家筹码不足以 Call，自动执行 All-in；若存在可跟注额且玩家筹码足够，自动执行 Fold。**FR-POKER-ACTION-05** 超时策略须在房间配置中预设（`timeoutAction` 字段），支持 "auto_fold" 与 "auto_check_fold" 两种模式。超时后当前行动位自动移交给下一位玩家，超时次数累计值在 `PlayerState` 中维护。

#### 11.4.3 亮牌控制

**FR-POKER-ACTION-06** SHOWDOWN 阶段，未在过程中弃牌的玩家须在摊牌时决定是否亮牌。若该玩家在最后一轮主动下注或加注且无人跟注至摊牌，或该玩家在 River 轮做出 All-in，则其底牌强制亮牌。**FR-POKER-ACTION-07** 其他情况下，玩家可选择主动亮牌（Show）或不亮弃牌（Muck，即放弃手牌但不展示）。选择 Muck 的玩家不参与牌型比较，但若其为唯一剩余竞争者（他人均 Muck），则自动赢得底池而无需亮牌。**FR-POKER-ACTION-08** Muck 操作的结果信息仅在该玩家客户端显示，其他玩家视角中该手牌标记为 "Mucked"，不揭示实际牌张。

### 11.5 AI设计

#### 11.5.1 EV决策框架

**FR-POKER-AI-01** 中等及以上难度的 AI 须基于期望值（Expected Value，EV）框架进行决策。EV 计算公式为：`EV = (赢率 × 赢得底池金额) - (输率 × 需投入金额)`。其中"赢率"（Equity）通过蒙特卡洛模拟估算：随机抽取剩余未现牌张组合，在大量样本（默认 10,000 次迭代）中统计当前手牌在各边池竞争者中的胜率占比。**FR-POKER-AI-02** AI 在决策时遍历所有合法操作，计算每个操作的预期 EV，选择 EV 最大的操作执行。若所有可选操作的 EV 均为负值，则选择 Fold（当 Fold 可用时）或 Check（当 Check 可用时）中 EV 较高的一个。

#### 11.5.2 对手建模

**FR-POKER-AI-03** 困难难度 AI 须实现对手建模（Opponent Modeling）机制，动态识别并记录各对手的统计特征。系统维护每名对手的多维指标：VPIP（Voluntarily Put Money In Pot， voluntarily 参与底池率）、PFR（Pre-flop Raise，翻牌前加注率）、AF（Aggression Factor，下注与加注次数 / 跟注次数）、3Bet 率、C-Bet 率、摊牌率（Went to Showdown %）。**FR-POKER-AI-04** AI 根据对手指标将其归类为四种风格之一：紧凶（TAG，低 VPIP + 高 AF）、松凶（LAG，高 VPIP + 高 AF）、紧弱（低 VPIP + 低 AF）、松弱（高 VPIP + 低 AF）。风格标签随统计样本量增加而动态更新，初始时以松弱为默认假设。**FR-POKER-AI-05** AI 根据对手风格调整决策：面对 TAG 型对手时收紧手牌范围、减少诈唬；面对 LAG 型对手时以强牌进行价值下注；面对松弱型对手时增加 C-Bet 频率；面对紧弱型对手时增加翻牌前偷盲频率。

#### 11.5.3 难度分级

**FR-POKER-AI-06** AI 须提供三级难度。简单（Easy）难度基于牌力阈值决策：预定义手牌分类表（强牌/中牌/弱牌），仅在持有强牌时下注，中牌时跟注或 Check，弱牌时 Fold，不考虑位置与对手行为。**FR-POKER-AI-07** 中等（Medium）难度在 EV 计算基础上引入位置意识：后位（靠近庄家）行动时手牌范围适当放宽，前位时收紧；加入底池赔率（Pot Odds）比较，当底池赔率大于手牌胜率时选择跟注。**FR-POKER-AI-08** 困难（Hard）难度完整实现对手建模（11.5.2）、诈唬检测与诈唬执行（在有利位置对紧弱型对手进行定时诈唬），以及锦标赛模式下的独立筹码模型（ICM，Independent Chip Model）决策：在接近奖金泡沫时调整策略，优先避免淘汰而非最大化筹码增长。

### 11.6 界面布局

#### 11.6.1 环形牌桌

**FR-POKER-UI-01** 插件游戏视图须渲染环形牌桌，2~9 个座位沿圆周均匀分布，座位号固定从 0 开始按顺时针编号。每个座位区域显示玩家头像、昵称、当前筹码量、本局累计下注额（以筹码堆叠视觉效果呈现）、玩家状态标签（等待中/行动中/已弃牌/全押）。**FR-POKER-UI-02** 当前行动位须以持续脉冲高亮动画标识（如座位边框呼吸灯效果），动画颜色与速度可根据玩家剩余思考时间变化：正常阶段为白色/绿色渐变，剩余 5 秒内切换为红色快闪以提示紧迫。**FR-POKER-UI-03** 庄家位、SB 位、BB 位分别以 D/S/B 标记在座位旁显示，标记随每局轮换自动更新位置。

#### 11.6.2 公共牌动画

**FR-POKER-UI-04** 公共牌的发牌过程须分三段动画呈现。Flop 阶段：3 张牌以同步翻转动画从桌面中央背面朝上翻至正面，翻转时长 400ms，3 张牌翻转间隔 80ms。Turn 阶段：第 4 张牌从牌堆位置滑至公共牌区后执行翻转动画，时长 500ms。River 阶段：第 5 张牌动画与 Turn 阶段一致。**FR-POKER-UI-05** 每个阶段完成时须在公共牌区上方短暂显示阶段标签（"Flop"、"Turn"、"River"），显示时长 1.5 秒后自动淡出，标签文字通过框架国际化接口 `context.localize()` 获取以支持多语言。

#### 11.6.3 手牌评估

**FR-POKER-UI-06** 系统须在客户端提供实时手牌评估提示功能：根据玩家当前的 2 张底牌与已揭示的公共牌，实时计算并显示当前最佳牌型（如"一对 A"、"同花听牌"等）。提示文字显示在玩家底牌区域下方，字号适中、半透明背景避免遮挡操作按钮。**FR-POKER-UI-07** 手牌评估提示须为可开关选项，在设置面板中以 "Show Hand Evaluation" 复选框呈现，默认开启。该评估仅在本地客户端计算，不向服务器发送额外请求，也不向其他玩家暴露信息。

## 12. 非功能需求

### 12.1 性能需求

#### 12.1.1 服务端延迟

框架服务端必须满足低延迟通信要求，确保玩家在实时对局中获得即时反馈。**NFR-PERF-01** 单房间单条消息的处理延迟（从服务端接收至完成处理）P99 必须≤50ms，该指标涵盖操作合法性校验、状态机驱动和结果计算的全流程。**NFR-PERF-02** 状态广播延迟（服务端处理完成后至所有房间成员收到状态更新消息）P99 必须≤100ms，此指标依赖于 WebSocket 连接质量和网关层的消息分发效率。**NFR-PERF-03** 同一房间内所有玩家的状态变更事件广播顺序必须保持一致，严禁出现乱序或丢包现象，服务端须以房间为单位维护单调递增的事件序列号。

#### 12.1.2 客户端性能

客户端渲染性能直接影响用户操作体验。**NFR-PERF-04** 游戏内主界面（牌桌、棋盘场景）的 UI 渲染帧率必须稳定在 60fps，掉帧率不得超过 1%。**NFR-PERF-05** 游戏资源包（含纹理、音效、动画）的首次加载时间必须≤3 秒，条件为中端移动设备（参照基准机型：iPhone 12 / 骁龙 870 安卓机）在 WiFi 网络环境下。**NFR-PERF-06** 游戏运行时的峰值内存占用必须≤200MB（iOS/Android 原生客户端），Web 端峰值内存占用必须≤300MB。**NFR-PERF-07** 客户端从点击应用到进入游戏大厅的冷启动时间必须≤5 秒。

#### 12.1.3 并发扩展

框架须具备支撑大规模在线用户的能力。**NFR-PERF-08** 单个服务实例必须支持≥10,000 名同时在线用户，该指标在长连接保持和心跳机制开启的常态负载下衡量。**NFR-PERF-09** 单个游戏房间必须支持≥100 名观战者同时在线，且观战者数量不影响房间内玩家的操作延迟。**NFR-PERF-10** 房间服务的单实例承载房间数必须≥500 个活跃房间，活跃房间定义为存在至少一名玩家且状态非"已解散"的房间。

#### 12.1.4 数据库性能

持久化层的性能直接影响玩家数据查询和写入体验。**NFR-PERF-11** 玩家常用数据查询（如用户信息、资产余额、好友列表）的响应延迟 P99 必须≤20ms。**NFR-PERF-12** 缓存层（Redis 集群）的查询命中率必须≥95%，对于低于此阈值的业务场景，必须触发缓存策略优化告警。**NFR-PERF-13** 数据库写入操作（如对局记录落盘、资产变更日志）必须采用异步批量写入策略，单条写入延迟 P99 必须≤100ms。

下表汇总全部性能指标及其验收条件：

| 指标项 | 目标值 | 测试条件 | 优先级 |
|---|---|---|---|
| 单房间消息处理延迟（P99） | ≤50ms | 单房间4名活跃玩家，每秒10次操作 | 高 |
| 状态广播延迟（P99） | ≤100ms | 同一房间含50名观战者 | 高 |
| 消息保序 | 零乱序/零丢包 | 压力测试持续30分钟 | 高 |
| UI渲染帧率 | ≥60fps | 中端设备，标准游戏场景 | 高 |
| 游戏资源加载 | ≤3秒 | WiFi网络，首次加载完整资源包 | 高 |
| 客户端内存峰值 | ≤200MB（原生）/≤300MB（Web） | 游戏运行中峰值 | 高 |
| 冷启动时间 | ≤5秒 | 应用完全关闭后重新启动 | 中 |
| 单实例在线用户 | ≥10,000人 | 长连接常态负载 | 高 |

### 12.2 可用性需求

#### 12.2.1 系统可用性

**NFR-AVAIL-01** 系统月度可用性指标（SLA）必须达到 99.9%，即月度计划外停机时间不得超过 43.2 分钟。**NFR-AVAIL-02** 故障恢复时间目标（RTO）必须≤15 分钟，从故障检测触发到服务完全恢复并重新接入流量为计时终点。**NFR-AVAIL-03** 系统必须提供健康检查端点 `/health`，负载均衡器以此判定实例是否可接收流量，检查周期为每 10 秒一次，连续 3 次失败即标记实例为不可用并剔除。

#### 12.2.2 维护策略

**NFR-AVAIL-04** 计划维护窗口必须实现零停机更新，采用蓝绿部署（Blue-Green Deployment）策略，新版本实例健康检查通过后通过流量切换完成更新，旧版本实例保留至新版本稳定运行至少 10 分钟后方可下线。**NFR-AVAIL-05** 数据备份策略必须包含每小时增量备份和每日全量备份，备份数据必须存储在异地灾备节点，全量备份保留周期≥30 天，增量备份保留周期≥7 天。

#### 12.2.3 灾难恢复

**NFR-AVAIL-06** 数据恢复点目标（RPO）必须≤5 分钟，即灾难场景下可接受的最大数据丢失时间窗口为 5 分钟。**NFR-AVAIL-07** 核心服务必须部署在至少两个可用区（Availability Zone），可用区之间通过低延迟专线同步数据。**NFR-AVAIL-08** 系统必须支持自动故障转移：当主可用区的核心服务实例异常率超过 50% 时，流量必须在 3 分钟内自动切换至备用可用区，切换过程对在线玩家无感知。

### 12.3 安全性需求

#### 12.3.1 传输安全

**NFR-SEC-01** 所有客户端与服务端之间的通信必须全链路采用 TLS 1.2 及以上版本加密，严禁明文传输任何敏感数据。**NFR-SEC-02** TLS 证书必须支持自动轮换，证书过期前 30 天触发告警，过期前 7 天自动完成新证书签发和部署。**NFR-SEC-03** WebSocket 连接必须使用 WSS（WebSocket Secure）协议，握手阶段的 HTTP 升级请求必须携带有效的身份认证令牌。

#### 12.3.2 防作弊体系

**NFR-SEC-04** 系统必须实现异常操作模式检测引擎，通过实时分析玩家胜率偏离度（超出统计置信区间≥3 个标准差时触发标记）、操作频率异常（单位时间内操作次数超过人类合理上限）和操作时序模式（如多账号同步操作、固定间隔响应）识别可疑账号。**NFR-SEC-05** 系统必须对接外挂行为识别模块，通过客户端行为指纹采集（操作轨迹、触控模式、渲染特征）检测模拟点击、透视辅助等外挂程序。**NFR-SEC-06** 所有被标记的可疑账号必须自动生成人工审核工单，工单须包含完整的对局回放、统计数据和触发原因摘要，审核员必须在 24 小时内完成初审。

#### 12.3.3 资产安全

**NFR-SEC-07** 所有虚拟资产操作（创建、变更、转移、销毁）必须以服务端为唯一权威来源，客户端仅允许提交操作意图指令，严禁在本地计算或预测资产变更结果。**NFR-SEC-08** 服务端必须对每笔资产操作执行原子性校验，校验内容包括操作者身份、操作合法性、资产余额充足性和操作幂等性（防重放）。**NFR-SEC-09** 所有资产操作日志必须写入防篡改存储（WORM, Write Once Read Many），日志字段须包含操作者 ID、操作类型、变更前后数值、时间戳、事务 ID 和对局上下文。

#### 12.3.4 隐私合规

**NFR-SEC-10** 系统必须遵循 GDPR 和中国《个人信息保护法》的数据收集最小化原则，仅收集业务运行所必需的用户信息，所有数据收集行为必须明示目的并获得用户 consent。**NFR-SEC-11** 系统必须提供用户数据导出接口，用户提交请求后 30 天内必须可下载其全部个人数据的结构化副本。**NFR-SEC-12** 系统必须提供用户数据删除接口，删除请求必须在 30 天内完成全部数据副本（含备份）的物理清除，并返回删除确认凭证。**NFR-SEC-13** 所有用户 consent 记录必须持久化存储，包含 consent 事项、版本、时间戳和用户标识，支持按用户维度查询和撤回。

下表覆盖四个安全维度的核心机制：

| 安全维度 | 机制名称 | 核心功能 | 技术实现 |
|---|---|---|---|
| 传输安全 | 全链路TLS | 防止中间人攻击和窃听 | TLS 1.2+，证书自动轮换 |
| 传输安全 | WSS协议 | 保障WebSocket通信安全 | 握手阶段嵌入JWT认证 |
| 防作弊 | 异常模式检测 | 识别胜率和频率异常玩家 | 实时流计算，3σ阈值 |
| 防作弊 | 外挂行为识别 | 检测模拟点击和透视辅助 | 客户端指纹+服务端校验 |
| 防作弊 | 人工审核工单 | 可疑案例人工复核 | 自动生成含回放和证据的工单 |
| 资产安全 | 服务端权威模型 | 杜绝客户端篡改资产 | 指令-校验-执行分离架构 |
| 资产安全 | 防篡改操作日志 | 确保资产变更可追溯 | WORM存储，事务级记录 |
| 隐私合规 | 数据最小化 | 仅收集必需个人信息 | 字段级 consent 控制 |
| 隐私合规 | 用户数据导出 | 支持数据可携带权 | 标准化数据导出接口 |
| 隐私合规 | 用户数据删除 | 支持被遗忘权 | 全副本级联物理删除 |

上表所列安全机制构成框架纵深防御体系。传输安全层保障数据在传输过程中的机密性和完整性；防作弊层通过算法检测与人工审核相结合，维护对局公平性；资产安全层以服务端权威和不可篡改日志为核心，杜绝虚拟资产被非法操作；隐私合规层确保系统在全球各运营地区满足当地法律法规要求。各维度机制独立运行但可通过关联日志实现跨维度审计，例如将防作弊标记与资产操作日志关联，可追踪作弊行为是否涉及资产异常流转。

### 12.4 兼容性需求

#### 12.4.1 平台覆盖

框架客户端必须覆盖主流终端平台，确保玩家可在任意设备上接入游戏。**NFR-COMP-01** iOS 平台必须支持 iOS 13.0 及以上版本，推荐 iOS 15.0+。**NFR-COMP-02** Android 平台必须支持 Android 8.0（API Level 26）及以上版本，推荐 Android 10.0+。**NFR-COMP-03** Web 端必须兼容 Chrome 90+、Safari 14+ 和 Firefox 88+，这三款浏览器的合计市场份额覆盖率须≥90%。**NFR-COMP-04** PC 客户端必须支持 Windows 10（64 位）和 macOS 11（Big Sur）及以上版本。

下表列明各平台的版本矩阵：

| 平台 | 最低支持版本 | 推荐版本 | 测试基准机型/环境 | 终止支持策略 |
|---|---|---|---|---|
| iOS | 13.0 | 15.0+ | iPhone 12, iPhone SE 2 | 跟随Apple官方终止支持节奏 |
| Android | 8.0 (API 26) | 10.0+ (API 29+) | 骁龙870/6GB RAM | 最低版本覆盖≥90%活跃设备 |
| Chrome | 90 | 最新稳定版 | Windows 10, 1920x1080 | 落后最新版≤18个月 |
| Safari | 14 | 最新稳定版 | macOS 12, iPad Air 4 | 跟随macOS/iOS版本迭代 |
| Firefox | 88 | 最新稳定版 | Windows 10, 1920x1080 | 落后最新版≤18个月 |
| Windows | Windows 10 64位 | Windows 11 | 主流办公配置 | 跟随Microsoft生命周期 |
| macOS | 11 Big Sur | 13 Ventura+ | MacBook Air M1 | 跟随Apple官方终止支持节奏 |

上表中的"终止支持策略"定义了各平台最低版本的淘汰机制。当某平台版本的全球活跃设备占比低于 5% 且官方已终止安全更新时，框架将在下一主版本中移除对该版本的支持，并提前 3 个月通过公告通知用户升级。此策略平衡了用户覆盖率和开发维护成本。

#### 12.4.2 插件API兼容

**NFR-COMP-05** 框架的插件 API 必须向后兼容至少 2 个主版本，即使用 API vN 开发的插件在框架 vN+1 和 vN+2 版本中必须可正常运行，无需修改代码。**NFR-COMP-06** 对于计划废弃的 API，框架必须在废弃前的主版本中标记为 `@Deprecated`，并在版本更新日志中提供替代方案，过渡期不少于 6 个月，过渡期内被废弃的 API 必须保持完整功能。

#### 12.4.3 网络降级

**NFR-COMP-07** 在 WiFi、4G 和 5G 网络环境下，系统必须以正常模式运行，所有功能完整可用。**NFR-COMP-08** 在弱网环境（网络延迟≥500ms 或丢包率≥10%）下，客户端必须启用操作合并机制：将短时间内（≤300ms）的多次 UI 操作合并为单条网络请求发送，减少无效传输。**NFR-COMP-09** 弱网环境下，系统必须启动延迟容忍策略：操作超时阈值从默认 15 秒延长至 30 秒，服务端状态广播频率从实时降频为每 500ms 批量发送一次，客户端以本地预测动画维持界面响应。

### 12.5 可维护性需求

#### 12.5.1 监控体系

**NFR-MAINT-01** 系统必须接入 APM（Application Performance Monitoring）性能监控平台，追踪全部服务接口的响应延迟、吞吐量和错误率，采样率必须≥10%。**NFR-MAINT-02** 业务指标 Dashboard 必须提供实时在线人数、活跃房间数、各游戏参与度、资产流通量等核心业务指标的分钟级刷新视图。**NFR-MAINT-03** 告警系统必须支持自定义阈值配置，当指标偏离正常基线（超出 2 倍标准差）时必须在 30 秒内触发告警通知，通知渠道至少包括邮件和即时消息（如企业微信/Slack）。

下表定义关键监控指标及其告警阈值：

| 监控类别 | 指标名称 | 告警阈值 | 告警级别 | 响应SLA |
|---|---|---|---|---|
| 性能 | API P99延迟 | >100ms（持续3分钟） | 警告 | 15分钟内响应 |
| 性能 | 单实例CPU使用率 | >80%（持续5分钟） | 警告 | 15分钟内响应 |
| 性能 | 单实例内存使用率 | >85%（持续5分钟） | 严重 | 5分钟内响应 |
| 业务 | 同时在线人数突降 | 5分钟内下降>30% | 严重 | 5分钟内响应 |
| 业务 | 房间创建失败率 | >1%（持续2分钟） | 严重 | 5分钟内响应 |
| 业务 | 资产操作异常率 | >0.1%（任意时段） | 紧急 | 立即响应 |
| 可用性 | 服务实例健康检查失败 | 连续3次失败 | 紧急 | 立即响应 |
| 可用性 | 数据库连接池耗尽 | 活跃连接>90%容量 | 严重 | 5分钟内响应 |
| 安全 | 防作弊标记触发率 | 单小时>50次 | 警告 | 30分钟内响应 |
| 安全 | 异常登录尝试 | 单账号5分钟内>10次 | 严重 | 5分钟内响应 |

上表共覆盖性能、业务、可用性和安全四大监控类别。告警级别从"警告"到"紧急"共四级，对应不同的响应 SLA。"警告"级别仅需值班人员关注并在工作时间内处理；"严重"级别要求值班人员在 5 分钟内介入；"紧急"级别要求立即响应并启动应急预案。业务指标中的"资产操作异常率"因涉及虚拟资产安全，阈值设定极为严格（>0.1%），一旦触发须立即冻结相关交易路径并启动审计。

#### 12.5.2 日志规范

**NFR-MAINT-04** 系统日志必须采用五级分层结构：DEBUG（开发调试信息）、INFO（正常运行信息）、WARN（潜在风险警告）、ERROR（功能性故障）和 FATAL（系统性灾难），生产环境默认输出级别为 INFO。**NFR-MAINT-05** 所有日志必须以结构化 JSON 格式输出，字段至少包含时间戳（ISO 8601 格式，精确到毫秒）、日志级别、服务名、Trace ID、消息体和调用链上下文。**NFR-MAINT-06** 日志必须通过 Filebeat/Fluentd 等采集代理实时汇聚至集中式日志平台（如 ELK Stack 或 Loki），采集延迟必须≤30 秒，日志保留周期≥90 天。

#### 12.5.3 插件开发支持

**NFR-MAINT-07** 框架必须提供完整的插件 SDK 文档，文档须覆盖所有接口定义、数据类型、事件契约和最佳实践，并随每个主版本更新。**NFR-MAINT-08** 框架必须提供本地调试工具，支持插件开发者在不依赖完整服务端环境的情况下运行和测试插件逻辑，调试工具须包含模拟玩家输入、状态快照注入和断点单步执行能力。**NFR-MAINT-09** 框架必须提供游戏模拟器，允许开发者在 PC 上快速验证插件的 UI 布局和交互流程，模拟器须支持常见屏幕尺寸切换和触控事件模拟。**NFR-MAINT-10** 框架必须提供配套的单元测试框架，包含 Mock 服务端环境、Mock 玩家对象和常见测试场景模板，插件开发者编写测试用例的覆盖率应≥80%。

### 12.6 可扩展性需求

#### 12.6.1 新游戏接入

**NFR-SCALE-01** 从需求确认到新游戏插件上线运行的标准流程周期必须≤4 周，该周期包含需求评审、插件开发、联调测试和灰度发布四个阶段。**NFR-SCALE-02** 框架必须定义标准化的插件开发流程模板，每个阶段须明确的输入、输出和验收标准：需求评审阶段输出游戏设计文档和接口契约；开发阶段输出可运行的插件包；测试阶段输出测试报告（含功能、性能、兼容性）；发布阶段输出上线检查清单。**NFR-SCALE-03** 新游戏插件的灰度发布必须支持按用户群体（如内部员工、5% 用户、全量）渐进式放量，每个灰度阶段须持续≥48 小时且无 P1 级 Bug 方可进入下一阶段。

#### 12.6.2 水平扩容

**NFR-SCALE-04** 房间服务必须设计为无状态（Stateless）架构，所有房间状态持久化于 Redis 集群，任意服务实例均可处理任意房间的请求，确保水平扩容时不涉及数据迁移。**NFR-SCALE-05** 系统必须配置自动扩缩容触发条件：当单实例 CPU 使用率持续 3 分钟≥70% 或内存使用率持续 5 分钟≥80% 时，自动创建新实例并加入负载均衡；当上述指标持续 10 分钟≤40% 时，自动缩减实例数量，但保留至少 2 个运行实例。**NFR-SCALE-06** 容量规划模型必须基于历史数据预测峰值负载，预测因子包括：日均活跃用户（DAU）、高峰时段并发系数（通常为 DAU 的 15%-20%）、平均房间人数和新游戏上线带来的流量增益（预留 30% 缓冲）。

#### 12.6.3 多地区部署

**NFR-SCALE-07** 框架必须支持多地区部署架构，不同地区的数据中心可独立运行，玩家就近接入以降低网络延迟。**NFR-SCALE-08** 跨地区对战的数据同步必须通过专线或加速网络完成，跨地区玩家同房间对战时的额外延迟增量必须≤50ms。**NFR-SCALE-09** 地区隔离与合并策略必须支持以下场景：同一地区玩家优先匹配同地区房间以降低延迟；当某地区在线人数不足时，系统自动将匹配范围扩展至相邻地区； tournaments 等全局赛事必须支持跨地区同场竞技。**NFR-SCALE-10** 系统必须正确处理多时区场景，所有服务端时间戳统一使用 UTC 存储，客户端根据设备时区本地化显示，跨时区对局的结算时间和赛事排期必须以创建者所在时区为基准并向前兼容显示所有参赛玩家的本地时间。

上表列出的 11 个接口方法是插件契约的核心。框架在加载插件时首先调用 `registerPlugin` 获取元数据并校验冲突（**FR-FW-PLUGIN-02**）；创建房间时调用 `getSettingsSchema` 渲染配置面板（**FR-FW-PLUGIN-03**）；玩家点击"开始"后调用 `createGameController` 初始化对局（**FR-FW-PLUGIN-04**）；游戏中每一步由框架通过 `validateMove` → `applyMove` → `checkGameEnd` 的管道处理（**FR-PLUGIN-GC-01** 至 **FR-PLUGIN-GC-05**）；AI 托管时调用 `createAIController` 获取决策实例（**FR-PLUGIN-AI-01**）；回放功能依赖 `serializeState` 与 `deserializeState` 实现状态快照存取（**FR-FW-DB-06**）。

**表 13-5 框架提供给插件的回调接口（Plugin → Framework）**

| 接口方法 | 参数 | 说明 |
|----------|------|------|

### 13.5.2 开发环境搭建

插件开发环境搭建遵循以下有序步骤。

步骤 1：安装 SDK。执行 `npm install -g @chess-framework/plugin-sdk@latest` 安装命令行工具，版本须不低于 2.1.0。SDK 内置 TypeScript 类型定义、脚手架模板与本地调试服务器。

步骤 2：初始化项目。执行 `chess-plugin init my-game` 创建插件项目骨架，选择对应的游戏类别模板（board-chess / board-tile / card-trick / card-poker）。脚手架自动生成 `src/index.ts`、`src/GameController.ts`、`src/AIController.ts`、`tests/` 目录与 `plugin.json` 配置文件。

步骤 3：配置调试环境。在项目根目录执行 `chess-plugin config --debug-port 9229`，SDK 将在本地启动框架模拟器（MockFramework），模拟器实现了 `IFrameworkAPI` 的全部回调方法，并预置 4 个虚拟玩家账号供测试使用。编辑 `launch.json` 添加 Node.js Attach 配置，指向端口 9229。

步骤 4：实现核心接口。按 Checklist 顺序依次实现各接口方法。每完成一项，执行 `chess-plugin test --item <#>` 运行对应自动化验证。GameController 的状态更新必须遵循不可变语义——`applyMove` 返回新状态对象而非修改原对象，以确保时间旅行调试与回放功能正确性。

步骤 5：本地联调。执行 `chess-plugin serve` 启动本地开发服务器，SDK 将插件打包为 Hot-Reload 模式并注入 MockFramework。使用浏览器访问 `http://localhost:3456/dev` 打开开发者控制台，可实时查看状态变更、动作日志与 WebSocket 消息流。

步骤 6：打包提交。全部 Checklist 验证通过后，执行 `chess-plugin build` 生成生产包（`.chep` 格式），执行 `chess-plugin submit --plugin-file dist/my-game.chep` 上传到框架插件市场。上传前 SDK 自动执行静态分析、类型检查与安全扫描，任一检查未通过则中止提交。
