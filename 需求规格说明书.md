# 通用棋牌游戏框架软件需求规格说明书
**版本 1.0**

---

## 1. 引言

### 1.1 目的
本文档旨在定义一款**通用、可扩展的棋牌游戏平台框架**的完整需求。该框架的核心目标是实现“一次搭建，多游戏复用”，通过标准化的插件接口，快速集成麻将、斗地主、德州扑克、象棋等各类棋牌游戏。

### 1.2 范围
本框架涵盖客户端大厅、用户系统、房间管理、游戏状态机抽象、AI托管、社交与成就等全部通用能力。所有具体棋牌游戏均作为独立插件运行于框架之上。

### 1.3 术语与缩写
| 术语 | 说明 |
|------|------|
| 游戏插件 | 遵循框架接口规范开发的独立游戏模块，如德州扑克插件、斗地主插件。 |
| 游戏大厅 | 展示所有已安装游戏插件的统一入口界面。 |
| 房间 | 由一名玩家创建、其他人可加入并共同对局的虚拟空间。 |
| 快速匹配 | 玩家设定偏好后，由系统自动分配至合适房间的机制。 |
| 状态机 | 管理一局游戏从等待到结算的全部生命周期。 |

## 2. 总体描述

### 2.1 产品视角
本产品定位为**棋牌游戏平台启动器与运行时环境**。系统通过插件化架构，将公共基础能力（账号、社交、支付、匹配、房间）与具体游戏逻辑（规则、UI、AI）彻底分离。游戏作为可热插拔的独立资源包，可动态上下架。

### 2.2 核心设计理念：插件化架构
框架必须具备一套**游戏插件接口规范**，任何开发者只需实现该接口，即可让新游戏接入大厅，自动获得匹配、房间、聊天、成就等全部能力。

### 2.3 用户特征
- **普通玩家**：希望在一个App内玩到多种棋牌游戏，共享资产与好友。
- **游戏开发者**：希望专注于游戏规则开发，不重复造轮子处理账号、社交等问题。

## 3. 系统架构需求（插件化规范）

这是本框架最核心的章节，定义如何“扩展游戏类型”。

### 3.1 游戏插件接口规范
每个具体游戏必须作为独立插件，实现以下生命周期接口：

- **FR-PLUGIN-01 注册信息**：插件须向框架提供元数据 `PluginMeta`，包括：
  - 游戏唯一标识 (game_id)
  - 游戏名称（如“德州扑克”、“斗地主”）
  - 游戏图标（用于大厅展示）
  - 游戏类型标签（类别：棋类、牌类、麻将类）
  - 最小/最大玩家人数 (2-10)
  - 简要规则描述文本
  - 版本号
- **FR-PLUGIN-02 设置界面**：须提供一个方法返回该游戏的参数设置UI组件（或Web视图），供房主在创建房间时配置。
- **FR-PLUGIN-03 逻辑控制器**：须提供一个标准的 `GameController`，负责驱动该游戏的完整对局流程。
- **FR-PLUGIN-04 资源包加载**：插件须能加载其专属资源（牌面纹理、棋盘模型、特效、音效），并需支持远程动态下载与更新。

### 3.2 游戏状态机抽象
框架定义通用的对局状态机，所有游戏插件必须遵循此流程，但可在具体状态中实现差异逻辑。

通用状态流转：
> `[等待加入]` → `[准备阶段]` → `[发牌/开局]` → `[玩家操作循环]` → `[结算展示]` → `[返回房间或解散]`

- **FR-STATEM-01 玩家操作循环**：在“玩家操作”状态中，框架通过事件总线轮询当前应行动的玩家，并等待其通过 `submitAction()` 提交操作。插件负责验证操作合法性。
- **FR-STATEM-02 超时处理**：框架提供统一的倒计时服务，若玩家超时，框架自动调用插件定义的 `onTimeout()` 逻辑（如自动出最小牌、弃牌或托管）。

## 3.3 游戏插件接口技术规范（详细定义）

为保障框架的开放性与一致性，每个棋牌游戏必须以**插件**形式实现以下技术契约。所有接口均采用面向对象抽象，实际实现时可根据技术栈翻译为对应语言的 `interface`、`protocol` 或 REST API 消息契约。

### 3.3.1 插件元数据与注册

框架启动时，会扫描所有已安装插件，调用其注册方法获取元数据，并存入游戏注册表。

- **FR-PLUGIN-01-01 元数据结构**  
  插件必须向框架提供一个 `PluginMeta` 对象，内容须包含：

  | 字段 | 类型 | 说明 |
  |------|------|------|
  | `gameId` | String | 全局唯一标识，如 “texas_holdem” |
  | `gameName` | String | 显示名称，支持多语言 key |
  | `category` | Enum | 游戏分类：麻将、牌类、棋类、其他 |
  | `minPlayers` | Int | 最少玩家人数 (≥1，1代表可AI补位) |
  | `maxPlayers` | Int | 最大玩家人数 |
  | `icon` | AssetRef | 大厅平铺所使用的图标资源引用 |
  | `description` | String | 规则简介，支持多语言 key |
  | `version` | String | 插件版本号，用于热更新校验 |
  | `supportedModes` | List<String> | 支持的模式列表，如 ["cash","tournament"] |
  | `currencySlug` | String | 本游戏专用积分币种代号，如 “chips” |

- **FR-PLUGIN-01-02 注册回调**  
  插件需导出标准注册入口函数 `registerPlugin(context: FrameworkContext): PluginMeta`。框架传入上下文对象，供插件获取通用能力（如日志、网络、存储）。插件返回元数据完成注册。

- **FR-PLUGIN-01-03 卸载能力**  
  插件应提供 `unregister()` 方法，用于释放占用的内存、移除事件监听。框架可随时调用来移除游戏。

### 3.3.2 游戏参数设置接口

当房主创建房间时，框架需动态渲染该游戏的专属设置面板。

- **FR-PLUGIN-02-01 参数Schema定义**  
  插件须提供一个方法 `getSettingsSchema(): UISchema`，返回一个声明式的UI描述。该Schema需足够灵活，能表达：
  - 数值输入（如盲注、人数）
  - 单选/多选（如对局模式）
  - 开关（如是否允许观战）
  - 滑块（如行动时间）

  `UISchema` 建议采用平台无关的结构，例如：
  ```json
  {
    "fields": [
      {
        "key": "small_blind",
        "type": "number",
        "label": "小盲注",
        "default": 10,
        "min": 1,
        "max": 10000,
        "step": 10
      },
      {
        "key": "num_players",
        "type": "radio",
        "label": "玩家数量",
        "options": [2, 6, 9],
        "default": 6
      }
    ]
  }
  ```
  框架客户端负责将此Schema渲染成原生UI组件。

- **FR-PLUGIN-02-02 参数校验**  
  插件须提供 `validateSettings(settings: Map<String, Any>): ValidationResult`，在房主点击“创建”时，由框架调用来验证参数合法性。`ValidationResult` 须包含是否通过及具体错误信息。

### 3.3.3 游戏逻辑控制器（GameController）接口

这是插件最核心的契约，负责驱动一局游戏的全部状态。

- **FR-PLUGIN-03-01 控制器生命周期**  
  插件须提供 `createGameController(roomConfig: RoomConfig): GameController`，每次房间开始游戏时框架会调用此工厂方法生成一个控制器实例。`RoomConfig` 包含房主设置的参数、玩家列表、AI配置等。

- **FR-PLUGIN-03-02 通用操作与事件**
  `GameController` 至少应包含以下方法：

  | 方法 | 说明 |
  |------|------|
  | `start()` | 初始化游戏状态，触发开局流程（洗牌、发牌等） |
  | `getCurrentState(): GameState` | 返回当前完整游戏状态快照，用于断线重连或观战 |
  | `getActivePlayer(): String` | 返回当前应行动的玩家ID，无则返回 null |
  | `submitAction(playerId: String, action: Action)` | 接收玩家操作，由控制器验证并处理 |
  | `getLegalActions(playerId: String): List<Action>` | 返回当前玩家可选的所有合法操作 |
  | `endGame()` | 强制结束游戏（如房主解散） |

- **FR-PLUGIN-03-03 事件总线**  
  控制器须通过框架提供的事件总线 `EventBus` 广播状态变更，而**不能直接调用UI渲染**。标准事件包括：
  - `onGameStarted()`：游戏正式开始
  - `onStateUpdated(GameState)`：任何状态变更（如公共牌更新、玩家弃牌）
  - `onPlayerTurn(playerId, legalActions, timeout)`：轮到某玩家行动，须附带超时时间
  - `onRoundEnd(roundResult)`：一局牌结束，包含赢家、损益等信息
  - `onGameFinished(finalResult)`：整场游戏结束（如锦标赛淘汰到只剩1人）

  框架大厅侧监听这些事件来驱动UI刷新、播放动画、记录日志。

### 3.3.4 AI控制器接口

为使每个游戏都能接入AI托管，插件须实现AI。

- **FR-AI-01-01 AI工厂**  
  插件须提供 `createAIController(difficulty: String, gameController: GameController): AIController`。`difficulty` 为 “easy”、“hard” 等，由房主在设置中指定。

- **FR-AI-01-02 AI决策**  
  `AIController` 须实现方法 `requestAction(playerId: String, gameState: GameState, legalActions: List<Action>, callback: (Action) -> void)`。  
  框架会在需要AI行动时调用此方法，AI必须在设定的超时时间内通过回调返回一个合法操作。游戏控制器不关心操作来自人类还是AI。

- **FR-AI-01-03 托管模式标识**  
  AI返回的操作需携带 `source: "ai"` 标记，以便UI展示“托管中”图标。

### 3.3.5 资源与渲染接口

框架负责通用UI（聊天、头像框），游戏插件负责牌桌/棋盘渲染。

- **FR-PLUGIN-05-01 游戏视图组件**  
  插件须提供一个 `getGameView(): ViewComponent`（或同等抽象），框架将在进入游戏房间后将其嵌入主界面。该组件内部完全由插件控制，负责绘制牌面、棋子、操作按钮等。

- **FR-PLUGIN-05-02 资源包加载**  
  插件所有资源（图片、音效、动画）必须通过框架的 `AssetLoader` 异步加载，便于框架统一缓存和更新。插件须在元数据中声明资源清单，框架在游戏启动前或后台预加载。

- **FR-PLUGIN-05-03 多分辨率适配**  
  插件视图必须实现自适应布局，支持手机竖屏、平板横屏及PC窗口大小。框架提供安全区、基准设计尺寸等环境变量，插件据此调整。

### 3.3.6 回放与序列化接口

- **FR-PLUGIN-06-01 动作序列化**  
  每个 `Action` 对象必须支持序列化为通用格式（如JSON），并包含操作类型、玩家ID、数据及时间戳。框架负责存储这些序列化后的操作日志。

- **FR-PLUGIN-06-02 从日志重建**  
  插件须提供 `replayFromLog(initialState: GameState, actions: List<Action>): GameController`，基于初始状态和操作序列重建一个“回放专用”控制器。该控制器只允许顺序前进，不接受玩家输入，用于回放器。

### 3.3.7 国际化（i18n）支持

- **FR-PLUGIN-07-01 文本资源外挂**  
  插件中所有面向用户的文本（按钮、提示、规则说明）必须使用字符串 key 并在独立语言文件中提供默认英文翻译。框架根据当前语言加载对应翻译表，插件须通过 `context.localize(key)` 获取最终显示文本。

## 4. 功能需求

### 4.1 主界面与大厅
*对应您原始需求第2、3条。*

- **FR-LOBBY-01 游戏类型平铺**：系统启动后进入游戏大厅，以网格或列表平铺形式展示所有已安装的游戏插件图标与名称。
- **FR-LOBBY-02 筛选与隐藏**：
  - 顶部提供搜索框，支持按名称关键词实时筛选。
  - 提供分类标签页（全部、牌类、棋类、麻将）。
  - 长按游戏图标，可将其临时隐藏；在“设置-隐藏的游戏”中可恢复显示。
- **FR-LOBBY-03 快速入口**：大厅应同时显示“最近在玩”的游戏列表，排序权重高于其他游戏。

### 4.2 用户与资产体系

- **FR-USER-01 统一账号**：支持手机号、邮箱、第三方账号登录。所有游戏内身份（昵称、头像）统一。
- **FR-USER-02 多币种资产**：系统须维护一个**多币种虚拟钱包**：
  - 通用币：金币（可用于所有游戏房间的低保底注入场）。
  - 独立积分：如德州扑克筹码、麻将欢乐豆、斗地主金豆。各游戏插件可注册自己的币种，兑换比例与规则由插件定义。

### 4.3 房间与匹配系统
*对应您的原始需求第4条（点击进入子界面）的核心补充。*

- **FR-ROOM-01 创建房间**：在大厅选择游戏后，玩家点击“创建房间”。系统调起该游戏插件提供的**参数设置界面**（见您的第5点，如设置盲注、人数等）。玩家确认后，系统创建房间并生成唯一房间码。
- **FR-ROOM-02 加入房间**：其他玩家可通过输入房间码、从大厅房间列表挑选、或点击“快速匹配”加入。
- **FR-ROOM-03 房间列表**：选择某款游戏后，子界面应展示**当前可加入的活跃房间列表**，含房主昵称、人数（3/6）、底注/级别、游戏模式等关键信息，并支持按“等待中”、“进行中”筛选。
- **FR-ROOM-04 快速匹配**：玩家选择游戏后点击“快速开始”，框架根据其当前资产和偏好，自动匹配符合入场门槛且状态为“等待中”的房间。若无可匹配房间，则自动创建一个新房间并等待他人加入。

### 4.4 社交与通讯系统

- **FR-SOCIAL-01 好友系统**：独立于具体游戏，支持搜索、添加、删除好友，查看在线状态。
- **FR-SOCIAL-02 聊天系统**：提供世界频道、游戏房间内频道、私聊频道。框架提供通用聊天UI组件及预设表情/短语库，各游戏可在牌桌上直接调用。
- **FR-SOCIAL-03 语音快捷回复**：支持预设语音消息（如“快点出牌啊”、“这牌不错”）。

### 4.5 AI托管系统
*对应您原始需求第5点“游戏难度”的系统级支撑。*

- **FR-AI-01 AI控制器接口**：框架定义 `AIController` 抽象类，每个游戏插件需实现自己的AI逻辑。
- **FR-AI-02 难度分级**：AI须提供至少“新手”、“高手”两种难度。房主可在创建房间时设定AI难度级别。
- **FR-AI-03 托管与自动接管**：对局中，玩家可主动点击“托管”由AI代打。若玩家在操作回合中断线或超时超过设定次数，系统自动切换为AI托管，并在界面上向其他玩家清晰标示。

### 4.6 成就与任务系统

- **FR-ACHV-01 跨游戏成就**：系统定义全局成就，如“初次胜利（任意游戏）”、“社交达人（加10个好友）”、“全能王（体验5款不同游戏）”。
- **FR-ACHV-02 游戏专属成就**：插件可向框架注册专属成就列表，如德州扑克的“同花顺获胜”、斗地主的“春天”等。框架负责监听并触发。

### 4.7 回放与记录

- **FR-RECORD-01 对局记录**：服务端须以通用格式存储每局游戏的完整操作序列（初始状态+每步操作+时间戳），形成可复盘的对局记录。
- **FR-RECORD-02 回放器**：提供通用回放UI（播放/暂停/步进/退出），可加载对局记录文件进行复盘。框架提供底层控制，插件提供渲染。

## 5. 具体游戏需求附录示例：德州扑克

*此为框架如何被插件填充的示例，对应您的原始需求第5点。*

### 5.1 创建房间时的参数设置
进入德州扑克子界面后，房主可设置的参数须包括：
- 小盲注 / 大盲注：提供多个固定档位或自定义输入。
- 玩家人数上限：2人（单挑）、6人、9人。
- 对局模式：现金桌、坐满即玩赛。
- 买入下限/上限。
- AI托管难度：关闭、新手、高手。
- 行动时间限制：15秒/30秒/45秒。

### 5.2 牌桌界面布局（游戏内）
- **公共区**：桌面中央显示5张公共牌。
- **玩家区**：各座位显示头像、昵称、筹码量、当前状态（等待/行动中/弃牌/全下）、本局已下注筹码。
- **操作区**：弃牌、过牌、跟注、加注（可拖拽滑块）、全下按钮，以及清晰可见的倒计时环。
- **结果区**：摊牌时高亮获胜手牌，播放筹码飞向胜者动画。

## 6. 非功能需求

- **NFR-PERF-01** 框架层对玩家操作的响应延迟应低于200ms，不增加额外性能开销。
- **NFR-PORT-01** 客户端框架须适配iOS、Android及主流Web浏览器。插件开发者可据此开发跨平台游戏内容。
- **NFR-SEC-01** 虚拟资产变更必须由服务端框架统一校验与结算，插件只上报操作指令，不可直接修改资产。
- **NFR-FAIR-01** 框架提供统一的认证随机数生成器（RNG）服务接口，供所有棋牌、麻将类插件调用，确保公平。
